# DataFlowKit 界面逻辑解耦实施方案

> 目标：让 Qt/Tkinter/Web/Electron/.NET 等界面只负责显示与交互，把计划编辑、配置 schema、表格解析、校验、执行、输出、插件、权限、跳转等逻辑沉到后端服务层。
>
> 本文是在 `界面逻辑解耦方案.md` 的基础上整理的实施版：保留其主方向，并补充工程边界、接口分层和推荐优先级。
>
> v2 更新重点：补入 `界面逻辑解耦实施方案 - 分析补充.md` 中指出的硬约束，包括 schema 覆盖不足、动态表单表达力、旧 plan 迁移、预览零副作用、大表跨进程传输、双 UI 过渡和统一 issue schema。

---

## 1. 总原则

前端只做三件事：

1. 请求 schema/state。
2. 渲染界面。
3. 发送 command 并显示 event/result。

后端负责：

1. 计划结构与节点状态变更。
2. 节点目录、默认配置、字段 schema。
3. 表格/计划文件读写。
4. 节点配置校验。
5. 工作流预览、执行、取消、进度。
6. 输出落地、备份、导出。
7. 插件、表权限、跳转、多表服务。

换界面时，不应重写业务规则，只重写控件渲染和事件绑定。

---

## 2. 当前基础

目前已经具备较好的解耦地基：

| 能力 | 位置 | 状态 |
|---|---|---|
| Headless 引擎 | `engine/headless.py` | 已支持 UI-free 校验、预览、执行部分节点 |
| 节点稳定协议 | `workflow/protocol_nodes.py` | 已支持 `node_type_id`，兼容旧中文 `type` |
| 节点 UI schema | `workflow/node_ui_schema.py` | 已能输出中文菜单、说明、风险、表单分组 |
| stdio worker | `engine/stdio_worker.py` | 已支持跨语言 JSON-lines 调用 |
| Qt6 shell | `ui_qt/` | 已有原版面板形态，但仍有部分逻辑在 UI |
| 纯数据节点 | `workflow/nodes/*.py` | 大部分可在 headless 中执行 |

核心问题不是方向错，而是 UI 层还有一些可继续下沉的逻辑，并且副作用能力尚未统一服务化。

同时必须正视几个现实约束：

1. `NODE_CONFIG_LAYOUTS` 当前只精细覆盖 `core.new_columns` 和 `core.replace`，其他节点主要还是“其他参数”扁平表单。
2. 真实 `plan/*.json` 仍大量使用旧中文 `type`，很多没有 `node_type_id`。
3. 复杂节点的旧 Tkinter 配置 UI 仍是事实规格来源，不能简单丢弃。
4. 预览必须永远零副作用，不能因为服务化后把数据库/文件写入带进预览路径。
5. 未来 stdio/HTTP 前端不能长期依赖全量 `headers/rows` JSON 传输大表。

---

## 3. 推荐后端分层

不要把所有能力都塞进 `HeadlessWorkflowEngine`。建议拆成多个后端模块，再由一个 facade 统一暴露给 Qt/stdio/HTTP。

```text
engine/
  headless.py              # 只负责校验、预览、执行主循环
  workflow_facade.py       # 面向 UI/stdio/http 的统一门面，可后续新增
  job_service.py           # 后台任务、进度、取消、事件
  output_service.py        # 输出、导出、写库、备份
  plan_io.py               # 计划文件读写、模板扫描、迁移
  table_io.py              # 表格文件解析、TableData 加载
  table_store.py           # 大表句柄、分页、跨进程传输
  services.py              # WorkflowServices 接口定义
  safety_policy.py         # dry_run / execute_actions 安全边界
  issue_schema.py          # 统一 issue/error 结构

workflow/
  plan_commands.py         # 节点增删改移复制等纯 plan command
  plan_migration.py        # 旧中文 type 计划迁移到稳定 node_type_id
  node_ui_schema.py        # 节点显示与表单 schema
  protocol_nodes.py        # 节点类型 ID、显示名兼容
  config_validation.py     # 节点配置校验，可后续新增
```

`HeadlessWorkflowEngine` 保持相对纯净：

- `validate_plan`
- `preview_plan`
- `run_plan`
- `list_node_catalog`
- `get_node_type`
- `get_node_ui_schema`
- `make_default_node`

计划编辑、文件读写、输出、插件、权限、后台任务不直接塞进它，而是通过 facade 或 service 调用。

## 4. 需要从 UI 下沉的逻辑

### 4.1 表单渲染逻辑

当前 `ui_qt/config_form.py` 仍会自己推断字段类型：

- `bool/int/float/json/text`
- choice
- long text
- field picker

但 `workflow.node_ui_schema` 已经能产出：

```json
{
  "key": "target_field",
  "label": "目标字段",
  "type": "field_select",
  "choices": ["A", "B"],
  "default": "A",
  "help": "要处理的当前表字段。"
}
```

推荐改法：

1. Qt 表单优先消费 `schema["form"]["groups"]`。
2. 前端只实现 `field.type -> 控件` 映射。
3. 保留 fallback，用于旧节点或 schema 未覆盖字段。
4. 后续扩展高级字段类型。

推荐字段类型：

```text
text
textarea
number
bool
select
field_select
multi_field_select
table_select
path
directory
json
rule_list
mapping_table
condition_list
```

必须补充一点：schema-first 不是“Qt 表单改一下就完成”。当前精细布局只覆盖 `core.new_columns` 和 `core.replace`，复杂节点仍会退回扁平参数表单。因此第一阶段应定义为“schema-first 基础设施 + 少量节点试点”，后续以旧 Tkinter 配置 UI 为事实规格逐节点补 schema。

推荐迁移顺序：

1. `core.new_columns`、`core.replace`：保持当前试点。
2. `core.extract`、`core.datetime_format`、`core.merge_columns`：数据处理但相对可控。
3. `core.filter`、`core.writeback`、`core.selected_columns_write`：复杂多表/写入节点。
4. `core.group`、`core.loop_start`、`core.loop_judge`：流程/子工作流节点。
5. `core.plugin`：由插件 manifest/schema 动态生成。

Schema v2 还需要支持动态联动。复杂节点不能只靠 `key/label/type/choices/default/help`，至少要补：

```json
{
  "visible_when": {"field": "match_mode", "in": ["正则匹配"]},
  "enabled_when": {"field": "source_type", "equals": "SQLite表"},
  "depends_on": ["source_table"],
  "options_source": {
    "type": "table_fields",
    "table_field": "source_table"
  },
  "validation": {
    "required": true,
    "min": 1,
    "pattern": "..."
  }
}
```

其中 `options_source` 由后端解析，UI 只负责在依赖字段变化时请求新 options。

### 4.2 计划编辑逻辑

当前 Qt 直接修改 `current_plan["nodes"]`：

- 添加节点
- 删除节点
- 复制节点
- 上移/下移
- 启用/禁用
- 清空节点

这些规则应该沉到纯函数服务中。推荐不要做一堆零散 API，而是统一 command：

```python
apply_plan_command(plan, command, *, preview_headers=None) -> result
```

示例 command：

```json
{
  "type": "insert_node",
  "node_type_id": "core.replace",
  "after_index": 2
}
```

```json
{
  "type": "duplicate_node",
  "index": 3
}
```

返回：

```json
{
  "ok": true,
  "plan": {},
  "selected_index": 4,
  "issues": []
}
```

这样 Qt、Web、stdio 都不用重新实现“复制时如何重命名、插入到哪里、node_id 如何处理”。

### 4.3 计划读写与模板管理

当前 Qt 直接：

- 读 JSON
- 写 JSON
- 扫描 `plan/`
- 把输出设置塞进 plan dict

推荐下沉为：

```python
list_plan_templates(plan_dir) -> templates
load_plan(path) -> {plan, warning}
build_plan_document(plan, input_table, output_settings) -> plan_document
save_plan(path, plan_document) -> result
```

UI 只负责：

- 弹文件选择框
- 显示模板列表
- 显示 warning/error

### 4.4 表格文件解析

当前 `ui_qt/table_io.py` 解析 JSON/CSV/TSV。这个属于数据 I/O，不属于界面。

推荐迁到后端：

```python
load_table(path) -> TableData
save_table(path, table, format="csv|tsv|json|xlsx") -> result
```

UI 只传路径。

### 4.5 节点类型解析

当前 Qt 有自己的：

```python
node.get("node_type_id") or node.get("type")
```

应统一使用后端协议层：

```python
resolve_node_type_id(node) -> "core.replace"
```

避免 UI 显示和执行层对旧中文 `type` 的兼容规则不一致。

### 4.6 配置校验与强转

短期：

- UI 可以做控件层基本转换。
- 类型来源必须来自 schema。

中期：

```python
coerce_and_validate_config(node_type_id, raw_values, context) -> {
  "config": {},
  "issues": []
}
```

校验规则包括：

- 必填字段
- 字段是否存在
- 正则是否合法
- 行号范围
- 输出字段冲突
- 多表字段是否存在
- 写回规则是否完整

### 4.7 计划迁移与版本化

旧计划大量只有中文 `type`，没有 `node_type_id`。换界面前必须把迁移当成一等能力，而不是临时兼容。

推荐接口：

```python
migrate_plan(plan, target_schema_version="1.0") -> {
  "plan": {},
  "changed": true,
  "issues": []
}
```

迁移内容：

1. 为节点补 `node_type_id`。
2. 为节点补 `node_version`。
3. 为节点补稳定 `node_id`。
4. 保留旧 `type` 作为显示/兼容字段。
5. 记录 `schema_version` 或 `protocol_version`。
6. 保留未知字段，避免损坏旧模板。

验收必须包含真实 `plan/*.json` 回归测试，不能只用人工构造的小样本。

### 4.8 单一数据源，避免 choices 漂移

当前字段选项、默认配置、执行校验存在多处维护风险，例如 `FIELD_CHOICES`、`default_configs`、执行层比较/运算支持范围可能不一致。

推荐规则：

1. choices 尽量从执行/校验支持的同一份定义派生。
2. 默认配置和 schema 默认值统一由后端生成。
3. UI 不手写算子、模式、策略列表。
4. 测试覆盖“schema choices 与执行支持项一致”。

## 6. 副作用能力：WorkflowServices

不建议让 Qt 直接 import 数据库、插件、文件操作模块。应定义统一服务接口，由引擎调用。

已落地最小版 `WorkflowServices`：

```python
WorkflowServices.write_table(name, table, mode, backup=True, db_path=None)
WorkflowServices.export_xlsx(path, table, sheet_name="结果")
```

当前最小版服务已用于最终输出阶段，支持：

1. SQLite 新表输出，同名表存在时自动加时间戳。
2. SQLite 覆盖输出，可覆盖前备份旧表。
3. xlsx 文件导出。

后续仍需继续扩展为完整服务接口：

```python
class WorkflowServices:
    def list_tables(self): ...
    def read_table(self, name): ...
    def write_table(self, name, table, mode, backup=True): ...
    def save_transit(self, name, table, scope="workflow"): ...
    def read_transit(self, name): ...
    def export_xlsx(self, path, table): ...
    def list_plugins(self): ...
    def run_plugin(self, plugin_id, input_table, params, context): ...
    def rename_file(self, old_path, new_path, dry_run=True): ...
    def audit(self, event): ...
```

执行时：

```python
engine = HeadlessWorkflowEngine(services=services)
engine.run_plan(plan, input_table=table, execute_actions=True)
```

这样副作用节点只依赖服务接口，不依赖具体 UI。

### 6.1 预览零副作用安全红线

预览必须是强制 dry-run，不是 UI 约定。

安全不变量：

1. `preview_plan` 永远不能写 SQLite。
2. `preview_plan` 永远不能改文件、重命名文件、删除文件。
3. `preview_plan` 永远不能导出 xlsx 或写外部目标。
4. 插件在预览中只能走 probe/dry-run，不能执行危险动作。
5. 即使节点配置里勾选了写入、导出、实际重命名，后端也必须统一拦截。

推荐把判断集中到 `SafetyPolicy`：

```python
SafetyPolicy(mode="preview").assert_can_write_table(...)
SafetyPolicy(mode="execute").assert_can_write_table(...)
```

所有 `WorkflowServices` 的副作用方法都必须接收并检查 `dry_run/safety_policy`，不能靠各节点零散判断。

优先接入顺序：

1. [已落地第一版] `core.save_transit`
2. `core.selected_columns_write`
3. `core.writeback`
4. `core.filter`
5. `core.match_value_output`
6. `core.plugin`
7. `core.file_list`
8. `core.batch_rename`
9. `core.group`
10. `core.loop_start / core.loop_judge`

## 7. 输出系统：OutputService

`执行计划` 和 `输出设置` 不能只靠 `execute_actions=True`。需要明确输出阶段。

已落地第一版结构：

```python
class OutputSettings:
    mode: str
    target: str
    path: str
    db_path: str
    backup_before_overwrite: bool
```

```python
apply_output(headers, rows, logs, settings) -> {
  "ok": true,
  "mode": "...",
  "table": {"headers": [], "rows": []},
  "logs": [],
  "issues": [],
  "action": {"type": "update_frontend_table", "target": "preview"}
}
```

当前模式：

```text
输出到主界面预览区：已支持，返回前端表格 action。
保存为SQLite新表：已支持，需要 db_path 和输出表名。
覆盖当前表：已支持，需要 db_path 和输出表名，可覆盖前备份。
导出为xlsx：已支持，需要 output_path。
```

Qt 的输出下拉已从后端读取：

```python
list_output_modes() -> modes
```

Qt 执行 `run_plan` 后会进入 `apply_output` 阶段。输出阶段已经改为调用 `WorkflowServices.write_table/export_xlsx`；如果缺少 `db_path/output_path`，会返回结构化 issue，不会把未落地输出误报为成功。

## 7.1 权限系统：AccessPolicyService

已落地第一版结构：

```python
build_table_access(node) -> {table_access, issues}
precheck_access(plan, output_settings, db_path, policy) -> {
  "ok": true,
  "can_continue": true,
  "requires_confirmation": false,
  "issues": [],
  "summary": "",
  "blocking_count": 0
}
format_access_issue(issue) -> text
record_access_audit(event) -> result
list_access_audit_logs(status, keyword) -> result
```

第一版目标不是替换权限执行器，而是把“默认权限规格、执行前预检、输出写入风险、审计日志展示数据”从 Tk/Qt 界面中抽出来。服务会把 `node_type_id` 计划转换为旧节点显示名再复用现有纯 helper，因此不需要节点逻辑改写，也不会让新界面依赖中文 `type`。

当前边界：

1. 支持 old `type` 和 new `node_type_id`。
2. 支持 SQLite 表存在性预检、输出模式预检、中转副表生产/读取顺序预检。
3. 支持 `audit/prompt/strict/off` 策略输出 `can_continue/requires_confirmation/blocking_count`。
4. 轻量审计日志先保存在进程内；后续如需跨进程/长期保留，再接持久化日志服务。
5. 插件声明读取先支持注入 `plugin_registry`，完整插件注册表加载留给 `PluginService`。

## 8. 后台任务与事件：JobService

Qt/Web 都不应直接阻塞执行工作流。推荐统一 job/event 模型。

接口：

```python
start_job(action, payload) -> job_id
cancel_job(job_id) -> result
get_job_status(job_id) -> status
get_job_result(job_id) -> result
iter_job_events(job_id) -> events
```

事件：

```json
{
  "type": "node_start",
  "job_id": "job_001",
  "node_index": 3,
  "node_name": "批量替换",
  "node_type_id": "core.replace",
  "message": "开始执行节点 4.批量替换"
}
```

Qt 用信号更新进度条。Web 用 WebSocket/SSE。stdio 可先用同步响应，后续扩展 event stream。

## 9. 统一 Issue / Error Schema

校验、配置强转、执行、输出、权限预检、跳转预检应统一 issue 结构，避免每个前端写多套错误渲染器。

推荐结构：

```json
{
  "severity": "error",
  "code": "missing_field",
  "message": "字段不存在：A",
  "node_index": 2,
  "node_id": "node_001",
  "node_type_id": "core.replace",
  "path": "/nodes/2/config/target_field",
  "suggestion": "请选择当前表中存在的字段。",
  "details": {}
}
```

规则：

1. `severity` 固定为 `error/warning/info`。
2. `code` 用稳定英文机器码。
3. `message/suggestion` 可以中文显示。
4. `path` 使用 JSON Pointer，便于 UI 定位字段。
5. 抛异常只用于不可恢复错误；可展示问题统一返回 `issues`。

## 10. 跳转、权限、插件、多表服务

这些当前还偏旧 Tkinter/window 体系，应逐步服务化。

### 10.1 JumpAnalysisService

已落地第一版：

```python
analyze_jumps(plan) -> {anchors, relations, issues, summary}
validate_jumps(plan) -> {ok, issues, summary, counts}
format_jump_issue(issue) -> text
```

当前已支持旧 `type` 和新 `node_type_id` 计划，Qt “校验”会合并显示跳转 issue。

后续完整形态：

```python
collect_anchors(plan) -> anchors
collect_jump_relations(plan) -> relations
validate_jumps(plan) -> issues
preview_execution_path(plan, input_table=None) -> path
```

UI 只负责展示锚点、关系、问题详情。

### 10.2 AccessPolicyService

已落地第一版：

```python
build_table_access(node) -> {table_access, issues}
precheck_access(plan, context) -> {ok, can_continue, issues, summary}
format_access_issue(issue) -> text
record_access_audit(event) -> result
list_access_audit_logs(filter) -> logs
```

权限策略：

```text
audit
prompt
strict

```

后续完整形态需要把审计日志持久化、插件权限声明加载和权限配置编辑服务继续补齐。

### 10.3 PluginService

已落地第一版：

```python
list_plugins() -> plugins
get_plugin_schema(plugin_id) -> schema
make_plugin_default_config(plugin_id) -> config
```

当前已复用 `plugin_runtime.scanner.scan_plugins()`，并把 `module` 等 Python 对象过滤为 JSON-safe 元数据。`list_node_catalog/list_node_ui_schemas` 在 `include_unsupported=True` 时会合并扫描到的插件节点；插入 `plugin.xxx` 节点时由后端生成旧配置结构，包括 `config.plugin_id`、`config.params`、独立环境路径、输出与日志设置。

后续完整形态：

```python
run_plugin(plugin_id, input_table, params, context) -> result
validate_plugin_config(plugin_id, params, context) -> issues
```

插件 schema 已合并进后端 schema 输出，但 Qt 当前第一版仍以 `params` JSON 兜底；后续要继续做嵌套参数表单或插件专用表单，让 UI 不需要知道插件内部规则。

### 10.4 TableDataService

已落地第一版：

```python
list_tables(db_path=None) -> tables
load_table(source, db_path=None, table_name=None, path=None, limit=None, offset=0) -> TableData
get_table_page(table, offset=0, limit=None) -> {headers, rows, page}
```

当前支持 SQLite 表列表/读取、文件表读取、内联 table 分页、进程内 table handle 创建/分页/释放。Qt 的查看表下拉已可按数据库路径列出 SQLite 表并读取；HTTP/Web/stdio 后续可用 handle 避免反复传完整大表。

后续完整形态：

```python
list_tables() -> tables
get_table_schema(table_name) -> headers
load_table(source) -> TableData
save_table(target, table, mode) -> result
create_table_handle(table_or_source) -> handle
get_table_handle_page(handle, offset=0, limit=200) -> {headers, rows, total_rows}
release_table_handle(handle) -> result
```

用于高级筛选、写回、匹配值输出列名、多表字段选择。

跨进程前端不要默认传 20 万行完整 JSON。推荐：

1. 小表可内联 `headers/rows`。
2. 大表返回 `table_handle`。
3. UI 按页请求预览行。
4. 执行时传 handle 或数据源描述，由后端读取。

## 12. 实施路线

### 阶段 0：建立安全和兼容基线

先把红线和历史数据守住。

1. [已落地第一版] 定义统一 issue schema。
2. [已落地第一版] 定义 `SafetyPolicy`，明确 preview 零副作用。
3. 新增 `migrate_plan`，用真实 `plan/*.json` 做回归。
4. 明确双 UI 过渡策略：旧 Tkinter 在高级能力未迁完前继续作为完整能力入口，Qt 逐步追平。
5. engine/schema/command 测试必须 headless 可导入，不经 Tkinter/Qt。

### 阶段 1：先消除 UI 里的纯逻辑

低风险，高收益。

1. Qt 表单改为 schema-first 基础设施，并以 2~3 个节点试点。
2. 新增 `workflow/plan_commands.py`，把节点增删改移复制下沉。
3. Qt 改用 `apply_plan_command`。
4. 新增节点类型解析 facade，Qt 不再自己拼 `node_type_id`。
5. 把 `ui_qt/table_io.py` 能力迁到后端 `engine/table_io.py`。
6. 把计划读写迁到 `engine/plan_io.py`。

验收：

- Qt 功能不退化。
- plan command 有纯单测。
- Qt 测试仍通过。
- stdio 可选暴露 `apply_plan_command/load_table/load_plan`。
- 所有新增 engine/workflow 测试不依赖 Tkinter/Qt。

### 阶段 2：补齐模板、校验和表单 schema

1. [已落地第一版] 扩展 `node_ui_schema` 的字段类型和动态表达力。
2. [已落地第一版] 加 `validate_config`。
3. [已落地第一版] 计划模板列表、校验、迁移、保存全部后端化。
4. Qt 不直接 import `shared.atomic_json_utils`。

验收：

- 高级节点表单不再靠 Qt 写业务判断。
- 同一 schema 可被 Qt/stdio/Web 消费。

### 阶段 3：执行、输出、后台任务

1. [已落地最小版] 定义 `WorkflowServices`。
2. [已落地第二版] 定义 `OutputService`。
3. [已落地第一版] 定义 `JobService`。
4. [已落地第一版] Qt 执行计划改为后台 job。
5. 接入取消按钮和进度事件。
6. [已落地基础版] 输出设置真正落地。

验收：

## 13. 推荐下一步

建议从阶段 1 开始，第一刀做：

```text
schema-first-config-form-infra
```

原因：

1. 风险最低，只影响表单渲染基础设施。
2. 对“换界面”价值最大。
3. 能倒逼 `node_ui_schema` 继续完善。
4. 先选 2~3 个节点试点，不假设一次覆盖全部复杂节点。
5. Qt/Web/Electron 都会直接受益。

第二刀做：

```text
plan-command-service
```

原因：

1. 当前 Qt 里最多业务规则就是节点列表操作。
2. 纯函数好测。
3. 做完后 UI 状态管理会明显变薄。

第三刀做：

```text
plan-migration-service
```

原因：

1. 真实计划大量仍是旧中文 `type`。
2. 新 UI 和新 schema 都依赖稳定 `node_type_id`。
3. 迁移测试能提前发现旧模板兼容问题。

第四刀做：

```text
plan-table-io-service
```

原因：

1. 计划文件和表格文件解析属于通用后端能力。
2. 以后 stdio/HTTP 前端可以直接传路径或数据源描述。

## 14. 不建议现在做的事

暂时不建议：

1. 立即引入 C/C++。
2. 立即改成 Electron/Web。
3. 一次性重写所有旧 Tkinter 配置窗口。
4. 把所有服务都塞进 `HeadlessWorkflowEngine`。
5. 让 Qt 直接调用数据库、插件、文件副作用模块。

当前最重要的是把 Python 后端边界立稳。界面换不换、什么时候换，都应该建立在这套后端服务之上。

## 15. 最终目标形态

理想调用链：

```text
Qt/Web/Electron/.NET
  -> WorkflowFacade / StdioWorker / HTTP API
    -> PlanCommandService
    -> NodeCatalogService
    -> PlanIOService
    -> TableDataService
    -> JobService
      -> HeadlessWorkflowEngine
        -> WorkflowServices
          -> SQLite / Plugin / File / Export / Audit
```

UI 不再关心：

- plan 字段怎么拼。
- 节点怎么复制。
- 字段类型怎么推断。
- 表格文件怎么解析。
- 输出怎么落地。
- 插件怎么运行。
- 权限怎么预检。
- 跳转怎么分析。

UI 只关心：

- schema 怎么显示。
- command 什么时候发送。
- result/event 怎么展示。

这就是后续可以平滑替换界面的核心。
