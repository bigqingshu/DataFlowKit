我觉得你这次的收敛方向更合理。

最大的变化是：

```text
跳转锚点节点 = 纯定位节点
```

不负责：

```text
表映射
字段映射
数据转换
权限控制
数据处理
```

只负责：

```text
作为合法跳转目标
作为工作流结构锚点
作为可视化管理节点
```

这样以后逻辑非常清晰。

---

# 一、最终跳转体系

建议保留四种节点

```text
1. 无条件跳转节点
2. 条件判断节点
3. 条件跳转节点
4. 跳转锚点节点
```

职责如下：

---

## 1. 无条件跳转节点

作用：

```text
直接跳转到指定锚点
```

相当于：

```text
GOTO Anchor_A
```

执行流程：

```text
执行到当前节点
↓
检查目标锚点
↓
存在
↓
跳转
```

如果：

```text
未绑定锚点
锚点不存在
锚点被禁用
```

则：

```text
默认不跳转
继续执行后续节点
记录警告日志
```

---

### 配置界面

```text
无条件跳转节点

目标锚点：
[Anchor_A ▼]

状态：
√ 已绑定

[定位锚点]
[打开跳转管理]
```

---

## 2. 条件判断节点

作用：

```text
生成布尔结果
生成状态结果
生成数值结果
```

类似：

```text
error_count > 0
```

输出：

```text
TRUE
```

或者：

```text
status = "异常"
```

输出：

```text
ERROR
```

---

注意：

条件判断节点

```text
不负责跳转
```

只负责：

```text
计算结果
```

例如：

```text
输出变量：
check_result

值：
TRUE
```

---

### 示例

```text
异常表行数 > 0

输出：
has_error = TRUE
```

---

## 3. 条件跳转节点

作用：

```text
读取条件判断结果
决定跳转到哪个锚点
```

例如：

```text
has_error

TRUE → Anchor_Error

FALSE → Anchor_Normal
```

---

### 条件跳转节点不计算条件

只读取：

```text
变量
标志位
状态值
节点输出结果
```

---

例如：

```text
条件判断节点

输出：
status = ERROR
```

条件跳转节点：

```text
ERROR → Anchor_Error

SUCCESS → Anchor_Success

DEFAULT → 不跳转
```

---

### 默认规则

你提的这个很好：

```text
条件未映射
目标锚点不存在
```

统一：

```text
默认不跳转
继续执行
```

并记录：

```text
Warning:
Jump target not configured.
```

这样最安全。

---

# 4. 跳转锚点节点

作用：

```text
仅作为合法跳转目标
```

不参与：

```text
计算
表操作
权限
字段映射
```

---

例如：

```text
Anchor_Error

↓
异常处理节点

↓
人工复核节点
```

---

锚点节点本质：

```text
Label
```

类似：

```text
LABEL:
```

---

# 二、为什么保留“条件判断节点”

很多工作流喜欢做：

```text
条件跳转节点
↓
里面写条件
```

但你这种工具不建议。

因为以后会出现：

```text
条件A

条件B

条件C
```

多个地方都要复用。

---

拆开后：

```text
条件判断节点
↓
产生结果

条件跳转节点
↓
使用结果
```

好处：

```text
一个结果
多个跳转节点复用
```

例如：

```text
has_error
```

可以用于：

```text
跳转1

跳转2

日志节点

通知节点
```

---

# 三、跳转锚点设计

建议锚点有唯一ID

```json
{
  "id":"anchor_error",
  "name":"异常处理锚点",
  "enabled":true
}
```

---

运行时：

```text
条件跳转
↓
anchor_error
↓
定位锚点节点
↓
执行锚点后第一个节点
```

# 六、跳转关系可视化

建议类似：

```text
Jump_001
   │
   ├─TRUE─────► Anchor_Normal
   │
   └─FALSE────► Anchor_Error
```

---

复杂一点：

```text
Check_Status
      │
      ▼

Jump_Status
      │
 ┌────┼─────┐
 │    │     │
 ▼    ▼     ▼

Normal Error Review

 │      │      │
 ▼      ▼      ▼

A      B      C
```

# 七、跳转管理窗口设计

建议独立窗口。

---

## 左侧

```text

□ Anchor_Start
□ Anchor_Normal
□ Anchor_Error
□ Anchor_Review
```

---

## 中间

跳转关系表

```text
┌────────────┬──────────────┬──────────────┬────────┐
│ 跳转节点     │ 条件值         │ 目标锚点       │ 状态     │
├────────────┼──────────────┼──────────────┼────────┤
│ Jump_001   │ TRUE         │ Normal       │ √      │
│ Jump_001   │ FALSE        │ Error        │ √      │
│ Jump_002   │ REVIEW       │ Review       │ √      │
└────────────┴──────────────┴──────────────┴────────┘
```

---

## 右侧

```text
Jump_001

来源节点：

has_error

TRUE
↓
Anchor_Normal

FALSE
↓
Anchor_Error
```

# 十一、最终推荐结构

我建议最终定为：

```text
条件判断节点
    ↓
产生变量

条件跳转节点
    ↓
根据变量选择锚点

无条件跳转节点
    ↓
直接选择锚点

跳转锚点节点
    ↓
仅定位
```

然后新增：

```text
跳转关系管理器
```

负责：

```text
查看所有跳转关系
查看所有锚点
查看失效引用
查看循环引用
查看孤立锚点
可视化跳转结构
运行前校验
```

而且最关键的一点：

```text
锚点节点彻底去业务化
```

它不做任何映射、不做任何权限、不做任何计算。

表映射继续由：

```text
选定列写入指定表
高级筛选
插件节点
数据库节点
```

这些业务节点完成。

这样跳转系统和数据系统完全解耦，后面维护会轻松很多。
