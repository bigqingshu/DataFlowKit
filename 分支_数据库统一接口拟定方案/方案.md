这个方案完全成立，而且比上一版更清晰。

你现在的边界可以定死为：

```text
表节点映射系统不负责业务规则匹配。
表节点映射系统只负责：
1. 表绑定
2. 字段映射
3. 表级权限
4. 字段级权限
5. 统一读取接口
6. 统一写入接口
7. 写入模式管控
8. 操作日志输出
9. 给工作流进度条 / 调试窗口提供状态信息
```

也就是说：

```text
节点负责“怎么匹配、怎么筛选、怎么生成结果”
映射系统负责“能不能读、能不能写、写到哪里、写了多少、记录日志”
```

这个边界非常稳。

# 一、最终职责划分

## 1. 节点负责业务逻辑

例如：

```text
高级筛选节点：
负责筛选规则、匹配条件、输出筛选结果

选定列写入指定表节点：
负责选择来源列、目标表、匹配逻辑、生成待写入数据

插件节点：
负责复杂多表计算、输出结果表

条件跳转节点：
负责判断条件、决定跳转锚点
```

节点自己决定：

```text
哪些数据符合条件
哪些行要写
哪些字段要输出
哪些数据算异常
```

---

## 2. 表节点映射系统负责管控

映射系统只负责：

```text
当前节点是否允许读取这张表
当前节点是否允许写入这张表
是否允许新建表
是否允许追加
是否允许覆盖
是否只允许写入指定字段
字段名如何对齐
写入后记录日志
```

它不关心：

```text
为什么这行要写
为什么这列要更新
为什么这个数据匹配成功
```

这些全部交给节点。

# 二、权限分类建议

按照你最新思路，权限可以分成两层。

```text
表级权限
字段级权限
```

不再做复杂规则权限。

---

## 1. 表级权限

表级权限管大动作。

建议保留这些：

```text
read_table           允许读取表
write_table          允许写入表
create_table         允许新建表
append_rows          允许追加行
update_rows          允许更新已有行
clear_table          允许清空表数据
replace_table        允许替换整表
alter_schema         允许修改表结构 / 新增字段
delete_rows          允许删除行
drop_table           允许删除表
```

第一版可以只做常用的：

```text
read_table
write_table
create_table
append_rows
update_rows
clear_table
replace_table
```

`alter_schema / delete_rows / drop_table` 可以先作为高级危险权限，默认关闭。

---

## 2. 字段级权限

字段级权限只管“哪些字段能被写”。

建议保留这些：

```text
read_field           允许读取字段
write_field          允许写入字段
create_field         允许自动创建字段
protect_field        保护字段，禁止修改
```

简化成 GUI 可以是：

```text
☑ 可读
☑ 可写
☐ 可自动创建
☐ 保护字段
```

字段层面不再做复杂规则，例如：

```text
非空才写
匹配才写
状态为成功才写
```

这些都交给节点自己处理。

# 三、把“写入场景补充”和“推荐写入模式”重新分类

按照你现在的边界，写入模式可以分为三类：

```text
A. 表级写入模式
B. 行级写入模式
C. 字段级写入模式
```

其中规则匹配类不放进映射系统。

---

# A. 表级写入模式

这类需要表级权限。

## 1. 新建表写入

```text
模式：create_new
说明：目标表不存在时创建新表并写入数据。
所需权限：create_table + write_table
```

适合：

```text
插件输出新结果表
生成异常明细表
生成中转表
生成日志表
```

---

## 2. 替换整表

```text
模式：replace_table
说明：删除或重建目标表，按来源表结构重新写入。
所需权限：replace_table + write_table
```

风险高。

适合：

```text
临时结果表
中转结果表
结构可能变化的插件输出表
```

---

## 3. 清空保留结构写入

```text
模式：clear_keep_schema
说明：保留目标表字段结构，清空旧数据，再写入新数据。
所需权限：clear_table + write_table
```

适合：

```text
固定结构的结果表
每次运行刷新结果
```

---

## 4. 目标表结构保留补写

这个是你之前第 3 类的安全版本。

```text
模式：keep_schema_insert
说明：目标表字段结构不变，只按已有字段写入来源数据；来源多余字段忽略或报错。
所需权限：write_table
```

适合：

```text
目标表格式固定
不允许插件改变字段结构
```

---

# B. 行级写入模式

这类主要管行操作。

## 5. 追加写入

```text
模式：append
说明：将来源数据追加到目标表数据边界后。
所需权限：append_rows
```

适合：

```text
日志表
异常表
历史记录表
多次运行累计结果
```

---

## 6. 覆盖写入

```text
模式：overlay
说明：从指定位置开始覆盖目标表对应区域，来源边界外的数据不影响。
所需权限：update_rows + write_table
```

适合：

```text
当前表局部覆盖
指定区域更新
保留目标表边界外数据
```

注意：这个不做条件判断，只执行节点已经准备好的数据。

---

## 7. 按主键更新

```text
模式：update_by_key
说明：根据指定键字段定位目标行，更新允许写入的字段。
所需权限：read_table + update_rows + write_table
字段要求：key_fields 可读，update_fields 可写
```

适合：

```text
根据编码回填
根据文件名+行列更新状态
根据唯一ID更新字段
```

节点负责生成要更新的数据，映射系统只负责检查权限和执行。

---

## 8. 匹配更新或追加

```text
模式：upsert_by_key
说明：匹配到则更新，匹配不到则追加。
所需权限：read_table + update_rows + append_rows
字段要求：key_fields 可读，update_fields 可写
```

适合：

```text
缓存表
中转表
插件结果表
累计但需要去重的记录表
```

---

# C. 字段级写入模式

这类主要管“只允许写哪些列”。

## 9. 指定字段写入

```text
模式：write_fields_only
说明：只允许写入字段权限中勾选的字段，其他字段忽略或报错。
所需权限：write_table + 字段 write_field
```

适合：

```text
只更新状态列
只回填结果列
只写入新值列
```

这是你现在最应该保留的字段级核心能力。

---

## 10. 字段补齐写入

```text
模式：fill_blank_fields
说明：只向目标表指定字段的空白位置写入。
所需权限：write_table + 字段 write_field
```

这个是否保留要看你需求。

如果你想让“空缺补齐”作为节点逻辑，那可以不放进映射系统。

但如果它只是简单写入策略，也可以保留。

我建议：

```text
第一版保留为写入模式，但不支持复杂条件。
只判断目标字段是否为空。
```

---

## 11. 字段结构对齐写入

```text
模式：align_fields_write
说明：来源字段通过字段映射写入目标字段，不匹配字段忽略或报错。
所需权限：write_table + 字段 write_field
```

这个实际上是所有写入模式的基础，不一定作为单独模式显示。

---

# 四、建议最终写入模式总表

按照你的最新边界，推荐保留这些。

| 分类  | 模式                   | 中文名      | 主要权限                        | 说明         |
| --- | -------------------- | -------- | --------------------------- | ---------- |
| 表级  | `create_new`         | 新建表写入    | create_table + write_table  | 创建新表并写入    |
| 表级  | `replace_table`      | 替换整表     | replace_table + write_table | 重建目标表，风险高  |
| 表级  | `clear_keep_schema`  | 清空保留结构写入 | clear_table + write_table   | 清空数据，保留字段  |
| 表级  | `keep_schema_insert` | 保留结构写入   | write_table                 | 只按目标表已有字段写 |
| 行级  | `append`             | 追加写入     | append_rows                 | 追加到目标表末尾   |
| 行级  | `overlay`            | 覆盖写入     | update_rows + write_table   | 覆盖来源边界内数据  |
| 行级  | `update_by_key`      | 按键更新     | read_table + update_rows    | 按主键/匹配字段更新 |
| 行级  | `upsert_by_key`      | 匹配更新或追加  | update_rows + append_rows   | 有则更新，无则追加  |
| 字段级 | `write_fields_only`  | 指定字段写入   | write_field                 | 只写允许字段     |
| 字段级 | `fill_blank_fields`  | 字段空缺补齐   | write_field                 | 只写目标字段空白处  |

第一版建议只做：

```text
read_table
append
overlay
clear_keep_schema
replace_table
update_by_key
write_fields_only
```

然后再加：

```text
create_new
upsert_by_key
fill_blank_fields
```

# 五、默认权限策略

你提到：

```text
对当前表，则默认每个节点都可完全写入读取，记录即可。
默认节点无写入权限。
```

这里可以这样设计。

## 1. 当前表默认权限

如果节点操作的是“当前表”，例如：

```text
删除列
复制列
合并列
高级筛选输出到当前表
```

默认：

```text
当前表：允许读取 + 允许写入
系统自动记录日志
不强制打开映射系统
```

也就是说：

```text
当前表操作 = 默认授权 + 强制记录
```

---

## 2. 非当前表默认权限

如果节点要操作其他表，例如：

```text
选定列写入指定表
高级筛选读取其他表
插件读取多个表
文档写入节点输出到指定表
```

默认：

```text
非当前表：只读可按节点声明开放，写入默认禁止
```

如果需要写入，必须进入：

```text
表节点映射系统
```

进行授权。

---

## 3. 默认规则

建议定成：

```text
当前表：
  默认可读、可写、可更新
  危险操作仍需节点本身确认
  映射系统只记录日志

非当前表：
  默认无写入权限
  读取权限由节点类型决定
  写入必须显式授权
```

这样很安全。

# 七、统一接口设计

你说“所有节点都需要通过这个函数进行写入，统一接口”，这个非常重要。

建议主程序内部所有表操作都走同一个入口。

可以抽象为：

```text
TableAccessManager
```

核心函数：

```text
read_table(node_id, table_name, fields=None, purpose=None)

write_table(node_id, target_table, data, write_mode, field_mapping=None)

create_table(node_id, table_name, schema)

append_rows(node_id, table_name, rows)

update_rows(node_id, table_name, rows, key_fields)

replace_table(node_id, table_name, data)

clear_table(node_id, table_name)
```

但对节点暴露时，可以只暴露一个统一接口：

```text
table_manager.read(...)
table_manager.write(...)
```

所有节点禁止直接访问数据库写入函数。

---

## 写入统一流程

```text
节点准备好结果数据
    ↓
调用 table_manager.write()
    ↓
检查节点ID
    ↓
检查目标表是否当前表
    ↓
检查表级权限
    ↓
检查字段级权限
    ↓
应用字段映射
    ↓
根据 write_mode 执行
    ↓
写入日志
    ↓
返回写入结果
```

# 十、条件跳转节点如何使用这套系统

条件跳转节点不需要额外设计表权限。

它只要调用：

```text
table_manager.read()
```

读取它需要判断的数据。

例如：

```text
条件：异常表行数 > 0
```

执行流程：

```text
条件跳转节点
    ↓
调用 table_manager.read(node_id, "异常表")
    ↓
映射系统检查：该节点是否允许读取异常表
    ↓
读取成功
    ↓
条件跳转节点自己计算 row_count > 0
    ↓
跳转到指定锚点
    ↓
日志记录：读取异常表 10 行，跳转 anchor_error_review
```

也就是说：

```text
条件怎么判断，是条件跳转节点自己的事。
能不能读取这张表，是表节点映射系统的事。
```

非常清楚。

# 十二、表级权限和写入模式校验关系

统一接口执行时，只要做简单校验。

```text
write_mode = append
必须有 append_rows

write_mode = update_by_key
必须有 read_table + update_rows

write_mode = replace_table
必须有 replace_table

write_mode = clear_keep_schema
必须有 clear_table + write_table

write_mode = create_new
必须有 create_table + write_table

write_mode = write_fields_only
必须有 write_table + 对应字段 write_field
```

字段权限只做这个判断：

```text
目标字段没有 write_field → 不允许写
目标字段 protect_field → 不允许写
```

# 十四、这个系统作为主程序核心是合理的

你这个想法非常对：

```text
所有节点读写表，都通过这个系统。
```

这样后续你会得到几个好处。

## 1. 日志统一

所有节点读写都能记录。

不用每个节点自己写日志。

---

## 2. 进度统一

工作流进度条可以直接用表访问日志。

---

## 3. 权限统一

不会出现某个插件绕过主程序直接写表。

## 4. 调试统一

出错时可以看到：

```text
哪个节点
读了哪张表

写了多少行
哪些字段被写入
是否权限不足
```

# 十五、最终收敛版结论

你现在的设计可以收敛成：

```text
主程序核心：TableAccessManager / 表节点映射系统

职责：
1. 所有节点读表必须经过它
2. 所有节点写表必须经过它
3. 当前表默认允许读写，但必须记录日志
4. 非当前表默认无写入权限
5. 非当前表写入必须通过节点下方“表节点映射系统”按钮授权
6. 字段映射只负责字段对齐
7. 字段权限只负责哪些字段能读、能写
8. 业务规则匹配全部由节点自己处理
9. 条件跳转节点只调用它读取数据，不由它判断条件
10. 该系统输出日志给主程序日志和工作流进度条
```

这版边界非常清楚，不会变成第二套工作流。

最推荐的开发顺序是：

```text
第一步：所有节点读写表统一走 TableAccessManager
第二步：记录基础日志：节点、表名、操作、行数、状态
第三步：当前表默认读写，只记录
第四步：非当前表写入默认禁止
第五步：增加节点下方“表节点映射系统”按钮
第六步：支持表级权限
第七步：支持字段级权限
第八步：支持写入模式校验
第九步：日志接入工作流进度条
第十步：条件跳转节点调用统一读取接口
```

这样就稳了。

# 十六、2026-06-11 当前更改记录

本轮更改围绕“字段权限层 / 插件权限声明 / 可视化映射写入计划”继续收束。

## 1. 插件权限声明预检

主程序增加了插件权限声明识别：

```text
plugin_needs_table_access_declaration
plugin_has_table_access_declaration
```

作用：

```text
1. 插件如果标记为 db_write / database_write，但没有声明表权限规格，权限预检会提示。
2. 插件如果声明动态写表范围为 * / %，并且拥有写权限，权限预检会提示范围过宽。
3. 插件返回 database_requests 时，仍由主程序 TableAccessManager 执行真实写库，插件不直接绕过权限层。
```

目前适合强制声明表权限的插件类型：

```text
1. 会直接请求 SQLite 写入的插件
2. 会返回 database_requests 的插件
3. danger_level 标记为 db_write / database_write 的插件
```

不需要纳入表权限声明的插件类型：

```text
1. 只读取主程序传入的内存表
2. 只生成写入计划表
3. 只写 Word/Excel 文件，不直接写 SQLite 表
```

这些插件仍由主程序插件节点管理输入表、输出中转表、输出 SQLite 表和插件日志表。

## 2. 可视化映射写入计划插件输入契约

`visual_mapping_write_plan_plugin_v1.py` 增加了输入契约校验：

```text
1. 文档读取表别名必须存在
2. 新内容表别名必须存在
3. 文档读取表必须包含 text 或 内容 字段
4. 新内容表必须至少有一个可用字段
```

这样可以更早发现：

```text
1. Word/Excel 读取插件输出没有正确传入
2. 插件节点输入表别名配置错误
3. 新内容表为空或字段未生成
```

失败时，插件直接返回明确错误，不再继续生成空写入计划。

## 3. 联动写入表字段上下文补齐

联动写入规则现在会把当前内容行和表上下文传给：

```text
1. 联动锚点定位
2. 联动目标格匹配
```

因此这些位置现在可以使用：

```text

2. 文档读取表字段

```

这与普通映射、全局替换中的字段来源逻辑保持一致。

## 4. 写回插件契约保持

写回插件已经优化过，本轮未改写回插件核心逻辑。

当前固定的写入计划字段契约为：

## 5. 关于“未解析的引用 context:11269”

原代码位置：

```python
return self.get_table_manager(context if 'context' in locals() else None).get_columns(name)
```

问题原因：

```text
build_group_node_config(config, headers, transit_context=None)
```

这个函数里没有名为 `context` 的局部变量，只有 `transit_context`。

`context if 'context' in locals() else None` 在运行时通常不会报错，因为 `locals()` 里没有 `context` 时会走 `else None`。
但是静态检查器仍然会认为 `context` 是未解析引用，所以出现：

```text
未解析的引用 'context'
```

同时，这段代码实际也永远不会把执行期上下文传进去，只会退化成 `None`。

已修复为：

```python
return self.get_workflow_sqlite_columns(
    name,
    context=transit_context if isinstance(transit_context, dict) else None,
)
```

修复后的好处：

```text
1. 静态检查器不再提示未解析引用
2. 配置界面读取 SQLite 字段时可以复用 transit_context / workflow_snapshot
3. 后台、预览、主线程读取 SQLite 字段的路径更统一
4. 不再使用 locals() 这种不稳定兜底写法
```

同时清理了其他同类写法：

```text
context if 'context' in locals() else None
```

这些位置所在函数本身已有 `context=None` 形参，因此统一改为直接使用 `context`，或在传入 `get_table_manager` 前判断 `isinstance(context, dict)`。
