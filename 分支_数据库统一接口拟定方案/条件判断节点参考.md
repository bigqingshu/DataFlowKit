晚上好。你这个思路是对的：**多表写入、权限、输出表管理、写入方式，最好由主程序统一管控；插件只负责“计算并输出结果包”**。这样安全性、可预览、可撤销、日志、异常处理都能集中管理，不然每个插件自己写表，很容易出现覆盖错表、结构不一致、旧数据污染、新旧版本插件行为不统一的问题。

我建议把整个设计分成三层：

```text
插件层：只读输入表 → 计算 → 输出一个或多个“结果表”
主程序调度层：检查插件声明、输入权限、输出表策略、写入模式
数据库写入层：预览、校验、备份、写入、日志、回滚
```

---

## 一、核心原则：插件不要直接写数据库

插件最好不要拿到完整数据库写权限，而是只拿到主程序分配给它的数据。

### 推荐模式

插件输入：

```python
input_tables = {
    "文档读取表": DataFrame,
    "新内容表": DataFrame,
    "辅助映射表": DataFrame,
}
```

插件输出：

```python
result = {
    "tables": {
        "输出表A": DataFrame,
        "输出表B": DataFrame,
    },
    "logs": [],
    "warnings": [],
    "metadata": {}
}
```

主程序再根据工作流 JSON 决定：

```json
{
  "node_type": "external_plugin",
  "plugin_name": "文档表格写入映射插件",
  "input_tables": {
    "read_table": "文档读取表",
    "new_data_table": "新内容表"
  },
  "outputs": {
    "输出表A": {
      "target_table": "文档写入中转表",
      "write_mode": "replace_keep_schema"
    },
    "输出表B": {
      "target_table": "异常明细表",
      "write_mode": "append"
    }
  }
}
```

这样插件只关心“算出来什么”，主程序关心“怎么安全写进去”。

# 二、写入场景补充

你目前想到的 4 种已经覆盖了大部分情况：

1. **空缺补齐写入**
2. **覆盖写入**
3. **目标表结构保留写入**
4. **追加写入**

我建议再补充这些常见场景。

---

## 1. 新建表写入

适合插件输出全新结果，例如：

```text
插件输出：文档差异结果表
目标表不存在 → 自动创建
目标表存在 → 根据策略报错/覆盖/改名
```

写入模式可以叫：

```text
create_new
```

安全选项：

```json
"if_exists": "error | overwrite | rename"
```

例如：

```json
{
  "write_mode": "create_new",
  "if_exists": "rename"
}
```

如果目标表已存在，就生成：

```text
文档差异结果表_20260608_001
```

---

## 2. 完全替换表写入

这个和你说的“目标表数据结构保留写入”不同。

你说的第 3 种是：

```text
保留目标表字段结构 → 清空数据 → 写入当前表
```

还需要一种：

```text
直接删除旧表 → 按输出表结构新建 → 写入
```

写入模式可以叫：

```text
replace_table
```

适合插件输出结构可能变化的表。

---

## 3. 按主键 / 匹配字段更新写入

这个非常重要。

例如目标表有：

| 编码 | 名称 | 型号 | 数量 |
| -- | -- | -- | -- |

插件输出：

| 编码 | 数量 |
| -- | -- |

希望根据“编码”匹配，只更新数量。

这种不是简单覆盖，也不是追加，而是：

```text
按指定字段匹配行 → 更新指定字段
```

写入模式：

```text
update_by_key
```

JSON 示例：

```json
{
  "write_mode": "update_by_key",
  "key_fields": ["编码"],
  "update_fields": ["数量"],
  "on_missing_key": "skip | append | error"
}
```

这个后面会非常有用，尤其适合 BOM、物料、文档定位表、字段回填表。

---

## 4. 插入或更新 Upsert

这是 `update_by_key` 的增强版。

逻辑是：

```text
匹配到 → 更新
匹配不到 → 追加
```

写入模式：

```text
upsert_by_key
```

适合中转数据表、缓存表、插件结果表。

```json
{
  "write_mode": "upsert_by_key",
  "key_fields": ["文件名", "表格号", "行", "列"],
  "update_fields": ["新值", "状态", "备注"]
}
```

---

## 5. 仅写入指定列

有时插件输出很多列，但目标表只想接收其中几列。

```json
{
  "write_mode": "update_columns",
  "target_table": "文档读取表",
  "columns": {
    "新内容": "写入值",
    "状态": "写入状态"
  }
}
```

这个适合“回填结果”，比如：

```text
原始表不动，只新增/更新几个结果列。
```

---

## 6. 仅写入指定区域

类似 Excel 思路：

```text
从目标表第 N 行、第 M 列开始写入
```

写入模式：

```text
write_range
```

可用于表格结构固定的场景。

```json
{
  "write_mode": "write_range",
  "start_row": 2,
  "start_col": "D",
  "expand_rows": true,
  "expand_cols": false
}
```

不过 SQLite 表不天然适合二维单元格区域，所以这个更适合你已经把数据抽象成“可视化表格”的场景。

---

## 7. 差异写入 / 只写变化项

先比较目标表和输出表，只写入不同的地方。

```text
目标值 == 新值 → 不写
目标值 != 新值 → 写入并记录日志
```

写入模式：

```text
diff_update
```

适合你这种文档、BOM、确认函、校对结果场景。

优点是日志很清楚：

```text
文件A.docx 表格2 行3列5：旧值 xxx → 新值 yyy
```

---

## 8. 条件写入

满足条件才写。

例如：

```text
只写入 状态 = "有效" 的行
只写入 新值不为空 的行
只写入 匹配结果 != "未匹配" 的行
```

写入模式可以不单独作为模式，而是作为所有写入模式的附加条件：

```json
"row_filter": {
  "field": "状态",
  "op": "!=",
  "value": "未匹配"
}
```

---

## 9. 冲突隔离写入

当目标表已有数据冲突时，不直接覆盖，而是输出到冲突表。

```text
匹配到但值不同 → 不覆盖 → 写入冲突明细表
```

这个很适合安全场景。

```json
{
  "write_mode": "upsert_by_key",
  "key_fields": ["编码"],
  "on_conflict": "write_to_conflict_table",
  "conflict_table": "编码冲突明细"
}
```

---

## 10. 临时表 / 中转表写入

有些插件输出不应该直接进入正式表，而是先进入中转表。

```text
插件结果 → 中转表 → 人工确认/预览 → 正式表
```

写入模式：

```text
write_to_staging
```

或者作为表属性：

```json
"is_staging": true
```

这个非常适合你的 90/10 思路：自动处理 90%，剩下异常导出复核。

---

# 三、推荐的写入模式总表

可以先定义这些枚举：

```text
append                  追加写入
fill_blank              空缺补齐写入
overlay                 边界内覆盖写入
clear_keep_schema       清空数据，保留目标表结构
replace_table           删除旧表，按输出结构重建
create_new              新建表
update_by_key           按主键/匹配字段更新
upsert_by_key           有则更新，无则追加
update_columns          只更新指定列
write_range             指定区域写入
diff_update             只写差异项
write_to_staging        写入中转表
```

你当前 4 个可以对应成：

```text
空缺补齐写入 → fill_blank
覆盖写入 → overlay
目标表数据结构保留写入 → clear_keep_schema
追加写入 → append
```

# 四、输出表管理建议：每个输出表带写入属性

你这个想法非常对。

建议工作流 JSON 中，每个输出表都独立配置：

```json
"outputs": {
  "plugin_result_table_1": {
    "target_table": "正式目标表A",
    "write_mode": "upsert_by_key",
    "key_fields": ["文件名", "表格号", "行", "列"],
    "update_fields": ["新值", "状态"],
    "backup_before_write": true,
    "preview_before_write": true,
    "on_error": "stop"
  },
  "plugin_result_table_2": {
    "target_table": "异常明细表",
    "write_mode": "append",
    "backup_before_write": false,
    "preview_before_write": true,
    "on_error": "continue"
  }
}
```

这样一个插件可以输出多个表，但每个表怎么落库由主程序决定。

---

## 主程序应该管控的内容

主程序负责：

```text
1. 插件能读取哪些表
2. 插件输出了哪些表
3. 输出表是否允许写入目标表
4. 写入模式是否合法
5. 字段结构是否匹配
6. 是否需要备份
7. 是否需要预览
8. 是否允许覆盖
9. 是否记录日志
10. 是否允许继续执行后续节点
```

插件不应该自己决定：

```text
我要覆盖哪个正式表
我要删除哪个表
我要清空哪个表
我要直接写数据库
```

除非你以后给“高级插件”开放权限，但默认不建议。

# 五、插件声明机制

插件可以声明自己需要什么输入、会输出什么，但不负责落库。

例如插件内部提供：

```python
PLUGIN_MANIFEST = {
    "name": "文档表格映射插件",
    "version": "1.0.0",
    "inputs": {
        "read_table": {
            "type": "table",
            "required": True,
            "access": "read_only"
        },
        "new_data_table": {
            "type": "table",
            "required": True,
            "access": "read_only"
        }
    },
    "outputs": {
        "write_plan_table": {
            "type": "table",
            "description": "文档写入计划表"
        },
        "error_table": {
            "type": "table",
            "description": "异常明细表"
        }
    }
}
```

主程序读取这个声明后，在节点配置界面渲染：

```text
输入表绑定：
read_table      → 选择数据库表
new_data_table  → 选择数据库表

输出表绑定：
write_plan_table → 目标表 + 写入方式
error_table      → 目标表 + 写入方式
```

这样插件节点界面会非常清晰。

---

# 六、多表操作权限建议

可以分成 4 种权限：

```text
read_only       只读
write_staging   只能写中转表
write_target    允许写指定目标表
admin_write     允许建表、删表、清空表
```

普通插件默认：

```text
read_only + 输出结果包
```

主程序负责写入，所以插件本身不需要 `write_target`。

只有极少数特殊插件才给高级权限。

---

# 七、写入前最好统一进入“写入计划表”

我建议无论插件输出什么，都先转换成主程序内部的“写入计划”。

例如：

```text
插件输出表
    ↓
主程序解析 outputs 配置
    ↓
生成写入计划
    ↓
预览写入影响
    ↓
用户确认 / 自动执行
    ↓
实际写数据库
```

写入计划可以包含：

| 序号 | 输出表 | 目标表 | 写入方式 | 影响行数 | 新增行数 | 更新行数 | 冲突数 | 状态 |
| -- | --- | --- | ---- | ---- | ---- | ---- | --- | -- |

进一步点开可以看明细：

| 目标表 | 行号/主键 | 字段 | 旧值 | 新值 | 操作 |
| --- | ----- | -- | -- | -- | -- |

这样安全感会高很多。

# 八、工作流跳转方案

你说得很准确：你的工作流已经越来越像“汇编 / 流程图 / 数据管线”的混合体。

但是它和汇编最大的区别是：

```text
汇编跳转基于寄存器/标志位/地址
你的工作流跳转基于表、字段、条件结果、节点状态
```

所以我建议不要直接模拟汇编的 `jmp / je / jne`，而是设计成：

```text
无条件跳转节点
条件判断节点
条件跳转节点
循环控制节点
标志位/变量节点
```

---

## 1. 无条件跳转节点

最简单：

```text
执行到这里 → 跳到指定节点
```

节点配置：

```json
{
  "node_type": "jump",
  "target_node_id": "node_008"
}
```

用途：

```text
跳过某些节点
流程分段
测试某一段流程
异常后跳到收尾节点
```

---

## 2. 条件判断节点

这个节点只负责算条件，输出一个布尔结果：

```text
true / false
```

例如：

```json
{
  "node_type": "condition",
  "condition": {
    "left": {
      "table": "表A",
      "field": "编码"
    },
    "op": "in",
    "right": {
      "table": "表B",
      "field": "编码"
    }
  },
  "output_flag": "has_matched_code"
}
```

执行后产生：

```text
变量 has_matched_code = true
```

或者输出一张结果表：

| 条件名              | 结果   | 命中数量 |
| ---------------- | ---- | ---- |
| has_matched_code | true | 15   |

---

## 3. 条件跳转节点

条件跳转节点读取条件结果：

```json
{
  "node_type": "conditional_jump",
  "flag": "has_matched_code",
  "if_true": "node_010",
  "if_false": "node_020"
}
```

这样结构更清晰：

```text
条件判断节点：负责算
条件跳转节点：负责跳
```

不要一开始就把判断和跳转全部塞在一个节点里，否则后面调试很痛苦。

---

# 九、条件表达式设计

你想到的“列等于列、有则为一、多个条件、与或非、括号顺序”是对的。

但我建议分阶段做，不要一上来做完整表达式解析器。

---

## 第一阶段：条件组模式

先做 GUI 好配置的结构化条件。

例如：

```text
条件组 A：
  条件1：表A.编码 等于 表B.编码
  条件2：表A.数量 大于 0
  条件3：表A.状态 不包含 未匹配

组内关系：全部满足 AND
最终结果：true / false
```

JSON：

```json
{
  "logic": "AND",
  "conditions": [
    {
      "left": {"table": "表A", "field": "编码"},
      "op": "exists_in",
      "right": {"table": "表B", "field": "编码"}
    },
    {
      "left": {"table": "表A", "field": "数量"},
      "op": ">",
      "right": {"value": 0}
    },
    {
      "left": {"table": "表A", "field": "状态"},
      "op": "not_contains",
      "right": {"value": "未匹配"}
    }
  ]
}
```

这个阶段就已经能解决 80% 场景。

---

## 第二阶段：条件组嵌套

支持：

```text
(A AND B) OR (C AND NOT D)
```

JSON：

```json
{
  "logic": "OR",
  "groups": [
    {
      "logic": "AND",
      "conditions": [
        {"left": "表A.编码", "op": "exists_in", "right": "表B.编码"},
        {"left": "表A.状态", "op": "!=", "right": "未匹配"}
      ]
    },
    {
      "logic": "AND",
      "conditions": [
        {"left": "表A.数量", "op": ">", "right": 0},
        {
          "not": {
            "left": "表A.备注",
            "op": "contains",
            "right": "禁用"
          }
        }
      ]
    }
  ]
}
```

GUI 上可以做成：

```text
条件组
├─ 条件1
├─ 条件2
├─ 子条件组
│  ├─ 条件3
│  └─ 条件4
```

这个比手写括号表达式安全。

---

## 第三阶段：表达式文本模式

等前两个阶段稳定后，再加高级模式：

```text
(A.编码 in B.编码 AND A.状态 != "未匹配") OR A.数量 > 0
```

但这个要做解析、转义、字段名处理、错误提示，复杂度更高。

所以建议：

```text
先做结构化条件组，不急着做手写表达式。
```

# 十、条件判断的返回粒度

这个很关键。

条件判断不一定只返回一个 true/false。可以有三种返回方式。

---

## 1. 全局布尔

例如：

```text
只要有任意一条匹配 → true
完全没有匹配 → false
```

适合流程跳转：

```text
如果有异常 → 跳到异常处理节点
如果没有异常 → 继续正式写入
```

---

## 2. 行级布尔

每一行单独判断。

输出：

| 编码   | 数量 | 条件结果  |
| ---- | -- | ----- |
| A001 | 1  | true  |
| A002 | 0  | false |

适合筛选、分流、分支处理。

---

## 3. 统计型布尔

例如：

```text
命中数量 > 0
异常数量 = 0
匹配率 >= 95%
空值数量 <= 3
```

JSON：

```json
{
  "condition_type": "aggregate",
  "source_table": "异常表",
  "aggregate": "count",
  "op": "==",
  "value": 0
}
```

这个对工作流非常有用。

例如：

```text
如果异常表行数 = 0 → 自动写入正式表
如果异常表行数 > 0 → 跳到人工复核节点
```

---

# 十一、推荐的条件运算符

可以先支持这些。

## 字符串类

```text
==
!=
contains
not_contains
startswith
endswith
regex_match
is_empty
is_not_empty
```

## 数值类

```text
>
>=
<
<=
between
not_between
```

## 表关系类

```text
exists_in          A列值存在于B列
not_exists_in      A列值不存在于B列
row_count > N      表行数判断
duplicate_exists   是否存在重复
all_matched        是否全部匹配
any_matched        是否存在匹配
```

## 集合类

```text
in
not_in
intersects         两列有交集
no_intersection    两列无交集
```

## 状态类

```text
node_success
node_failed
node_has_warning
table_exists
table_not_exists
field_exists
field_not_exists
```

这些足够覆盖很多工作流场景。

# 十二、跳转和循环要防死循环

一旦加跳转，就必须加安全限制。

建议主程序有：

```json
{
  "max_total_steps": 1000,
  "max_loop_count_per_node": 50,
  "on_loop_limit": "stop"
}
```

每个节点执行时记录：

```text
node_id 执行次数
总执行步数
当前跳转来源
当前跳转目标
```

如果某个节点被执行太多次：

```text
提示：疑似死循环，已停止工作流
```

---

# 十三、工作流执行模型建议

你现在可以把工作流从“顺序列表”升级成“节点图”。

当前可能是：

```text
node1 → node2 → node3 → node4
```

升级后：

```text
node1 → node2 → condition
                 ├─ true  → node3
                 └─ false → node5
```

内部 JSON 可以这样：

```json
{
  "nodes": [
    {"id": "node_001", "type": "plugin"},
    {"id": "node_002", "type": "condition"},
    {"id": "node_003", "type": "write"},
    {"id": "node_004", "type": "manual_review"}
  ],
  "edges": [
    {"from": "node_001", "to": "node_002"},
    {"from": "node_002", "to": "node_003", "condition": "true"},
    {"from": "node_002", "to": "node_004", "condition": "false"}
  ]
}
```

但 GUI 上不一定要马上做成 ComfyUI 那种拖线界面。可以先做成：

```text
节点列表 + 每个节点配置“下一节点”
```

例如：

| 节点    | 类型 | 成功后   | 失败后   |
| ----- | -- | ----- | ----- |
| 读取表   | 插件 | 条件判断  | 停止    |
| 条件判断  | 条件 | 写入正式表 | 写入异常表 |
| 写入正式表 | 写入 | 结束    | 停止    |

这比直接做可视化连线简单很多。

# 十四、建议新增几个核心节点

## 1. 表写入节点

专门负责把某个中转表写到目标表。

```text
输入：来源表
配置：目标表、写入方式、字段映射、匹配字段
输出：写入日志表
```

这样插件节点不直接写，插件后面接写入节点即可。

---

## 2. 条件判断节点

```text
输入：一个或多个表
配置：条件组
输出：布尔结果 / 行级结果表 / 统计结果
```

---

## 3. 条件跳转节点

```text
输入：条件结果
配置：true 去哪里，false 去哪里
```

---

## 4. 标志位节点

用于保存临时变量：

```text
has_error = true
match_count = 15
current_file = xxx.docx
```

---

## 5. 异常分流节点

这是你工具很适合做的节点：

```text
正常数据 → 正式输出表
异常数据 → 异常复核表
```

例如：

```text
匹配成功的数据 → 写入计划表
未匹配的数据 → 人工复核表
```

---

## 6. 人工确认节点

这个对安全写入很重要。

```text
执行到这里暂停
显示预览
用户点击继续后再执行后续节点
```

可以用于：

```text
大量覆盖写入前
文档批量修改前
数据库结构变更前
```

---

# 十五、你这个系统的推荐执行闭环

我建议后续流程变成这样：

```text
1. 插件读取多个输入表
2. 插件输出多个结果表
3. 主程序生成写入计划
4. 条件判断是否有异常
5. 无异常 → 写入正式表
6. 有异常 → 写入异常表 / 跳到人工复核
7. 写入完成后生成日志
8. 后续节点读取日志继续判断
```

示例：

```text
文档读取表 + 新内容表
        ↓
外置插件：生成写入计划表、异常表
        ↓
条件判断：异常表行数 == 0 ?
        ├─ 是 → 写入正式文档写入表
        └─ 否 → 写入异常复核表，停止或等待人工处理
```

这个和你的 90/10 思路非常契合。

# 十八、优先级建议

不要一次做太大。按你的风格，我建议这样分阶段。

## 第一阶段：多表输出和写入策略

先完成：

```text
插件可输出多个表
每个输出表可配置目标表
每个输出表可配置写入模式
主程序统一写入
写入前预览
写入日志
```

先支持 5 种写入方式：

```text
append
overlay
clear_keep_schema
replace_table
upsert_by_key
```

这 5 个优先级最高。

---

## 第二阶段：条件判断

先支持：

```text
表是否存在
字段是否存在
表行数判断
列是否有空值
列是否包含指定值
A列是否存在于B列
异常表行数是否为0
```

先不用做复杂括号表达式。

---

## 第三阶段：条件跳转

加入：

```text
无条件跳转
条件跳转
成功/失败/警告后跳转
最大执行次数限制
```

---

## 第四阶段：条件组嵌套

再做：

```text
AND / OR / NOT
条件组嵌套
括号优先级
表达式预览
```

---

## 第五阶段：可视化流程图

最后再考虑：

```text
节点拖拽
节点连线
流程图布局
类似 ComfyUI 的界面
```

这个不要太早做，否则 UI 会拖慢核心逻辑。

│  ├─ 插件参数
│  ├─ 插件输出结果包
│  └─ 插件异常隔离
│
├─ 写入计划器
│  ├─ 输出表 → 目标表
│  ├─ 写入模式
│  ├─ 字段映射
│  ├─ 冲突处理
│  └─ 写入预览
│
├─ 条件引擎
│  ├─ 条件组
│  ├─ 表关系判断

## 我的建议结论

你现在这个判断很关键：**插件不应该直接管理写入，写入策略应该由主程序统一控制。**

比较稳的路线是：

```text
插件负责：多表读取后的复杂计算、匹配、生成结果表
主程序负责：输出表落库、写入方式、权限、安全、预览、日志、跳转
```

条件跳转也不要一开始就做成复杂表达式语言，建议先做：

```text
条件判断节点 → 输出 true/false 或结果表
条件跳转节点 → 根据结果跳到指定节点
```

这样最稳，后面也容易扩展成完整的条件表达式系统。
