# 插件协议字段命名收口建议

## 1. 文档目的

本文用于把当前 DataFlowKit 插件协议里已经出现的字段命名做一次收口，目标是：

1. 减少同一语义出现多套写法。
2. 降低 Qt / Tkinter / .NET / Web 各端各自兼容的成本。
3. 给后续插件改造、模板更新、UI 承接实现提供统一口径。

这份文档不是推翻现有实现，而是基于当前代码现状，明确：

1. 哪些命名可以直接定为主口径。
2. 哪些命名应视为历史兼容别名。
3. 哪些字段需要逐步补齐。

---

## 2. 当前代码现状判断

从当前代码看，已经形成了几层事实标准：

1. 插件参数入口以 `get_parameter_schema()` 为主。
2. 复杂插件配置入口以 `describe_config()` 为主。
3. 局部配置修改以 `validate_config_patch()` / `apply_config_patch()` 为主。
4. `engine/plugin_service.py` 已经对部分字段做了标准化映射。
5. `ui_qt/config_form.py` 已经支持一部分动态状态与结构化列表承接。

所以现在不是“没有标准”，而是“标准已经长出来了，但还没被正式命名并彻底统一”。

---

## 3. 参数 schema 字段建议主口径

### 3.1 参数标识

推荐主口径：

1. `name`
2. `label`
3. `type`

兼容别名：

1. `title` 可兼容映射到 `label`
2. 不建议继续新增 `key` 作为插件参数 schema 主字段名

建议：

- 插件作者层面，后续统一写 `name / label / type`。
- UI / service 层继续兼容旧别名，但不再把别名当主文档口径。

### 3.2 默认值与帮助

推荐主口径：

1. `default`
2. `help`
3. `required`

兼容别名：

1. `description` 可兼容映射到 `help`

建议：

- 后续模板与插件文档中统一使用 `help`。
- `description` 只保留兼容读取。

### 3.3 枚举项

推荐主口径：

1. `choices`

兼容别名：

1. `options` 可兼容映射到 `choices`

建议：

- 新插件统一写 `choices`。
- `options` 不再作为推荐公开字段名。

### 3.4 自定义输入能力

推荐主口径：

1. `allow_custom`

建议：

- 动态选择、字段选择、配置名选择等允许用户保留旧值或手输时，统一使用 `allow_custom`。

---

## 4. 参数类型命名建议主口径

当前 `engine/plugin_service.py` 已经做了一层映射，因此后续建议把“对外推荐写法”和“内部标准控件类型”分开描述。

### 4.1 对插件作者推荐的原始类型

推荐继续支持并主推：

1. `text`
2. `textarea`
3. `number`
4. `bool`
5. `select`
6. `field_select`
7. `multi_field_select`
8. `table_select`
9. `input_table_select`
10. `dynamic_select`
11. `folder_path`
12. `path`
13. `json`

原因：

1. 这套命名已经在现有插件里广泛使用。
2. 不需要大面积回改现有插件。
3. `plugin_service` 已经能做归一化。

### 4.2 UI / service 内部标准控件类型

建议内部标准化为：

1. `text`
2. `long_text`
3. `number`
4. `bool`
5. `choice`
6. `field_select`
7. `field_multi_select`
8. `table_select`
9. `directory`
10. `path`
11. `json`
12. `structured_list`

说明：

- 对插件作者层，不必强推这套内部名字。
- 对 UI 承接层，应尽量统一成这套控件语义，避免不同端再各自定义一次。

### 4.3 历史兼容映射建议

建议保留映射：

1. `folder_path -> directory`
2. `dynamic_select -> select`
3. `input_table_select -> select`
4. `multi_field_select -> field_multi_select`
5. `field -> field_select`

但文档中要明确：

- 这是**兼容映射**，不是未来对外建议命名的主口径。

---

## 5. 参数 UI 元数据字段建议主口径

### 5.1 建议正式收口为主字段

后续建议固定使用：

1. `group`
2. `group_order`
3. `order`
4. `placeholder`
5. `warning`
6. `empty_text`
7. `invalid_value_text`
8. `visible_when`
9. `enabled_when`
10. `depends_on`
11. `refresh_on_change`
12. `options_source`
13. `advanced`
14. `width_hint`

这些字段的意义已经比较明确，问题主要是还没有全面写进现有插件 schema。

### 5.2 条件表达式主口径

当前 Qt 表单层已经支持的条件表达式结构，应正式收口为：

```json
{
  "field": "path_source",
  "equals": "插件参数=固定目录路径"
}
```

以及：

1. `all`
2. `any`
3. `not`
4. `equals`
5. `not_equals`
6. `in`
7. `not_in`
8. `truthy`

建议：

- 后续所有参数联动条件都按这套写，不再给每个 UI 各写一套条件规则。

### 5.3 候选来源主口径

推荐主口径：

1. `options_source.type = preview_headers`
2. `options_source.type = table_headers`
3. `options_source.type = table_names`
4. `options_source.type = plugin_input_tables`
5. `options_source.type = plugin_dynamic_choices`

说明：

- 这些名称已经和当前 `plugin_service` 的行为比较接近。
- 后续优先围绕它们补强，不要再长出另一套近义命名。

---

## 6. 复杂插件 `describe_config()` 字段命名建议主口径

### 6.1 顶层字段

建议固定为：

1. `schema_version`
2. `protocol_family`
3. `plugin_id`
4. `config_key`
5. `summary`
6. `views`
7. `actions`
8. `context`
9. `models`
10. `warnings`
11. `capabilities`

当前现状：

- `schema_version / summary / views / actions / context / models / warnings` 已基本成型。
- `protocol_family / plugin_id / config_key / capabilities` 还应进一步补齐和固定。

### 6.2 `views` 内部字段

建议固定为：

1. `view_id`
2. `title`
3. `kind`
4. `protocol`
5. `config_path`
6. `item_count`
7. `columns`
8. `items`
9. `summary`
10. `meta`
11. `editor_kind`

说明：

- 当前 `visual_mapping_write_plan_plugin_v1.py` 已大量使用这套命名。
- 后续应把它视为复杂插件 view 的正式口径。

### 6.3 `actions` 内部字段

建议固定为：

1. `action_id`
2. `label`
3. `kind`
4. `editor_kind`
5. `view_id`
6. `config_path`
7. `enabled`
8. `warning`

说明：

- `action_id` 与 `view_id` 都应是稳定 ID，不应依赖显示文案。

### 6.4 `warnings` 建议升级方向

当前有些地方还是纯文本 warning 列表，建议长期主口径升级为结构化对象：

1. `code`
2. `level`
3. `message`
4. `view_id`
5. `field`
6. `hint`

兼容策略：

- 短期可以继续接受字符串列表。
- 中长期应尽量改为对象列表。

---

## 7. patch 字段命名建议主口径

建议 patch 长期统一使用：

1. `schema_version`
2. `config_name`
3. `section`
4. `operation`
5. `path`
6. `payload`
7. `target_id`
8. `target_index`

当前现状：

- Qt 结构化列表按钮目前更偏轻量 patch，常见字段是 `operation / target / index / to_index / value / enabled`。

建议收口方式：

1. 短期兼容当前轻量 patch。
2. 中期逐步向 `path / payload / target_index` 这一套靠拢。
3. 不要求一次性重写所有调用点。

---

## 8. 对现有插件的直接命名建议

### 8.1 `visual_mapping_write_plan_plugin_v1.py`

建议：

1. 保持现有 `view_id / action_id / editor_kind / config_path` 路线。
2. 补齐 `protocol_family / plugin_id / config_key / capabilities`。
3. `warnings` 逐步从字符串列表升级为结构化对象。

### 8.2 `word_excel_read_to_db_plugin_v1.py`

建议：

1. 继续保留当前参数字段命名。
2. 新增时统一使用 `choices`，不再引入 `options`。
3. 后续显隐与候选来源统一补到 `visible_when / options_source`。

### 8.3 `word_excel_write_from_table_plugin_v2.py`

建议：

1. 保持现有参数字段名不变。
2. 将字段候选来源统一显式化。
3. 大参数量区统一补 `group` 与 `advanced`。

### 8.4 模板插件

建议：

1. 直接按本文主口径更新模板说明。
2. 不再把历史别名写成模板首选写法。

---

## 9. 最终结论

当前 DataFlowKit 插件协议最大的问题，不是字段太乱，而是：

1. 已经形成了一套事实标准。
2. 但这套标准还没有被正式宣布为“主口径”。

所以后续最稳的路线不是大范围重命名，而是：

1. 先明确主口径字段名。
2. 再保留兼容映射。
3. 让新插件、新 UI、新文档全部优先按主口径前进。

这样可以在不打断当前工程进度的前提下，把插件协议真正收紧成可长期维护的统一接口。
