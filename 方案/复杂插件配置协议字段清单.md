# 复杂插件配置协议字段清单

## 1. 文档目的

本文用于收紧 DataFlowKit 复杂插件的配置协议，重点服务于以下目标：

1. 让复杂插件不再依赖某一个具体 UI 窗口。
2. 让 Qt / Tkinter / .NET / Web 都能消费同一套配置描述。
3. 让 `describe_config + validate_config_patch + apply_config_patch` 成为复杂插件的正式主路径。

本文主要针对类似 `plugins/visual_mapping_write_plan_plugin_v1.py` 这类：

1. 配置结构复杂。
2. 配置项存在列表、嵌套规则、联动候选。
3. 过去通常依赖 `open_config_window(...)` 自定义窗口。

---

## 2. 总体原则

复杂插件协议后续应遵循以下原则：

1. **参数层**负责基础入口参数。
2. **配置描述层**负责复杂结构的展示与编辑入口。
3. **patch 层**负责局部修改，而不是整包重写。
4. **上下文层**负责候选项、引用项、动态提示。
5. **旧窗口层**只保留 fallback 地位。

一句话：

**复杂插件不是“不好做表单”，而是要把复杂配置拆成可描述、可编辑、可增量提交的协议对象。**

---

## 3. 复杂插件推荐接口

复杂插件推荐具备以下接口：

1. `get_parameter_schema()`
2. `get_dynamic_parameter_options(param_name, params, context)`
3. `describe_config(params, context)`
4. `validate_config_patch(params, context, patch)`
5. `apply_config_patch(params, context, patch)`
6. `run(input_data, params, context)`

其中：

- `get_parameter_schema()` 负责基础参数。
- `describe_config()` 负责复杂结构的说明与编辑入口。
- `validate_config_patch()` / `apply_config_patch()` 负责结构化变更。
- `run(...)` 只负责执行，不负责 UI 配置组织。

---

## 4. `describe_config()` 顶层字段规范

推荐 `describe_config()` 返回以下顶层结构：

```json
{
  "schema_version": "DataFlowKit.visual_mapping.config.v1",
  "protocol_family": "plugin_complex_config",
  "plugin_id": "visual_mapping_write_plan_v1",
  "config_key": "default",
  "summary": {},
  "views": [],
  "actions": [],
  "context": {},
  "models": {},
  "warnings": [],
  "capabilities": {}
}
```

### 4.1 必需字段

#### `schema_version`

作用：

1. 标识当前配置描述协议版本。
2. 让 UI 可以做兼容分支。

要求：

1. 必须稳定。
2. 协议结构变动时才升级版本。

#### `summary`

作用：

1. 给 UI 提供概览信息。
2. 支持总览区、页签副标题、状态摘要。

要求：

1. 仅放摘要信息。
2. 不放需要大体量渲染的完整数据。

建议字段：

1. 当前配置名。
2. 规则数量。
3. 特征数量。
4. 联动规则数量。
5. 当前输入表别名。
6. 文档记录数 / 内容行数 / 辅助行数。

#### `views`

作用：

1. 定义可展示的配置视图。
2. 让 UI 知道有哪些可切换页签或面板。

要求：

1. 每个 view 必须有稳定 `view_id`。
2. 每个 view 必须声明 `kind`。
3. 若 view 可编辑，必须带 `config_path`。

#### `actions`

作用：

1. 告诉 UI 可以执行哪些配置操作。
2. 让 UI 不再硬编码“某个插件有什么编辑按钮”。

要求：

1. 每个 action 必须有稳定 `action_id`。
2. `action_id` 应为协议语义 ID，而不是 UI 按钮文本。

#### `context`

作用：

1. 提供候选项、引用项、动态提示所需共享上下文。
2. 降低 UI 侧自行反推业务候选的压力。

要求：

1. 只放 UI 可消费的轻量上下文。
2. 不直接泄露实现细节对象。

#### `models`

作用：

1. 提供默认对象模板。
2. 支持“新增一条规则 / 新增一项动作”这类操作。

要求：

1. 模型对象必须可直接用于 patch。
2. 模型字段应与真实配置结构保持一致。

#### `warnings`

作用：

1. 将当前配置风险、数据缺失、上下文不足显式抛给 UI。
2. 避免 UI 靠猜测判断是否异常。

要求：

1. 优先返回结构化 warning。
2. 至少保证有可显示的 message。

---

## 5. `views` 字段规范

每个 `view` 建议具备以下字段：

```json
{
  "view_id": "visual_mapping.rules",
  "title": "单元格映射规则",
  "kind": "structured_list",
  "protocol": "DataFlowKit.visual_mapping.config.v1",
  "config_path": ["plugin_settings", "configs", "default", "rules"],
  "item_count": 12,
  "columns": [],
  "items": [],
  "meta": {}
}
```

### 5.1 必需字段

1. `view_id`
2. `title`
3. `kind`

### 5.2 常用可选字段

1. `protocol`
2. `config_path`
3. `item_count`
4. `columns`
5. `items`
6. `summary`
7. `meta`
8. `editor_kind`

### 5.3 推荐 `kind` 枚举

#### `summary`

用途：

1. 总览页。
2. 概览信息展示。

#### `structured_list`

用途：

1. 规则列表。
2. 特征列表。
3. 动作列表。
4. 嵌套项列表。

要求：

1. 必须给 `columns`。
2. 应给 `items` 摘要。
3. 应给 `config_path`。

#### `form`

用途：

1. 单个配置对象编辑。
2. 规则详情表单。

#### `text_preview`

用途：

1. JSON 预览。
2. 模板文本预览。
3. 说明文本预览。

---

## 6. `actions` 字段规范

每个 action 建议结构：

```json
{
  "action_id": "visual_mapping.edit.rules",
  "label": "编辑单元格映射规则",
  "kind": "config_editor",
  "editor_kind": "visual_mapping.rules",
  "view_id": "visual_mapping.rules",
  "config_path": ["plugin_settings", "configs", "default", "rules"],
  "enabled": true,
  "warning": ""
}
```

### 6.1 推荐 `kind` 枚举

1. `config_editor`
2. `append_item`
3. `remove_item`
4. `move_item`
5. `duplicate_item`
6. `preview_effect`
7. `import_config`
8. `export_config`

### 6.2 约束

1. UI 只根据协议显示 action，不应硬编码插件名分支。
2. action 需要能独立表达“干什么”，而不是靠按钮位置推断语义。
3. 若 action 需要上下文条件，应直接返回 `enabled=false` 和 `warning`。

---

## 7. `context` 字段规范

`context` 是复杂插件多 UI 的关键。

推荐分成以下几类：

### 7.1 数据候选

示例：

1. `table_names`
2. `doc_headers`
3. `content_fields`
4. `aux_fields`
5. `sheet_names`

作用：

1. 供表单下拉框使用。
2. 供结构化列表列编辑器使用。

### 7.2 引用候选

示例：

1. `feature_names`
2. `rule_names`
3. `linked_trigger_options`

作用：

1. 供规则引用其他规则。
2. 供联动动作引用触发规则。

### 7.3 固定 choices

示例：

```json
{
  "choices": {
    "match_modes": ["包含", "等于", "不等于"],
    "linked_target_modes": ["触发格偏移", "指定坐标"],
    "linked_value_sources": ["固定值", "触发新值", "新内容字段"]
  }
}
```

作用：

1. 将插件内固定枚举显式暴露给 UI。
2. 避免 UI 再自己复制一份枚举常量。

### 7.4 上下文要求

1. 字段命名要稳定。
2. 返回值要 JSON 友好。
3. 不返回 Python 对象、回调、窗口句柄。
4. 大数据列表要考虑裁剪或摘要。

---

## 8. `models` 字段规范

`models` 用于新增对象时的默认值。

推荐结构：

```json
{
  "models": {
    "rule_default": {},
    "linked_rule_default": {},
    "feature_default": {},
    "empty_config": {}
  }
}
```

要求：

1. `*_default` 必须是可直接写入的结构。
2. 新增对象时，UI 不要自己猜默认字段。
3. 复杂对象默认值由插件定义，而不是 UI 定义。

---

## 9. `warnings` 字段规范

推荐 warning 结构：

```json
[
  {
    "code": "content_fields_missing",
    "level": "warning",
    "message": "当前新内容表没有可选字段。",
    "view_id": "visual_mapping.rules"
  }
]
```

建议字段：

1. `code`
2. `level`
3. `message`
4. `view_id`
5. `field`
6. `hint`

要求：

1. UI 应原样展示 `message`。
2. `code` 用于后续筛选和调试。
3. `view_id` / `field` 用于精准定位问题位置。

---

## 10. patch 协议建议字段

复杂插件的 patch，不建议只是随意字典，建议至少具备这些语义信息：

```json
{
  "schema_version": "DataFlowKit.visual_mapping.config.v1",
  "config_name": "default",
  "section": "rules",
  "operation": "update_item",
  "path": ["plugin_settings", "configs", "default", "rules", 3],
  "payload": {}
}
```

### 10.1 推荐字段

1. `schema_version`
2. `config_name`
3. `section`
4. `operation`
5. `path`
6. `payload`
7. `target_id`
8. `target_index`

### 10.2 推荐 `operation` 枚举

1. `append_item`
2. `update_item`
3. `remove_item`
4. `move_item`
5. `replace_section`
6. `duplicate_item`

### 10.3 要求

1. patch 应优先表达“意图”，而不是只表达“整段新 JSON”。
2. UI 不应直接改底层存储文件。
3. 所有 patch 都先走 `validate_config_patch(...)`。

---

## 11. 对 `visual_mapping_write_plan_plugin_v1.py` 的具体要求

该插件当前已经是最接近正式复杂协议样板的对象，后续建议按以下口径收紧：

1. 固定 `schema_version`、`view_id`、`action_id`、`editor_kind` 命名。
2. 把 `warnings` 从纯文本列表逐步升级为结构化 warning 对象。
3. 继续强化 `context` 中的字段候选、规则候选、触发候选。
4. 让 Qt / Tkinter 先统一消费 `describe_config()` 返回值。
5. 把 `open_config_window(...)` 明确降级为 fallback。

---

## 12. UI 侧必须遵守的边界

各 UI 应遵守以下边界：

1. UI 负责展示与交互，不负责重建插件业务规则。
2. UI 负责提交 patch，不直接改插件配置文件。
3. UI 负责显示 warning，不自行吞掉协议返回的错误。
4. UI 负责消费 `context` 和 `models`，不自己发明默认值。
5. UI 若遇到未知 `view.kind` / `action.kind`，应优雅降级显示。

---

## 13. 最终结论

复杂插件配置协议的核心，不是把所有复杂插件都改成“普通表单”，而是：

1. 用 `describe_config()` 暴露视图、动作、上下文、模板。
2. 用 `validate_config_patch()` / `apply_config_patch()` 处理结构化修改。
3. 用稳定的 `view_id / action_id / editor_kind / config_path` 让多 UI 承接一致。

只要这一层收紧，Qt 阶段能用，后面的 `.NET`、Web、Electron 也就不需要再重走一遍 Tkinter 插件窗口的老路。
