# 插件协议实现差距清单

## 1. 文档目的

本文用于回答一个很实际的问题：

**当前 DataFlowKit 主程序 / Qt / 插件服务，已经支持了哪些插件协议能力，还缺哪些承接能力需要补。**

这份清单的作用不是重新设计协议，而是：

1. 对照已经写出的协议文档。
2. 标出当前代码已实现部分。
3. 标出后续真正需要补的缺口。

这样后面无论先改插件，还是先补 Qt / 主程序承接器，都有明确落点。

---

## 2. 总体判断

当前项目在插件协议这条线上，已经不是“起步阶段”，而是处在：

**协议骨架已形成，关键差距集中在元数据补齐、复杂配置收口、跨 UI 一致性增强。**

### 2.1 2026-06-24 当前状态更新

最近一轮改造后，插件协议已经比本文早期版本更靠前：

1. `PluginService` 已输出 `plugin_config_compatibility.v1`，可以明确区分 `schema_patch`、`describe_config`、`schema_form`、`legacy_custom_config`。
2. 旧 `open_config_window(...)` 已统一标记为 `legacy_fallback`，并进一步带有 `preferred=false / ui_role=fallback_action / ui_prominence=low / ui_placement=compatibility_menu / requires_confirmation=true`，Qt 与后续 UI 应把它放在低优先级兼容入口，而不是推荐主路径。
3. 视觉映射插件已经提供 `protocol_manifest / patch_schema / warning_schema`，并把 patch 来源字段补到 `view_id / editor_kind / action_id`。
4. `PluginService` 已把 warning 归一成 `plugin_config_warning_target.v1`，把写回结果归一成 `plugin_config_patch_target.v1`，可描述 `view_id / field / path / config_path / target_index / target_id / focus_path`。
5. Qt 插件配置承接器已支持 `summary / form / structured_list / text_preview / resource_list`，并能把 structured-list 操作回写为插件 patch。
6. Qt 已能消费 `patch_result.target`，写回后自动切换到对应插件配置页签并选中目标条目。
7. `word_excel_write_from_table_plugin_v2.py`、`word_excel_read_to_db_plugin_v1.py`、模板插件已补入一批参数分组、显隐、候选、空状态、联动元数据。

### 2.2 2026-06-25 当前状态更新

本轮继续收口后，又新增了几项可作为后续基线的能力：

1. `describe_plugin_config().actions` 已成为 Qt 旧插件设置入口的权威来源，Qt 不再只靠 `schema.plugin.legacy_config_state` 自己拼按钮状态。
2. 旧插件窗口确认提示已下沉到 `WorkflowFacade.describe_confirmation_prompt(action="legacy_plugin_config")`，Qt / .NET / Web 可以复用同一确认 payload。
3. 可视化映射插件已补 `layout` 与 `ui_hints`，明确默认视图、视图顺序、主要/高级视图、视图说明、空状态和主动作。
4. `PluginService.describe_plugin_config()` 已将复杂插件的 `layout / ui_hints` 透传到标准 `plugin_config.v1` 顶层。
5. Qt 插件配置页签已消费 `layout.view_order / default_view_id` 和 `ui_hints.view_hints`，可按协议排序、默认聚焦并显示视图提示。
6. 数据源共享服务已为 `data_source_manager_state.v1` 增加 `data_source_manager_layout.v1 / data_source_manager_ui_hints.v1`，为后续 `.NET` / Web 数据源窗口提供布局与动作优先级说明。
7. `data_source_service.v1` 已继续补入 `data_source_client_profiles.v1 / data_source_transport_hints.v1`，固定 Qt、stdio、`.NET`、Web 的推荐消费方式、大表分页策略、句柄释放时机和危险操作确认语义。
8. Qt 数据源管理窗口已开始消费 `client_profiles / transport_hints`，窗口属性会记录客户端 profile、表传输 action、分页 action、释放 action 与服务层建议的默认分页大小。

### 2.3 2026-06-25 后续收口状态

继续补强后，插件协议又前进了一步：

1. 普通参数型插件即使没有自定义复杂配置描述，也会在 `describe_plugin_config()` 顶层输出默认 `plugin_config_layout.v1 / plugin_config_ui_hints.v1`。
2. 默认 layout 会把 `plugin.params`、`plugin.parameter_metadata`、资源/效果等视图组织成稳定 `view_order / default_view_id / primary_views / secondary_views`，并保留旧窗口 `compatibility_action_ids`。
3. 默认 UI hints 会把 `plugin_parameter_ui_hints.v1` 提升为 `parameter_field_hints`，让 Qt、`.NET`、Web 不必自己钻进 metadata 猜字段提示。
4. Qt 插件参数表单已经消费 `parameter_field_hints`，把字段 warning、placeholder、empty_text、advanced、unit/min/step 合并进表单 schema 与 tooltip。
5. stdio worker 已覆盖普通插件 `plugin_config_layout.v1 / plugin_config_ui_hints.v1`，后续 `.NET` / Web 可以直接从 `describe_plugin_config` 读取。
6. `plugin_config_action_state.v1` 已补 `button_order`，复杂插件 `structured_list` 工具按钮不再依赖 JSON object 顺序。
7. 旧插件窗口已补 `plugin_legacy_window_policy.v1`，明确 Tk/Qt 可作为兼容入口直接打开，`.NET` / Web / Electron / CLI 只显示迁移提示并走统一配置协议。
8. 普通参数型插件 metadata 已补 `plugin_parameter_field_state.v1`，通过 `field_state_schema / field_state_index` 描述字段默认状态、条件依赖、候选刷新和提示信息。
9. `protocol_manifest` 已纳入参数字段状态和旧窗口多端策略，后续 UI 可以从同一入口判断“能渲染什么、不能直接打开什么”。
10. `resolve_plugin_parameter_field_state` 已作为 headless/stdio/Qt client 共享接口落地，可返回 `plugin_parameter_field_runtime_state.v1`，解析字段 `visible / enabled / status / needs_options_refresh / dependents`。
11. Qt 参数表单动态刷新已优先消费共享运行时字段状态，后端不可用时再回退本地条件判断。

所以这份清单里的“差距”需要按新口径理解：多数已经不是“完全没有”，而是“覆盖面、稳定命名、体验细节还要继续收口”。

换句话说：

1. 插件执行协议，基础已经足够好。
2. 参数型插件表单承接，已经有比较实用的底子。
3. 复杂插件的配置协议，也已经有雏形。
4. 真正没完全收住的是“协议字段是否统一使用”以及“UI 是否把这些协议都完整消费”。

## 3. 已有能力盘点

## 3.1 `engine/plugin_service.py` 已支持的能力

当前已明确存在：

1. 参数 schema 归一化。
2. 参数类型归一化。
3. `field_select / table_select / input_table_select / dynamic_select / folder_path` 等基础映射。
4. 动态候选接口承接。
5. 复杂插件 `describe_config()` 承接。
6. `validate_config_patch()` 结果归一化。
7. `apply_config_patch()` 结果归一成 `plugin_config_patch_result.v1`，并携带 `plugin_config_patch_target.v1` 供 UI 定位。
8. warning 归一成 `plugin_config_warning_target.v1`，供 UI 按 view/field/path 展示与后续定位。
9. 插件配置表单分组的基础组织。

说明：

- 这意味着 service 层已经是插件协议的真实中枢。
- 后续很多工作不需要再另起新层，而是继续把这里的标准补完整。

## 3.2 Qt `config_form.py` 已支持的能力

当前已明确存在：

1. 参数表单基础渲染。
2. 结构化列表 `structured_list` 承接。
3. 动态候选刷新。
4. `visible_when` 条件显隐。
5. `enabled_when` 条件启用。
6. 字段级校验问题映射。
7. 结构化列表行新增、删除、移动。

说明：

- 这说明 Qt 端已经不只是“看协议文本”，而是开始真正消费协议了。
- 后续最需要补的是更完整的元数据支持，而不是从零重写表单系统。

## 3.3 Qt `main_window.py` 已支持的能力

当前已明确存在：

1. 插件配置视图 tabs 渲染。
2. `summary` 视图渲染。
3. `structured_list` 视图渲染。
4. 基础 patch 操作按钮：新增、删除、启停、上移、下移。
5. 插件配置 patch 回写。
6. 写回成功后消费 `patch_result.target`，可切换到对应插件配置视图并定位 structured-list 条目。
7. 旧 Tk 插件窗口入口按低优先级兼容按钮展示，不再作为推荐配置入口。

说明：

- Qt 端已经具备复杂插件配置协议的试运行基础。
- 现在的缺口更偏“能力不够强”，不是“完全没有能力”。

## 4. 与目标协议相比，当前已基本达标的部分

### 4.1 参数类型统一

当前状态：**基本达标**

原因：

1. `plugin_service` 已有类型映射。
2. 常见参数类型已经能跨插件使用。
3. Qt 表单也有承接基础。

剩余工作：

1. 主要是文档收口。
2. 少量新增类型或细粒度元数据需要补。

### 4.2 动态候选

当前状态：**已具雏形，属于可用阶段**

原因：

1. `dynamic_select` 已有 service 层支持。
2. Qt 表单已能刷新动态状态。

剩余工作：

1. 候选来源命名还需要彻底统一。
2. 各插件还没普遍补 `depends_on / refresh_on_change / empty_text`。

### 4.3 复杂插件配置视图

当前状态：**已具雏形，属于半成品可验证阶段**

原因：

1. `describe_config()` 已被 service 层和 Qt 端承接。
2. `structured_list` 视图和 patch 回写已可跑通。

剩余工作：

1. `view` / `action` 的字段口径还需要固定。
2. 更多 view kind 还没有真正落地。
3. patch 语义还偏轻量，未完全升级为正式协议对象。

## 5. 当前主要差距

## 5.1 参数元数据没有在插件层全面补齐

当前状态：**已补强一轮，仍需扩大覆盖面**

当前问题：

1. 读 Word/Excel、写 Word/Excel、模板插件已补入一批 `group / visible_when / options_source / empty_text / advanced / depends_on`。
2. 普通参数型插件已经有默认 `plugin_config_layout.v1 / plugin_config_ui_hints.v1`，不再只停留在底层 `parameter_metadata`。
3. `field_state_schema / field_state_index` 已经能描述字段状态、条件依赖、候选刷新和提示信息，`resolve_plugin_parameter_field_state` 已能按当前 config 解析运行时状态。
4. 仍需要继续检查其他插件、未来新增插件和边缘参数，避免回到“只写基础字段”的老模式。
5. 参数元数据需要继续稳定命名，尤其是 `depends_on / refresh_on_change / options_source / field_state` 的跨 UI 语义。

影响：

1. Qt 能渲染，但用户体验还不够稳定。
2. 后续 `.NET` 端已能读取更多状态信息，并可调用运行时 resolver；剩余重点是把更多插件和复杂字段纳入同一协议。

建议优先级：**高**

---

## 5.2 `describe_config()` 顶层字段还没完全收口

当前状态：**视觉映射样板已基本收口，仍需推广为所有复杂插件的固定口径**

当前问题：

1. 视觉映射插件已经稳定输出 `schema_version / protocol_family / plugin_id / config_key / summary / views / actions / context / models / warnings / capabilities / protocol_manifest`。
2. 其他复杂插件或后续新插件仍需要按同一口径补齐。
3. service 层需要继续保证这些字段在 `describe_plugin_config` 输出中被稳定透传。

影响：

1. UI 虽然能跑，但还缺少更明确的协议自描述能力。
2. 后续多端消费时，仍容易出现“这个字段是不是有、含义是否固定”的问题。

建议优先级：**高**

---

## 5.3 `warnings` 仍偏文本化

当前状态：**warning target 已落地，整体生态仍需推广**

当前问题：

1. 视觉映射插件已提供 `warning_schema`，warning 可携带 `code / level / view_id / field / path / config_path / hint`。
2. `PluginService` 已补 `plugin_config_warning_target.v1`，即使插件只给出轻量字段，也会归一出可定位 target。
3. Qt 已能展示 warning 的 view/field/path/config 信息，但“点击 warning 定位到具体 view/field”的交互还可以继续加强。
4. 其他节点、普通插件、部分 facade 输出仍可能只有文本或较粗粒度 issue，需要继续推广结构化 warning。

影响：

1. UI 只能笼统展示，不方便精准定位。
2. 后续 `.NET` 或 Web 端想做更细粒度提示时会受限。

建议优先级：**中高**

---

## 5.4 patch 协议还处在轻量阶段

当前状态：**复杂插件 patch 对象与写回定位 target 已升级一轮，Qt 仍需继续统一发送口径**

当前问题：

1. 视觉映射插件已支持 `schema_version / protocol_family / plugin_id / config_key / view_id / editor_kind / action_id / section / operation / path / target_index / target_id / to_index / payload`。
2. Qt 已能携带 `view_id / editor_kind / action_id` 发送 structured-list patch。
3. `PluginService` 已在 `patch_result` 中补 `plugin_config_patch_target.v1`，Qt 已能消费 `view_id / target_index / target_id / focus_path` 做页签与条目定位。
4. 仍需继续减少兼容轻量字段的新增使用，并把其他复杂 editor 的 patch 也统一到同一协议对象。

影响：

1. 现在够用，但扩展性一般。
2. 一旦后续复杂插件再多几类 editor，就容易继续分化。

建议优先级：**中高**

---

## 5.5 参数型插件的分组与折叠能力还未完全协议化

当前状态：**主要参数型插件已有分组，后续重点是规范化和 UI 体验**

当前问题：

1. 写入、读取、模板插件已经开始系统提供 `group / group_order / advanced`。
2. Qt 现在可以按分组渲染，但折叠、密度、失效值提示等体验仍可继续优化。
3. 插件模板需要继续明确“新增插件必须优先提供 UI 元数据”，避免后续新增插件退回平铺参数。

影响：

1. Qt 界面容易显得拥挤。
2. 后续多端界面都要自己再做一层人工整理。

建议优先级：**高**

---

## 5.6 复杂插件 `view.kind` 仍偏少

当前状态：**Qt 已支持更多 view kind，复杂业务覆盖仍需扩展**

当前问题：

1. Qt 当前已支持 `summary / form / structured_list / text_preview / resource_list`。
2. 支持范围已经可用，但复杂插件如果需要规则图、差异预览、批量 patch 历史、资源预检查等，仍需要新增通用 view kind 或增强现有 view。
3. 新增 view kind 必须先进协议，不应在 Qt 里写插件专用面板。

影响：

1. 当前复杂插件能展示一部分，但还不能完全覆盖未来更复杂的配置体验。

建议优先级：**中**

---

## 6. 分层差距清单

### 6.1 必须先补

1. 现有参数型插件补 `group / options_source / visible_when / empty_text`。
2. `describe_config()` 顶层字段收口为正式主口径。
3. `visual_mapping_write_plan_plugin_v1.py` 的 warning 结构化升级。
4. 复杂插件 patch 字段语义收口。

理由：

- 这几项直接决定后续 Qt 是否真正实用，以及 `.NET` 是否能低成本跟进。

### 6.2 Qt 阶段建议补

1. Qt 端支持更多 `view.kind`。
2. Qt 端对 warning 做更细定位显示，例如点击 warning 定位到 view/field。
3. Qt 端对空候选、失效值、候选刷新状态做更清晰反馈。
4. 结构化列表 editor 与 `editor_kind` 的承接规则继续补强。

### 6.3 可以留给 .NET / 多端阶段

1. 更高级的资源型配置视图。
2. 更完整的可视化 patch 历史与撤销。
3. Web / Electron 特有的更强编辑器体验。

---

## 7. 最自然的后续执行顺序

### 第一步

先补插件 schema 元数据：

1. 参数型插件先补 `group / options_source / visible_when / empty_text`。
2. 复杂插件先补 `protocol_family / plugin_id / config_key / capabilities`。

### 第二步

再补复杂插件 warning 与 patch 收口：

1. 把 warning 从字符串列表逐步换成结构化对象。
2. 把轻量 patch 慢慢往正式协议对象收口。

### 第三步

最后再补 UI 承接增强：

1. Qt 补强更丰富的 view kind。
2. Qt 优化候选刷新、失效值提示、问题定位。

---

## 8. 最终结论

当前插件协议这条线，已经过了“要不要做”的阶段，现在是“怎么把已经长出来的协议正式收口”的阶段。

最关键的判断是：

1. **service 层基础已经够了。**
2. **Qt 端承接器已经不是空白。**
3. **现阶段主要缺口是字段元数据补齐和协议字段统一使用。**

所以后续最划算的路线，不是大改框架，而是：

1. 先补协议字段。
2. 再补 warning / patch 结构。
3. 再补 UI 承接细节。

这条路线最稳，也最符合当前 DataFlowKit 已经形成的 headless + 协议化 + 多 UI 兼容方向。
