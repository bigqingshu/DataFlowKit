# 插件迁移优先级清单

## 1. 文档目的

这份清单用于回答两个实际问题：

1. 现有插件里，哪些已经基本符合多 UI 方向。
2. 后续插件协议化改造，应该先改哪一类、为什么。

这里的“迁移”，不是优先重写执行逻辑，而是优先把**插件配置方式**从“依赖某个具体 UI 窗口”逐步迁到“headless + schema + patch + describe_config”的统一路径上。

当前项目的判断结论很明确：

- 插件**执行协议**整体已经不差。
- 插件**多 UI 配置协议**才是后续主战场。
- `open_config_window(...)` 应降级为兼容方案，而不是长期主路径。

当前协议里的旧窗口入口应统一标记为：

1. `fallback: true`
2. `deprecated: true`
3. `lifecycle: legacy_fallback`
4. `migration_target: describe_config + parameter_metadata + config_patch`

退场门槛不是“窗口还能不能打开”，而是：

1. 插件是否已经提供等价 `describe_config` / `parameter_metadata`。
2. 复杂列表或嵌套项是否已有 `models`、`patch_schema`、`warning_schema` 或 `protocol_manifest`。
3. Qt 是否能完成主要配置路径，后续 .NET/Web 是否能按同一协议渲染。
4. 旧 Tk 窗口是否只剩兜底价值，而不是唯一可用入口。

---

## 2. 分档标准

### A 档：已接近多 UI 标准路径

满足大部分以下条件：

1. 具备稳定的 `PLUGIN_INFO`、`get_parameter_schema()`、`run()`。
2. 执行过程不依赖主程序具体窗口对象。
3. 输出、进度、日志、缓存、数据库访问已经走统一协议。
4. 配置主要是普通参数表单，不依赖复杂自定义弹窗。
5. Qt / Tkinter / .NET / Web 理论上都能直接消费。

### B 档：执行已统一，但配置能力还不完整

满足以下特征：

1. 执行协议已经能稳定跑。
2. 参数 schema 已存在，但复杂配置还没有完全进入标准协议。
3. 存在部分自定义配置逻辑，或者上下文候选不够强。
4. 可以作为下一阶段优先迁移对象。

### C 档：仍明显依赖旧窗口模式

满足以下特征之一：

1. 复杂配置主要藏在 `open_config_window(...)` 内。
2. 主程序只能通过弹窗完成关键配置。
3. 脱离 Tkinter / 某个特定 UI 后，配置能力明显下降。

这类插件不是不能用，而是**暂时不适合直接作为多 UI 样板**。

## 3. 现有插件盘点

当前 `plugins/` 下主要插件：

1. `plugin_template_输出日志_后台进度_插件缓存版.py`
2. `visual_mapping_write_plan_plugin_v1.py`
3. `word_excel_read_to_db_plugin_v1.py`
4. `word_excel_write_from_table_plugin_v2.py`

## 4. 当前分档结果

### 4.1 A 档

#### `plugins/plugin_template_输出日志_后台进度_插件缓存版.py`

定位：**标准参考模板**

现状判断：

1. 有明确的 `PLUGIN_INFO`。
2. 有 `get_parameter_schema()`。
3. 有统一输出 schema。
4. 演示了缓存、后台进度、数据库 API、标准 run 协议。
5. 不依赖插件私有配置窗口。

结论：

- 这不是复杂业务插件，但它已经很适合作为“新插件开发模板”。
- 后续新的外部插件，应该优先按它的运行协议来写。

建议动作：

1. 把它视为**插件执行协议样板**。
2. 后续若补文档，优先把“参数 schema 命名规范、动态候选规范、错误返回规范”写到模板说明中。

#### `plugins/word_excel_read_to_db_plugin_v1.py`

定位：**参数型插件，已较接近 A 档**

现状判断：

1. 参数集中在 `get_parameter_schema()`。
2. 配置以普通表单为主，没有看到自定义配置窗口。
3. 已有统一输出 schema。
4. 已有表访问约束 `get_table_access_spec(...)`。
5. 进度、输出、错误处理都走统一路径。

结论：

- 这类插件已经很适合多 UI 直接消费。
- Qt、Tkinter、后续 .NET 都只需要把参数 schema 渲染成表单即可。

剩余问题：

1. `field_select`、`folder_path`、表访问说明等控件语义，还需要继续在主程序层标准化。
2. 某些参数的联动显示规则，后续最好再抽成统一前端元数据，而不是各 UI 自己猜。

建议动作：

1. 作为“参数型插件标准样板”保留。
2. 后续只需补少量 UI 元数据和更细的参数联动描述。

#### `plugins/word_excel_write_from_table_plugin_v2.py`

定位：**参数型插件，已较接近 A 档**

现状判断：

1. 参数 schema 完整，字段选择项很多，但仍属于标准表单型配置。
2. 没有看到依赖自定义配置窗口。
3. 输出 schema 统一。
4. 执行逻辑与 UI 没有直接强耦合。

结论：

- 这个插件虽然参数多，但本质仍是“可表单化”的协议型插件。
- 它对多 UI 的要求，主要是前端把字段候选、布尔项、数字项、路径项渲染好。

剩余问题：

1. 参数数量较多，后续更适合增加分组、折叠、条件显示等 UI 元数据。
2. 某些字段依赖输入表 headers，候选上下文还需要统一加强。

建议动作：

1. 作为“复杂表单型插件样板”保留。
2. 不急着重构插件内部逻辑，优先补通用表单渲染能力即可。

### 4.2 B 档

#### `plugins/visual_mapping_write_plan_plugin_v1.py`

定位：**复杂配置插件，也是插件协议化的核心样板**

现状判断：

1. 已有 `get_parameter_schema()`，基础参数已标准化。
2. 已有 `describe_config(...)`，说明已经开始走协议化配置路线。
3. 已有 `validate_config_patch(...)` 和 `apply_config_patch(...)`。
4. 仍保留 `open_config_window(...)` 旧窗口路径。
5. 配置本身极复杂，含规则、特征、全局替换、联动写入等多段结构。

结论：

- 它不是落后插件，反而是**最关键的迁移试验田**。
- 当前状态更准确地说是：执行已标准，复杂配置协议正在成型，但还没有完全取代旧窗口。

为什么先做它：

1. 它最能检验多 UI 配置协议是否真的够用。
2. 如果它能迁通，后面大量插件都可以沿这个模式落地。
3. Qt / .NET / Web 后续都会受益，不用再为每个插件单独做窗口。

当前主要缺口：

1. `describe_config()` 输出虽然已经存在，但还需要进一步稳定成长期协议。
2. 各类 editor_kind 仍需要主程序 / Qt 端实现统一编辑器承接。
3. 候选上下文还可以继续增强，例如字段、表、规则引用、触发选项等。
4. 旧 `open_config_window(...)` 仍是实际兜底路径，说明协议路径还没完全成为第一入口。

建议动作：

1. 把它作为**第一优先级迁移插件**。
2. 继续补齐 `describe_config` 输出的稳定字段、动作、上下文候选。
3. 由主程序 / Qt 提供通用 structured-list / patch-editor 承接，而不是再给它做一个专用窗口。
4. 等协议路径稳定后，再把 `open_config_window(...)` 降为 legacy fallback。

### 4.3 C 档

当前仓库这四个主要插件里，**严格意义上的纯 C 档并不多**。

原因是：

1. 执行协议已经被统一过一轮。
2. 参数型插件基本都能靠 schema 表达。
3. 真正的问题集中在复杂配置插件，而不是所有插件都严重落后。

目前最接近 C 档特征的，只能说是：

- `visual_mapping_write_plan_plugin_v1.py` 中仍保留的旧 `open_config_window(...)` 路径。

但它整体不应被简单归类为“旧插件”，因为它已经有明显的协议化基础。

## 5. 推荐迁移顺序

### 第一优先级：`visual_mapping_write_plan_plugin_v1.py`

原因：

1. 它决定“复杂插件能不能脱离专用窗口”。
2. 它决定 `describe_config + patch` 这一套是否真的能成为主路径。
3. 它一旦打通，后续插件协议化就不再是概念问题，而是复制模式的问题。

目标：

1. 让 Qt 能直接消费它的配置描述。
2. 让 Tkinter 主程序也优先走协议化入口。
3. 把旧窗口降级为兼容模式。

### 第二优先级：`word_excel_write_from_table_plugin_v2.py`

原因：

1. 参数多、字段多，是“复杂表单型插件”的代表。
2. 可以检验统一表单渲染器是否足够实用。
3. 与可视化映射插件配合紧密，属于同一条业务链上的关键插件。

目标：

1. 补强字段候选上下文。
2. 增加参数分组、条件显示、帮助信息等 UI 元数据。
3. 让多 UI 对大参数量插件的呈现更稳定。

### 第三优先级：`word_excel_read_to_db_plugin_v1.py`

原因：

1. 结构已较规整。
2. 主要是把参数联动语义进一步标准化。
3. 它不太像“难点插件”，更像“把规范补完整”的收口项。

目标：

1. 统一目录 / 文件 / 字段来源切换规则。
2. 补齐参数动态显隐与校验元数据。

### 第四优先级：模板插件与新插件脚手架

原因：

1. 模板不是存量难点，但决定以后新增插件是不是继续走老路。
2. 如果模板规范不先收口，新插件还会不断长出新的 UI 耦合。

目标：

1. 明确“推荐实现接口”。
2. 把 `describe_config / patch / dynamic_options / table_access_spec` 写进新模板规范。

---

## 6. 建议的迁移动作分层

### 第一层：先统一协议，不先碰执行细节

优先做：

1. `get_parameter_schema()` 字段含义收口。
2. 动态候选接口收口。
3. `describe_config()` 返回结构收口。
4. `validate_config_patch()` / `apply_config_patch()` 语义收口。

理由：

- 这层改完，Qt / Tkinter / .NET / Web 都能先共享配置入口。
- 不必一开始就改插件内部业务算法。

### 第二层：主程序和 Qt 提供通用配置承接器

优先做：

1. 通用参数表单渲染器。
2. 通用 structured-list 编辑器。
3. 通用 patch 提交接口。
4. 通用动态候选刷新。

理由：

- 插件协议再好，没有前端承接器，也还是会回到“每个插件一个窗口”。

### 第三层：再逐步淘汰旧自定义窗口

优先做：

1. 先让协议路径可用。
2. 再把旧窗口改成 fallback。
3. 最后视情况决定是否彻底移除旧入口。

理由：

- 这样风险最小，也不会影响现有可用性。

---

## 7. 对 Qt 阶段的直接意义

Qt 阶段不是要把所有插件都重写一遍，而是要先证明：

1. 参数型插件可以直接表单化。
2. 复杂插件可以通过 `describe_config + patch` 进入统一编辑器。
3. 插件配置不再绑定 Tkinter 窗口。

如果这三点成立，那么后面做 `.NET` UI 时：

1. 不需要再重新理解插件内部逻辑。
2. 只需要消费统一协议。
3. 多端成本会明显下降。

---

## 8. 最终判断

当前插件体系并不是“要推倒重来”，而是已经到了一个比较好的阶段：

1. **执行层**已经基本成型。
2. **参数型插件**已经大多能被统一 UI 直接消费。
3. **复杂配置插件**已经出现协议化雏形。
4. 真正要做的是把这些雏形提升为正式主路径。

所以后续插件方向，不是“把所有插件都重写成新插件”，而是：

1. 先拿 `visual_mapping_write_plan_plugin_v1.py` 做复杂配置标准样板。
2. 再把 `word_excel_write_from_table_plugin_v2.py`、`word_excel_read_to_db_plugin_v1.py` 这类参数型插件的 UI 元数据补强。
3. 同时把模板规范更新，避免新插件继续长出旧式自定义窗口依赖。

这条路线，最符合当前 DataFlowKit 已经形成的 headless + 协议化 + 多 UI 兼容方向。
