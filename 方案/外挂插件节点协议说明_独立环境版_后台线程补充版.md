# 外挂插件节点协议说明：独立环境、多UI配置与后台线程统一版

> 本文整合以下几条主线：
>
> 1. 插件独立 Python 环境运行协议
> 2. 后台线程执行、进度与取消机制
> 3. 插件多 UI 统一配置协议
> 4. 旧自定义窗口的兼容定位
>
> 目标：让插件在保留现有可用性的同时，逐步从“依附某个具体 UI 的插件窗口”迁移到“Qt / Tkinter / .NET / Web 都可消费的统一协议”。

---

## 1. 总体目标

DataFlowKit 后续的插件体系，不应只解决“插件能不能运行”，还要解决：

1. 插件能否在不同 UI 下统一配置。
2. 插件能否在后台任务中安全运行。
3. 插件能否在预览模式下保持零副作用。
4. 插件能否通过标准协议描述复杂配置，而不是把规则藏在某个窗口回调中。

一句话说：

**执行协议要稳定，配置协议要统一，旧窗口只保留兼容地位。**

---

## 2. 推荐目录结构

```text
工具目录/
├─ 数据工作流工具.exe
├─ plugins/
│  ├─ 普通单文件插件.py
│  └─ example_plugin/
│     ├─ plugin.json
│     ├─ plugin.py
│     └─ requirements.txt
├─ plugin_envs/
│  └─ example_plugin/
│     └─ Scripts/python.exe
├─ plugin_data/
├─ logs/
└─ plan/
```

建议：

- 普通轻量插件可以继续使用单文件模式，在主程序环境中直接运行。
- 依赖第三方库、环境复杂、希望长期维护的插件，建议使用 `plugins/插件ID/plugin.json + plugin.py + requirements.txt` 结构。
- 独立环境放在 `plugin_envs/插件ID/`，由主程序通过 `subprocess` 调用。

---

## 3. 两种运行方式

插件节点继续支持两种运行模式：

```text
主程序内置环境
插件独立环境
```

### 3.1 主程序内置环境

主程序直接 import 插件模块，并调用：

```python
def run(input_data, params, context):
    ...
```

优点：

1. 启动快。
2. 适合轻量插件。
3. 与主程序上下文整合更直接。

限制：

1. 插件依赖必须和主程序环境兼容。
2. 打包时第三方依赖处理更麻烦。

### 3.2 插件独立环境

主程序不在自身进程中 import 插件，而是调用：

```text
plugin_envs/插件ID/Scripts/python.exe plugins/插件ID/plugin.py --input input.json --output output.json
```

优点：

1. 插件依赖独立。
2. 不需要把所有插件依赖都打进主程序。
3. 适合 Word / Excel / OCR / AI / 大型第三方库插件。

限制：

1. 启动开销更高。
2. 需要通过标准 JSON 协议交换数据。

## 4. plugin.json 元数据协议

独立环境插件与长期维护插件，推荐使用 `plugin.json` 注册，避免主程序为了读取插件信息而 import 插件代码。

示例：

```json
{
  "plugin_info": {
    "id": "example_plugin",
    "name": "示例插件",
    "version": "1.0.0",
    "api_version": "1.0",
    "category": "文件处理",
    "description": "示例说明",
    "input_type": "table",
    "output_type": "table",
    "danger_level": "safe_readonly",
    "run_mode": "external_python"
  },
  "entry": "plugin.py",
  "requirements": "requirements.txt",
  "schema": [
    {"name": "path_field", "label": "文件路径字段", "type": "field_select", "default": "完整路径"}
  ]
}
```

主程序会扫描 `plugins/*/plugin.json` 并注册为插件节点。

建议元数据层长期补充这些能力标记：

1. `supports_schema_config`
2. `supports_config_patch`
3. `supports_option_resolver`
4. `supports_preview_config_effect`
5. `legacy_custom_config`

这些标记可以来自 `plugin.json`，也可以由主程序扫描插件函数能力后自动推断，并显示在 `PluginService.get_plugin_schema(...).capabilities` 中。

## 5. 独立环境输入输出协议

主程序会生成：

```text
plugin_data/插件ID/runs/时间戳/input.json
plugin_data/插件ID/runs/时间戳/output.json
```

### 5.1 input.json

```json
{
  "input_data": {
    "type": "table",
    "headers": ["文件名", "完整路径"],
    "rows": []
  },
  "params": {},
  "context": {
    "app_dir": "...",
    "db_path": "...",
    "database_access": "managed_requests",
    "database_available": true,
    "plugins_dir": "...",
    "plugin_data_dir": "...",
    "log_dir": "...",
    "is_preview": true,
    "execute_actions": false,
    "workflow_name": "...",
    "node_name": "...",
    "plugin_id": "example_plugin",
    "transit_tables": {}
  }
}
```

### 5.2 output.json

```json
{
  "ok": true,
  "message": "执行完成",
  "output": {
    "type": "table",
    "headers": [],
    "rows": []
  },
  "database_requests": [],
  "logs": [],
  "summary": {}
}
```

### 5.3 失败返回建议

```json
{
  "ok": false,
  "message": "错误信息",
  "output": {
    "type": "table",
    "headers": ["错误信息"],
    "rows": [["错误信息"]]
  },
  "logs": [
    {"level": "ERROR", "message": "错误信息"}
  ]
}
```

建议插件即使遇到业务错误，也尽量写出标准 `output.json`，方便主程序显示错误表、记录日志、继续执行失败策略。

## 6. database_requests 受控写回协议

独立环境插件不能直接拿到主程序内存中的数据库对象，也不应自行连接工作流 SQLite 执行业务写入。

需要写数据库时，应返回受控请求，由主程序统一检查权限并执行。

推荐写表请求格式：

```json
{
  "operation": "write_table",
  "table_name": "src_demo",
  "headers": ["字段1"],
  "rows": [["值1"]],
  "mode": "replace"
}
```

规则：

1. 预览模式下主程序可以解析和校验请求，但不应真实写入。
2. 执行模式下，主程序按表权限、字段权限与危险级别统一检查。
3. 插件不应绕过主程序权限体系直接落库。

## 7. stdout 进度上报协议

独立环境插件无法直接使用 `context["report_progress"]`，因此通过 stdout 输出 JSON 行上报进度。

示例：

```python
print(json.dumps({
    "type": "node_progress",
    "current": i,
    "total": total,
    "message": f"正在处理 {i}/{total}"
}, ensure_ascii=False), flush=True)
```

主程序读取这些 JSON 行后更新当前节点进度。

建议：

1. 进度消息使用单行 JSON。
2. 重要阶段变化也可以输出摘要日志。
3. 避免 stdout 混杂大量非 JSON 调试文本。

## 8. 后台线程执行、取消与错误日志

主程序的计划 / 工作流执行采用后台线程方式运行，界面负责配置与显示，执行线程通过队列回传进度、结果、错误信息。

### 8.1 主程序内置环境插件的进度与取消

内置环境插件可以使用：

```python
context["report_progress"](current, total, message)
```

并定期检查：

```python
cancel_event = context.get("cancel_event")
if cancel_event is not None and cancel_event.is_set():
    ...
```

示例：

```python
def run(input_data, params, context):
    rows = input_data.get("rows", [])
    total = len(rows)
    for i, row in enumerate(rows, start=1):
        cancel_event = context.get("cancel_event")
        if cancel_event is not None and cancel_event.is_set():
            return {
                "ok": False,
                "message": "用户取消插件执行",
                "output": input_data,
                "logs": [{"level": "WARNING", "message": "用户取消插件执行"}]
            }

        # 插件业务逻辑

        if i == 1 or i % 100 == 0 or i == total:
            context["report_progress"](i, total, f"插件处理中 {i}/{total}")
```

### 8.2 独立环境插件的取消

独立环境插件由主程序负责终止外部进程，因此建议插件：

1. 定期输出进度。
2. 避免单个不可中断操作持续过久。
3. 将详细日志写入 `plugin_data_dir` 或 `log_dir`。

### 8.3 错误日志

工作流后台线程发生未捕获异常时，主程序会写：

```text
logs/workflow/workflow_error_时间戳.log
```

插件自己的详细日志仍建议写入：

1. `context["plugin_data_dir"]`
2. `context["log_dir"]`

界面上只显示摘要，避免大量日志阻塞 UI。

## 9. 插件内部缓存建议

复杂文件插件建议继续使用插件内部缓存，而不是主程序全局缓存。

推荐缓存位置：

```text
context["plugin_data_dir"]/cache.sqlite
```

建议缓存判定因子：

1. 文件路径
2. 文件大小
3. 修改时间 `mtime_ns`
4. 插件版本
5. 参数摘要
6. 可选文件 hash

这样即使主程序因为预览或字段分析多次调用插件，插件也能快速复用结果。

## 10. 多 UI 统一配置主线

这是本文最重要的新增部分。

插件后续不应再把“自带设置窗口”作为主要配置能力，而应切换到：

```text
插件
  -> 参数 schema
  -> 配置描述协议
  -> 配置 patch / action 协议
  -> 校验 / 预览
  -> 后端统一保存

Qt / Tkinter / .NET / Web / CLI
  -> 读取统一协议
  -> 渲染通用控件或专用 editor_kind
  -> 提交配置命令
  -> 不直接继承插件窗口逻辑
```

### 10.1 为什么要这样做

因为插件多 UI 兼容的难点，主要不在执行，而在配置：

1. 普通节点正在走 `node_ui_schema + headless` 主线。
2. 插件如果继续依赖 `open_config_window(...)`，就会成为架构中的最后一个大例外。
3. `.NET` / Web / Electron 不可能长期去兼容 Tkinter 子窗口。

一句话：

**插件的“怎么运行”和“怎么配置”必须拆开；配置必须能被任何 UI 消费。**

---

## 11. 插件配置能力分层

推荐将插件配置能力划分为四档：

| 档位 | 名称 | 用途 | UI 行为 |
| --- | --- | --- | --- |
| L0 | 旧窗口兼容 | 旧插件临时过渡 | 低优先级兼容入口，建议放入 compatibility menu，并提示不是推荐主路径 |
| L1 | 简单参数 schema | 字段、数字、路径、下拉、布尔值 | 自动表单渲染 |
| L2 | 结构化配置 | 映射表、规则列表、条件组、输入表规格 | 通用 `structured_list` / 表单编辑 |
| L3 | 专用配置协议 | 可视化映射、复杂规则编辑器 | 根据 `editor_kind` 打开专用编辑器 |

长期目标：

1. 新插件至少做到 L1。
2. 复杂插件应逐步迁到 L2/L3。
3. L0 只作为兼容层，不再作为推荐形态。

## 12. 配置描述协议

### 12.1 基础 schema

插件应优先通过 schema 描述基础参数：

```json
{
  "key": "path_field",
  "label": "文件路径字段",
  "type": "field_select",
  "required": true,
  "choices": ["完整路径", "文件名"],
  "help": "用于定位要处理的源文件。"
}
```

建议字段能力至少覆盖：

1. `label`
2. `type`
3. `default`
4. `help`
5. `required`
6. `choices`
7. `group`
8. `warning`
9. `placeholder`
10. `visibility / dependency`

### 12.2 describe_config

复杂插件可以额外提供：

```python
def describe_config(params, context):
    return {
        "schema_version": "your_plugin.config.v1",
        "summary": {},
        "views": [],
        "resources": [],
        "actions": [],
        "context": {},
        "models": {},
        "warnings": [],
    }
```

主程序可将其合并到标准 `plugin_config.v1` 结构中。

其中：

- `views`：描述可展示的配置区域，例如 `summary`、`form`、`structured_list`、`resource_list`
- `resources`：描述配置文件、缓存、外部资源，不要求 UI 直接操作真实路径
- `actions`：描述可触发动作，例如 `config_editor`、`import_config`、`export_config`
- `context`：提供候选值与上下文快照，例如表名、字段、规则名
- `models`：提供默认对象模板，供 UI 新增配置项
- `warnings`：提供迁移、兼容、损坏等非阻断提示

示例：

```json
{
  "schema_version": "DataFlowKit.visual_mapping.config.v1",
  "views": [
    {
      "view_id": "visual_mapping.rules",
      "title": "单元格映射规则",
      "kind": "structured_list",
      "editor_kind": "visual_mapping.rules",
      "config_path": ["plugin_settings", "configs", "default", "rules"],
      "columns": [
        {"key": "enabled", "label": "启用"},
        {"key": "name", "label": "规则"},
        {"key": "content_field", "label": "写入字段"}
      ],
      "items": []
    }
  ],
  "actions": [
    {
      "action_id": "visual_mapping.edit.rules",
      "label": "编辑单元格映射规则",
      "kind": "config_editor",
      "editor_kind": "visual_mapping.rules"
    }
  ]
}
```

## 13. 动态候选与 options resolver

很多插件不是固定字段表单，而是依赖当前配置和输入数据动态变化。

例如：

1. 选择某个输入表后，再刷新字段候选。
2. 选择某个模式后，再决定下一个参数可选项。
3. 读取当前工作流输入 headers、插件输入表 headers、transit table 后生成候选。

因此建议统一一个动态候选接口，例如：

```python
def resolve_options(config, context, request):
    ...
```

输入建议包含：

1. 当前完整 config
2. 当前 params
3. input tables 摘要
4. preview headers
5. table names / table columns
6. 请求字段 key
7. 触发依赖字段列表

返回建议包含：

1. `choices`
2. `warnings`
3. `disabled_reason`
4. `refresh_hints`

这样 Qt / .NET / Web 都能统一做依赖变化刷新，而不必各自重写候选推断逻辑。

## 14. 结构化 patch 协议

复杂插件经常包含：

1. 规则列表
2. 映射列表
3. 功能配置集
4. 嵌套条件
5. 输入表规格列表

这类结构不适合每次由 UI 整体覆写整个 `params`。推荐走局部 patch 协议。

当前主程序已经具备：

1. `validate_config_patch`
2. `apply_config_patch`

建议将其正式视为复杂插件的主路径能力。

示例 patch：

```json
{
  "section": "linked_rules",
  "action": "add",
  "payload": {
    "name": "规则1",
    "target_field": "结果字段"
  }
}
```

或者：

```json
{
  "section": "input_tables",
  "action": "update",
  "index": 0,
  "payload": {
    "source_type": "sqlite",
    "table_name": "src_demo"
  }
}
```

也可以进一步规范为：

```json
{
  "target": ["plugin_settings", "configs", "default", "rules"],
  "operation": "replace_item",
  "index": 0,
  "value": {}
}
```

主程序负责：

1. 检查目标路径是否允许修改。
2. 调用插件校验函数。
3. 统一保存配置文件。
4. 返回新的配置描述结果供 UI 刷新。

## 15. 配置效果预览接口

复杂插件配置后，用户通常需要提前知道：

1. 会增加哪些字段。
2. 会读取哪些输入表。
3. 会不会写数据库。
4. 是否需要独立环境。
5. 是否存在风险提示。

因此建议插件可选提供：

```python
def preview_config_effect(config, context):
    ...
```

返回建议包含：

1. `summary`
2. `warnings`
3. `issues`
4. `expected_output_fields`
5. `required_input_tables`
6. `side_effect_flags`

这个接口对 Qt 很有用，对后续 `.NET` / Web 同样重要。

## 16. 旧自定义窗口的定位

### 16.1 保留，但只作为兼容层

旧版：

```python
def open_config_window(parent, current_params, context):
    ...
```

可以继续保留一段时间，原因是：

1. 现有复杂插件已经在用。
2. 一次性废掉会影响旧插件可用性。
3. 某些复杂插件短期内还未完成 schema / patch 化。

但它的定位必须明确为：

**legacy fallback，不再作为推荐主路径。**

当前实现建议同时输出以下兼容状态字段：

1. `preferred=false`
2. `ui_role=fallback_action`
3. `ui_prominence=low`
4. `ui_placement=compatibility_menu`
5. `requires_confirmation=true`

### 16.2 各 UI 的处理建议

1. Tkinter 主程序
   - 继续允许调用旧窗口。
2. Qt
   - 可选兼容，但应明确提示“旧版插件设置窗口”，并按低优先级兼容入口展示。
3. `.NET` / Web / Electron / CLI
   - 默认不依赖旧窗口，提示该插件需要迁移到统一配置协议。

后续目标不是“让所有 UI 都兼容 Tk 窗口”，而是“把插件迁移到协议配置模式”。

## 17. 插件设计限制

为了让 Qt、.NET、Web、CLI 都能复用插件能力，插件应遵守以下限制：

1. 配置描述必须是 JSON 可序列化对象。
2. `describe_config` 必须无弹窗、无写入、无外部副作用。
3. 动态候选函数只能读取上下文并返回候选，不应修改配置。
4. 插件不应要求 UI 安装 Tk、Qt、WinForms、WPF 等 GUI 依赖。
5. 插件不能把业务规则只写在 UI 回调里；规则生成、校验、预览必须可由 headless 调用。
6. 大配置必须支持摘要、分页或限制条数，避免一次返回超大 payload。
7. 配置资源应有版本号和迁移策略。
8. 危险动作必须检查 `context["execute_actions"]`，预览模式不能真实执行。
9. 插件写数据库或文件应走主程序统一授权通道，不应绕过权限策略。

## 18. 新插件推荐开发流程

```text
1. 新建 plugins/插件ID/plugin.json
2. 新建 plugins/插件ID/plugin.py
3. 新建 plugins/插件ID/requirements.txt
4. 先实现基础 schema 配置
5. 如有复杂结构，再补 validate_config_patch / apply_config_patch
6. 如有动态候选，再补 resolve_options 或等效接口
7. 如需复杂可视化编辑，再补 describe_config + editor_kind
8. 再决定是否需要旧窗口兼容
9. 创建 plugin_envs/插件ID 虚拟环境
10. 安装 requirements.txt
11. 在主程序中测试预览 / 执行 / 取消 / 权限校验
```

建议默认顺序是：

**先协议化，再考虑是否还需要旧窗口。**

## 19. 当前实现状态与后续迁移方向

当前主程序实际上已经具备以下基础：

1. `PluginService.get_plugin_schema` 可输出插件基础 schema。
2. `PluginService.describe_plugin_config` 已能输出 `plugin_config.v1` 风格结果。
3. `PluginService.validate_plugin_config_patch` / `apply_plugin_config_patch` 已具备主干能力。
4. 外部插件协议已支持 `stdout` 进度、`database_requests`、`output.json`。
5. 旧版 `open_config_window` 仍保留为兼容入口。

后续迁移建议：

1. 先把现有 schema / patch 能力正式定义为插件主路径。
2. 给最复杂、最常用插件先做样板迁移。
3. 在 UI 中明确显示插件兼容能力与档位。
4. 逐步停止新增仅依赖旧窗口的插件。
5. 当 Qt 与 .NET 都能完成主要配置后，再把旧窗口降级为“高级兼容 / 调试入口”。

---

## 20. 当前一句话结论

DataFlowKit 当前的插件执行协议已经够用了，真正需要升级的是插件配置协议。

后续最稳的路线不是继续兼容越来越多的插件自带窗口，而是把插件配置正式收敛到：

- schema 描述
- 动态候选
- patch 编辑
- 配置效果预览
- 统一执行

旧 `open_config_window(...)` 可以继续保留，但只能作为兼容层，而不应再作为未来主路径。
