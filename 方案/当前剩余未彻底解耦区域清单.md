# 当前剩余未彻底解耦区域清单

> 目的：盘点 DataFlowKit 目前在“主程序逻辑 / Headless 引擎 / Qt 壳 / 插件协议”之间，哪些区域已经基本解耦，哪些区域还停留在旧窗口驱动模式，供后续分支拆分与任务排期使用。

---

## 1. 当前阶段判断

当前项目已经不处于“纯方案讨论”阶段，而是进入了“协议层基本成型，复杂区域继续收口”的阶段。

### 1.1 2026-06-24 当前基线

结合当前 `main` 分支代码，以下能力已经可以视为后续继续改造的基线，而不是待验证方向：

1. `HeadlessWorkflowEngine`、`StdioWorker`、`WorkflowFacade` 已形成多 UI 共享入口。
2. `node_ui_schema` 已覆盖节点目录、中文显示、说明、配置表单、动态候选、计划引用、运行时引用等核心 UI 描述。
3. `plan command` 已成为 Qt 和后续 UI 复用计划编辑语义的主路径。
4. `PluginService` 已成为插件配置协议中枢，旧 `open_config_window(...)` 已在协议里标记为 `legacy_fallback`。
5. 视觉映射插件已经具备复杂插件样板雏形：`describe_config / validate_config_patch / apply_config_patch / preview_config_effect / protocol_manifest / patch_schema / warning_schema`。
6. 插件 warning 与 patch 写回结果已经分别补入 `plugin_config_warning_target.v1`、`plugin_config_patch_target.v1`，Qt 可按 `view_id / target_index / target_id / focus_path` 做配置页签和条目定位。
7. 参数型插件已经开始补齐 `group / group_order / visible_when / enabled_when / options_source / empty_text / depends_on / refresh_on_change` 等 UI 元数据。
8. `TableDataService` 已提供 `data_source_service.v1`，剪贴板解析、字段规范化、首行提升、单元格 patch、搜索、保存/删除 SQLite、table handle/page 都已有 UI 无关服务入口，并已暴露给 headless / stdio / Qt client。
9. `data_source_service.v1` 已补 `table_actions` 描述，`list_tables / load_table / get_table_page / create_table_handle / get_table_handle_page / list_table_handles / release_table_handle` 可被 Qt / `.NET` / Web 发现和调用。
10. 高级筛选窗口已开始拆出 `advanced_filter_service.v1`：条件、匹配规则、输出字段、字段候选刷新等高频编辑动作已有 UI 无关 command/state service，旧 Tk 窗口动作层开始退化为适配器。

### 1.2 2026-06-25 增量基线

最新一轮收口后，可以再把以下能力视为当前基线：

1. 插件旧窗口入口已进一步转为 `describe_plugin_config().actions.open_legacy_config` 驱动，Qt 只消费协议动作，旧 schema 字段仅作为 fallback。
2. `legacy_plugin_config` 的确认 prompt 已由 `WorkflowFacade.describe_confirmation_prompt(...)` 统一生成，UI 不再自行拼兼容风险文案。
3. 可视化映射插件作为复杂插件样板，已补 `layout / ui_hints`，并通过 `PluginService` 透传到标准插件配置描述。
4. Qt 已消费复杂插件 `layout.view_order / default_view_id / ui_hints.view_hints`，插件配置页签可以按协议排序、默认切换和显示提示。
5. 数据源管理状态已补 `data_source_manager_layout.v1 / data_source_manager_ui_hints.v1`，`.NET` / Web 可从 stdio/headless 直接读取数据源窗口区域、动作优先级与提示信息。
6. 数据源共享服务已补 `data_source_client_profiles.v1 / data_source_transport_hints.v1`，把 Qt、stdio、`.NET`、Web 的推荐客户端 profile、分页传输、table handle 生命周期、编辑限制和删除确认语义放入共享 payload。

### 1.3 2026-06-25 后续收口基线

继续推进后，当前主干又可以新增以下基线判断：

1. Qt 数据源管理窗口已开始消费 `data_source_manager_layout.v1 / data_source_manager_ui_hints.v1`，窗口和控件会记录 section/action/prominence，按钮提示也来自共享 payload。
2. `advanced_filter_service.v1` 已补 `advanced_filter_command.v1 / advanced_filter_layout.v1 / advanced_filter_ui_hints.v1`，不再只是 command 名列表，后续 UI 可以按 section、command inputs、确认条件和提示来渲染高级筛选配置。
3. 高级筛选的 `filter_config_context.v1` 已透出 `service / command_schema / layout / ui_hints`，并通过 stdio 的 `describe_node_config_context` 验证。
4. 普通参数型插件在没有自定义 `describe_config().layout/ui_hints` 时，也会由 `PluginService.describe_plugin_config()` 生成默认 `plugin_config_layout.v1 / plugin_config_ui_hints.v1`。
5. Qt 插件参数表单已消费 `plugin_config_ui_hints.parameter_field_hints`，把 warning、placeholder、empty_text、advanced、unit/min/step 等提示合并到表单字段 schema。
6. stdio 测试已覆盖普通插件配置描述的 `plugin_config_layout.v1 / plugin_config_ui_hints.v1` 与高级筛选共享上下文的 `advanced_filter_layout.v1 / advanced_filter_ui_hints.v1`。
7. Qt 数据源管理窗口已开始读取 `data_source_transport_hints.v1`，默认分页大小、最大分页提示、推荐表传输 action、分页 action 与释放 action 会从服务描述同步到窗口属性。
8. 复杂插件 `structured_list` 的按钮顺序已通过 `view.action_state.button_order` 固定，Qt 可按协议渲染新增、更新、删除、启用、移动等按钮。
9. 插件旧窗口入口已补 `plugin_legacy_window_policy.v1`，明确旧 GUI 窗口只允许 Tk/Qt 兼容入口直接打开，`.NET` / Web / Electron / CLI 应显示迁移提示并走统一配置协议。
10. 普通参数型插件 metadata 已补 `plugin_parameter_field_state.v1`，`field_state_index` 可被 Qt/stdin/后续 UI 用来读取字段默认状态、条件依赖、候选刷新和提示文案。
11. `resolve_plugin_parameter_field_state` 已进入 `PluginService / HeadlessWorkflowEngine / StdioWorker / QtHeadlessEngineClient`，Qt 表单会优先消费共享运行时字段状态，减少前端自行解释 `visible_when / enabled_when`。

因此后续工作应避免重复讨论“是否要多 UI 解耦”，而应直接按以下方向推进：

1. 把旧复杂窗口能力迁成 schema / state / command / issue。
2. 把复杂插件的旧窗口能力继续拆到 `describe_config + patch`。
3. 让 Qt 继续消费共享协议，少增长插件或节点专用 UI 逻辑。
4. 为 `.NET` / Web 保持 stdio/headless payload 的稳定性。

目前已经相对稳定的主线是：

1. 节点稳定 ID 与兼容映射已经建立。
2. Headless 引擎已经能承担计划校验、预览、执行主循环。
3. `plan command` 已经把节点增删改移复制等计划编辑动作抽成共享命令。
4. `node_ui_schema` 已经能描述一部分节点菜单、说明、表单字段与中文显示。
5. Qt 壳已经不是演示样机，而是可实际操作的前端壳。

因此，当前的核心矛盾已经不是“方向是否正确”，而是：

- 旧主程序里仍有一批窗口直接参与业务逻辑。
- 若干复杂节点仍依赖旧窗口配置流程。
- 插件节点仍保留“插件自己弹窗口”的旧兼容路径。
- Qt 已经接入共享后端主线，但还没有覆盖全部复杂能力。

一句话总结：

**项目主干方向正确，基础设施已搭好，当前处于中后段收口期。**

## 2. 主程序与 Qt 当前状态对比

### 2.1 旧主程序的特点

旧主程序仍然带有明显的 Tkinter 主窗口时代特征：

1. 大量业务路径以 `window` 为中心传递上下文。
2. 配置、预览、执行、弹窗提示、字段选择、权限预检经常混在同一条调用链中。
3. 复杂节点往往依赖专门的 `Toplevel` 配置窗口，而不是纯 schema 驱动。
4. 插件支持旧式 `open_config_window(...)` 自定义设置窗口。

这类结构在单一 Tkinter 主程序内是可工作的，但会限制后续 Qt / .NET / Web / Electron 的复用。

### 2.2 Qt 壳当前的进展

Qt 壳已经明显比旧主程序更接近“前端壳 + 共享后端”的模式：

1. 通过 `QtHeadlessEngineClient` 接入 `HeadlessWorkflowEngine` 与 `WorkflowFacade`。
2. 节点目录、计划编辑、计划模板、基础校验/预览/执行、输出面板等主线功能，已优先走共享接口。
3. 一部分节点配置已经不再依赖旧 Tkinter 窗口，而是消费共享 schema。
4. 最近新增的“预览结果复用到后续节点配置字段候选”也已抽成共享模块，而不是只塞在 Qt 界面里。

但 Qt 目前仍然承担了较厚的控制器职责，包括：

1. 当前计划、输入表、预览结果、消息面板、作业状态的协调。
2. 一些动态字段候选与配置应用逻辑的中间拼装。
3. 对旧兼容路径的兜底处理。

因此，Qt 当前更适合定义为：

**已经接上共享后端主干的可用前端壳，但还不是完全轻量的纯渲染层。**

## 3. 已基本解耦的区域

以下区域已经具备继续扩展多 UI 的基础：

### 3.1 节点协议主干

已基本形成稳定方向：

1. `node_type_id` 作为稳定机器标识。
2. 旧中文 `type` 作为兼容显示层保留。
3. Headless / Qt / 后续 UI 都可围绕稳定 ID 建立逻辑。

### 3.2 计划编辑命令层

`plan command` 已经承担节点增删改移复制、配置 patch 等共享逻辑。

这意味着：

1. 不同 UI 不必各自重写“复制节点”“上移节点”“插入节点”等规则。
2. 后续 `.NET` / Web 可以直接复用同一套编辑语义。

### 3.3 Headless 执行主干

`HeadlessWorkflowEngine` 已经承接了：

1. 计划校验。
2. 计划预览。
3. 后台作业。
4. 一部分节点的无界面执行。
5. 计划模板、表格、输出、插件、表权限等多类服务的统一入口。

虽然还有部分复杂节点路径未完全纯化，但主引擎层已经站住了。

### 3.4 节点 UI schema 基础层

`node_ui_schema` 已经不只是菜单映射，而是在向“节点 UI 描述协议”发展：

1. 中文菜单。
2. 节点说明。
3. 风险/能力描述。
4. 表单字段基础信息。
5. 分类与排序。

这一层是后续所有 UI 兼容的关键基础。

### 3.5 预览结果复用给节点配置

最近补入的共享模块已经说明，项目开始把“界面体验问题”沉回共享逻辑：

1. 用户显式预览到某节点后，后续节点配置字段候选可复用前序结果。
2. 打开节点配置不再自动触发后台重跑。
3. 这类上下文复用可以被 Qt / 其他前端共同消费。

这属于很好的解耦方向示例。

### 3.6 插件配置兼容状态协议

插件旧窗口目前已经不再被描述为“普通配置入口”，而是通过 `plugin_config_compatibility.v1` 标记：

1. `primary_config_path`：说明当前插件推荐走 `schema_patch`、`describe_config`、`schema_form` 还是旧窗口。
2. `legacy_lifecycle`：旧窗口统一标为 `legacy_fallback`。
3. `migration_target`：明确迁移目标为 `describe_config + parameter_metadata + config_patch`。
4. `legacy_fallback_available`：允许旧 UI 暂时兜底，但不给新 UI 增加长期绑定。
5. `preferred=false`：明确旧窗口不是推荐主入口。
6. `ui_role=fallback_action / ui_prominence=low / ui_placement=compatibility_menu`：告诉 Qt、`.NET`、Web 将旧窗口入口放在低优先级兼容位置。
7. `requires_confirmation=true`：提示 UI 打开旧窗口前应说明兼容风险。
8. `legacy_window_policy`：明确旧窗口的直接打开边界，Tk/Qt 可兼容，`.NET` / Web / Electron / CLI 不直接承接插件 GUI 窗口。
9. `unsupported_behavior=show_migration_prompt`：告诉非 Tk/Qt UI 应显示迁移提示，而不是尝试嵌入旧窗口。

这解决了“旧插件窗口到底算不算正常能力”的口径问题。后续迁移时，应优先补协议路径，而不是继续扩旧窗口兼容。

### 3.7 数据源共享服务协议

数据源能力已经从 Qt/Tk 控件逻辑中抽出一层共享服务：

1. `data_source_state.v1`：描述当前表格来源、脏状态、行列数量。
2. `data_source_actions.v1`：描述保存、删除等动作的可用性与确认需求。
3. `data_source_service.v1`：告诉 Qt / stdio / `.NET` / Web 可调用哪些数据源动作。
4. `table_page.v1` 与 table handle：为后续大表分页和跨 UI 传输做准备。
5. `table_actions`：固定表相关动作的 action id、engine action、结果 schema，避免每个 UI 自己猜表载入/分页/释放句柄的调用方式。
6. `data_source_client_profiles.v1`：给 Qt、stdio、`.NET`、Web 定义推荐客户端 profile 和 state/render 策略。
7. `data_source_transport_hints.v1`：给多端 UI 定义大表优先 handle/page、释放句柄、分页大小、分页编辑限制、保存/应用 action 和删除确认字段。

这说明“输入数据源管理”已经进入第四阶段收口：服务层与接口已建立，Qt 已开始消费分页传输提示，后续重点是继续减少按钮状态、危险确认和独立输入库状态的前端硬编码。

## 4. 当前剩余未彻底解耦区域清单

以下按三档划分：

- 必须先做
- Qt 阶段建议做
- 可以留给 .NET / 多端阶段

## 5. 必须先做

这些区域如果不继续收紧，Qt 只能“部分接主线”，后续 .NET / Web 也会重复踩坑。

### 5.1 复杂节点配置仍依赖旧窗口

典型区域：

1. 高级筛选
2. 分组节点
3. 循环节点
4. 跳转节点
5. 表权限层相关窗口
6. 某些写回 / 输出配置

最新进展：

1. 高级筛选已经新增 `workflow/advanced_filter_command_service.py`，提供 `describe_advanced_filter_state(...)`、`apply_advanced_filter_command(...)`、`describe_advanced_filter_service(...)`。
2. 已覆盖条件增删清空、匹配规则增删清空、输出字段增删清空、字段候选刷新、无效规则过滤、结构化确认提示。
3. 已继续覆盖预览构建、预览去重、载入主预览快照、模板导出/应用等结果与模板语义。
4. `advanced_filter_service.v1` 已继续补 `save_template_file / load_template_file`，模板文件读写、备份恢复提示和 `advanced_filter_template_file.v1` 结果都可由共享服务返回，UI 只负责选择文件路径。
5. 高级筛选节点配置 helper 已补 `filter_config_template.v1 / filter_config_template_file.v1`，通过 `apply_node_config_command` 可在 stdio/headless 路径保存/载入节点版高级筛选模板，并保留 `value_source/right_table/remove_duplicates` 等节点特有字段。
6. `advanced_filter_service.v1` 已补 `save_preview_to_table`，预览结果保存到 SQLite 表走 `TableDataService.save_table`，返回 `advanced_filter_save_result.v1`，不再由旧 Tk 动作层直接写表。
7. `advanced_filter_service.v1` 已作为一等 stdio/headless action 暴露：`describe_advanced_filter_service / describe_advanced_filter_state / apply_advanced_filter_command`，专用 Qt/.NET/Web 面板可以直接消费，不必绕旧 Tk 窗口。
8. 旧 `advanced_filter_window_actions.py` 中规则/输出字段/字段刷新/预览/去重/模板/保存预览结果相关操作已改为复用该 service，Tk 层主要保留控件同步、弹窗确认、文件选择等适配行为。
9. 高级筛选节点配置面板已通过 `workflow/filter_config_helpers.py` 增加 service 适配层，条件/匹配规则的新增删除、输出字段全选/反选开始复用 `advanced_filter_service.v1`，同时保留节点配置特有的 `value_source/right_table` 展示格式。
10. 高级筛选节点配置面板已开始使用统一的 options/state 打包结果来驱动字段刷新、风险提示和输出文本，旧 UI 只保留少量联动协调。

仍未完成：

1. 模板文件对话框仍由 UI 负责，但读写和恢复语义已经进入共享 command。
2. 节点配置面板的值候选刷新和最终保存同步仍有少量 Tk UI 协调逻辑，需要继续向 service/options resolver 合并。
3. Qt 尚未提供完整的高级筛选专用配置面板，但已经可以从 `filter_config_context.v1` 取得 `advanced_filter_service/layout/ui_hints`，后续应按该协议扩展，而不是移植 Tk 窗口。

现状问题：

1. 配置编辑依赖专门窗口而不是纯 schema。
2. 窗口内部往往同时承担数据源选择、规则编辑、预览、校验、保存。
3. 配置结构事实来源仍部分藏在 Tkinter 子窗口中。

为什么必须先做：

因为这些节点不先协议化，Qt 只能继续兜底做兼容，后续其他 UI 也无法共享完整配置体验。

建议方向：

1. 先把旧窗口配置数据结构彻底抽象成标准 config JSON。
2. 再补对应 schema / command / validator / option resolver。
3. 最后让 Qt 以 schema 面板替代旧窗口配置路径。

### 5.2 运行时仍存在大量 `window` 参与的服务链

现状问题：

1. 多个运行时模块函数签名仍是 `...(window, ...)`。
2. 表读写、输出写回、分组、循环、插件、文件节点等，仍有窗口上下文参与。
3. 某些服务实际上依赖窗口上的状态变量、方法或弹窗函数。

影响：

1. Headless 能力边界会被旧窗口习惯拖住。
2. 将来做 worker / stdio / HTTP / .NET 时，这些能力不好直接搬走。

建议方向：

1. 先把 `window` 依赖梳成显式 service/context。
2. 把 UI 需要的反馈改成 issue/result/event，而不是直接 `messagebox`。
3. 保留旧窗口包装层，但执行主逻辑迁到纯 service。

### 5.3 插件“自带窗口”仍是多 UI 最大阻碍

现状问题：

1. 插件仍可通过 `open_config_window(...)` 自行弹出设置窗口。
2. 这条路默认绑定 Tkinter/旧主程序上下文。
3. 当前协议已把它降为低优先级兼容入口，但旧窗口内部的复杂配置能力尚未全部被 schema/patch 覆盖。
4. Qt 即使能临时兼容，也很难做到真正统一体验。

为什么必须先做：

因为普通节点已经在走 schema-first 路线，如果插件还长期依赖“自己弹窗”，就会形成双轨体系：

1. 普通节点可以多 UI 兼容。
2. 插件节点只能依附某个特定 UI。

建议方向：

1. 明确把插件 schema 配置作为主路径。
2. 保持旧 `open_config_window(...)` 为 low prominence legacy fallback，不再增加新 UI 专用兼容能力。
3. 补动态候选、局部 patch、配置校验等插件接口，逐步替代旧窗口。
4. 继续把 warning/patch target 扩展到复杂插件更多编辑器，让 UI 能定位问题和写回结果。

### 5.4 复杂节点的动态候选 / 联动仍不够统一

现状问题：

1. 字段候选、表候选、循环目标、跳转目标、插件动态参数，仍有部分逻辑散在 UI 或旧窗口路径里。
2. 插件参数字段状态已开始有统一运行时 resolver，但其他节点/复杂窗口的候选和联动仍未全部接入同一口径。
3. 并不是所有候选项都能通过统一接口获得。

影响：

1. Qt 会继续背较多控制器逻辑。
2. 新 UI 接入时要重新拼一次候选逻辑。

建议方向：

1. 补统一的 options resolver / picker context / dependency 解析能力。
2. 把“依赖哪些字段变化时刷新候选”也描述进 schema 或专用接口。

---

## 6. Qt 阶段建议做

这些工作不一定必须立即先做，但在 Qt 阶段完成，能明显减少后续返工。

### 6.1 扩大 `node_ui_schema` 对复杂节点的覆盖

当前 schema 基础已经有了，但还需要继续提高覆盖度：

1. 从基础数据节点扩到复杂流程节点。
2. 从简单字段面板扩到规则列表、映射表、条件组合、输出清单等结构。

建议目标：

1. Qt 里的节点配置尽量都优先走 schema。
2. 只有极少数 legacy 节点还走兼容路径。

### 6.2 继续削薄 Qt 主窗口控制器职责

现状：

Qt 主窗口已经不是纯 UI，它还承担状态协调与结果拼装。

建议继续下沉：

1. 选择反馈、消息面板、作业状态说明更多交给 facade 统一生成。
2. 节点详情、字段候选、配置应用结果尽量都走标准 payload。
3. Qt 负责显示与用户操作，不继续增长业务判断分支。

### 6.3 把旧“管理窗口”改造成面板状态 + command

优先关注：

1. 表权限层
2. 跳转管理
3. 分组模板/节点模板类编辑器
4. 复杂筛选规则编辑器

目标不是马上完全重写视觉，而是先把其交互抽象成：

1. state
2. command
3. issues/result

这样 Qt 和后续其他 UI 才能复用同一套行为语义。

### 6.4 插件 schema 面板能力补强

Qt 阶段应优先把插件 schema 面板做强，而不是继续扩旧插件窗口兼容。

建议优先补：

1. 动态候选字段/表名刷新。
2. 插件输入表规格编辑。
3. 参数分组、警告、运行环境说明。
4. 复杂列表型参数的 patch 式编辑。

## 7. 可以留给 .NET / 多端阶段

这些工作更偏长期结构优化，不必都压在当前 Qt 阶段完成。

### 7.1 大表跨 UI 传输优化

当前很多路径仍以 `headers + rows` 作为直接交换格式。对于本地 Qt 壳问题不大，但对多端来说还不够长期化。

后续可考虑：

1. 表句柄
2. 分页读取
3. 延迟加载
4. 结果引用而不是整表搬运

### 7.2 更稳定的前后端 API 边界

当前 `WorkflowFacade` 已经有雏形。后续多端阶段可以进一步明确：

1. request schema
2. result schema
3. event schema
4. job state schema

### 7.3 真正的多端统一状态协议

到 `.NET` / Web 阶段，再系统化处理：

1. 页面状态对象
2. 面板状态对象
3. 统一错误/警告/提示模型
4. 统一命令回执与用户确认协议

## 8. 插件多 UI 兼容的重点结论

插件是当前最需要单独看待的一块。

### 8.1 当前插件体系的优点

目前插件体系已经有较好的基础：

1. `plugin.json` 可描述插件元数据。
2. 插件 schema 已可描述基础参数。
3. 独立环境运行协议已经建立。
4. `input.json / output.json / stdout JSON lines` 路线明确。

这说明插件执行协议方向是对的。

### 8.2 当前插件体系的主要问题

主要问题不在执行，而在配置：

1. 有些插件仍把“如何配置”封装在 Tkinter 自定义窗口中。
2. 自定义窗口往往自带复杂交互、规则编辑、输入表选择、动态候选。
3. 这些交互如果没有协议化，换 UI 就要重做一遍。

### 8.3 推荐长期方案

插件配置协议建议收敛成以下层次：

1. `describe_config`
   - 返回 schema、默认值、分组、警告、说明。
2. `resolve_options`
   - 根据当前 config / context / input tables 返回某字段候选项。
3. `validate_config_patch`
   - 校验局部改动是否合法。
4. `apply_config_patch`
   - 用于复杂列表、规则集、映射集的结构化修改。
5. `preview_config_effect`
   - 返回配置摘要、输入输出字段影响、预估行为。
6. `run`
   - 真正执行插件。

这样不同复杂度的插件都能分层接入：

1. 简单插件：只要 schema + run。
2. 中等插件：schema + options + validate。
3. 复杂插件：再补 patch / preview 接口。

### 8.4 旧自定义窗口的定位

建议将旧 `open_config_window(...)` 明确定位为：

**兼容旧插件的临时 fallback，不再作为后续插件主路径。**

原因：

1. 它天然绑定某个 UI 技术栈。
2. 它难以标准化测试。
3. 它会让 Qt / .NET / Web 继续出现能力分叉。

---

## 9. 推荐执行顺序

如果后续继续按“解耦优先、Qt 落地、再考虑 .NET”推进，建议顺序如下：

1. 先继续处理复杂节点配置协议化。
2. 同步把运行时 `window` 依赖继续替换成 service/context。
3. 把插件配置主路径切到 schema / options / patch / validate。
4. Qt 继续优先消费共享协议，不再扩写死逻辑。
5. 等 Qt 基本稳定后，再考虑 `.NET` 壳复用同一套接口。

---

## 10. 当前一句话结论

当前项目已经完成“方向验证”和“大框架搭建”，剩余工作的重点不再是重想架构，而是把复杂节点、旧窗口逻辑、插件配置窗口这几块彻底拉回共享协议层。

只要这几块继续按当前方向收口，后续 Qt 稳定、`.NET` 接壳、甚至 Web/Electron 扩展，都会轻松很多。
