## 方向分析与方案

### 先定性：这个项目是什么

DataFlowKit 本质是一个**数据工作流编排工具**，核心价值在于：节点可视化编排 + 文件/Excel 批处理 + 灵活的多表匹配写回。性能瓶颈不在计算，而在 **I/O 密集**（大量读写 Word/Excel 文件）和**中等规模数据处理**（`MAX_EXPANDED_ROWS = 200,000`）。

这个定性直接决定了技术选型。

---

### 一、是否需要 C/C++ 加速？

先看真正的瓶颈：

| 瓶颈点 | 当前实现 | 是否需要 C/C++ |
|---|---|---|
| 多表 hash join（filter_engine） | Python 纯循环，20 万行 | 不需要，改用 SQLite 内查询或 pandas 可解决 |
| Word/Excel 文件读写 | python-docx / openpyxl / win32com | 不需要，瓶颈在 I/O 不在计算 |
| 正则批量提取 | Python re 模块（C 扩展） | 不需要，re 已是 C 实现 |
| 大量行数字运算 | Python Decimal 逐行循环 | 如果超 50 万行可考虑 numpy |
| 图像/AI/信号处理 | 不涉及 | 不适用 |

**结论：当前规模（~20 万行）C/C++ 性价比极低，numpy/pandas/SQLite 就能解决。真正要上 C 的场景是超过千万行的列式运算，目前不是这个项目的应用场景。**

如果未来确实有这个需求，可以用 Python C 扩展（pybind11）给 `workflow/nodes/data_nodes.py` 里的热点函数加速，因为它们已经是纯函数了，接口最干净。

---

### 二、界面替换方向

界面是当前最值得投入的方向，主要因为：

1. Tkinter 在高 DPI 显示器上缩放差
2. 节点可视化（连线、拖拽）Tkinter 做起来很吃力
3. 数据预览表格性能（大量行虚拟滚动）Tkinter 不擅长

对比各选项：

| 方案 | 优势 | 劣势 | 适合度 |
|---|---|---|---|
| **Qt（PyQt6 / PySide6）** | 原生性能好、高 DPI、虚拟表格成熟（QTableView）、在 Python 里可无缝调用现有逻辑 | 打包稍大、商业授权需考虑 | ⭐⭐⭐⭐⭐ |
| **Electron + TypeScript** | Web 技术栈灵活、节点画布生态好（React Flow / xyflow） | 内存重（~150MB 起）、Python 通信需 IPC | ⭐⭐⭐ |
| **.NET (WinForms/WPF/MAUI)** | Windows 原生体验好 | 只能 Windows、需重写 Python 逻辑或 IPC | ⭐⭐ |
| **Flutter** | 跨平台渲染一致 | 生态缺专业数据工具组件、Dart 与 Python 通信复杂 | ⭐ |
| **纯 Python 继续优化（加 Dear PyGui）** | 改动最小 | 性能上限有限、节点画布仍然麻烦 | ⭐⭐⭐ |

**推荐方向：PyQt6 / PySide6。**

原因：

- Python 直接 import，不需要 IPC，现有 `workflow/nodes/*.py` 纯函数一行不用改
- `QTableView + QAbstractTableModel` 的虚拟滚动支持百万行不卡
- `Qt Quick`（QML）或 `Qt Graphics View Framework` 可以做节点连线图
- 打包直接用 PyInstaller，和现在流程一样
- PySide6 是 LGPL，商业使用无障碍

---

### 三、如果要 Electron 前端（更远期）

Electron 的主要吸引力是 React Flow 这类节点画布组件非常成熟。如果走这条路，Python 就退化成纯后端服务，界面和引擎彻底分离。这是更彻底的架构转型。

通信方案：

```
Electron (前端)
     ↕ HTTP / WebSocket / Named Pipe
Python (后端：FastAPI + uvicorn)
     → workflow/executor.py
     → workflow/nodes/*.py
     → db/table_manager.py
```

现有的 `workflow/nodes/*.py` 纯函数签名 `(headers, rows, config, context)` 天然适合包 HTTP 接口——`headers/rows` 序列化成 JSON 传递，`context` 在服务端维护 session。

但这个方向改动量大，适合项目稳定后的长期规划，而不是现在。

---

### 四、节点扩展如何处理

这是核心问题。当前项目已经有完整的插件协议（`run(input_data, params, context)` + 独立子进程两种模式），节点扩展有三个层次：

**层次 1：Python 插件（现在就支持）**

直接实现 `run(input_data, params, context) → output_data` 接口，放进 `plugins/` 即可。插件可以用任意 Python 库（pandas、numpy、PIL 等），主程序会加载并在工作流中调用。这是现在的机制，可以继续沿用。

**层次 2：任意语言的独立进程插件（现在已支持框架，可完善）**

现有的插件已经有 `argparse` + `input.json/output.json` + stdout JSON 进度模式（`_run_external_entry`）。这意味着 **C++/Go/Rust/.NET 写的程序只要能读 JSON、写 JSON、往 stdout 输出进度行，就能作为节点接入**，不需要任何 Python 绑定。

这个协议值得固化成文档，关键字段：

```json
// input.json
{
  "headers": ["字段A", "字段B"],
  "rows": [["值1", "值2"]],
  "params": { "节点配置": "..." },
  "context": { "db_path": "..." }
}

// output.json
{
  "headers": ["字段A", "字段B"],
  "rows": [["结果1", "结果2"]],
  "logs": [{ "level": "INFO", "message": "..." }]
}

// stdout 进度（每行一个 JSON）
{"type": "progress", "current": 10, "total": 100, "message": "..."}
```

这样 C++ 写一个高性能文本处理节点、Go 写一个并发下载节点、甚至 Excel VBA 调用的工具，都可以无缝挂进工作流。

**层次 3：远端节点（未来可考虑）**

如果走 HTTP 后端路线，节点可以是远端 API 调用——工作流引擎发 POST 请求，等返回结果继续下一节点。适合把 AI 推理、云端数据服务等接入工作流。

### 六、一句话总结

**当前最值得做的是 PyQt6 替换 Tkinter**——改动只在 `ui/` 层，`workflow/nodes/*.py` 的纯函数完全不动，数据处理能力保留，但界面质量、虚拟表格性能、节点画布可能性都大幅提升。C/C++ 加速现在不是瓶颈，等数据规模真正超过百万行再说。节点扩展当前的 JSON 协议框架已经支持任意语言接入，优先固化协议文档，不需要重造机制。
