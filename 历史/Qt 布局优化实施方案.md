# Qt 布局优化实施方案

## 1. 背景

当前 Qt 工作流界面已经具备较完整的节点配置、节点说明、执行进度、输出设置、结果预览、消息反馈等能力，但右侧区域在实际使用上偏紧凑，尤其存在明显的 Y 轴压力。

当前右侧大致为：

- `4. 节点配置`
- `节点说明`
- 动作按钮行
- `执行进度`
- `5. 输出设置`
- `6. 结果预览`
- `说明 / 问题 / 日志`

这种布局的问题不是功能少，而是：

1. 高频主工作区被切成多个小块，配置和预览都不够舒展。
2. 辅助信息长期占位，例如节点说明、输出设置、消息区，不是每一刻都必须同时可见。
3. 进度区域视觉占比偏大，但信息密度不高。
4. 右侧更像“功能堆叠”，不像“主工作区 + 辅助页”的结构。


## 2. 优化目标

这次优化不以增加功能为主，而以提高 Qt 界面的实用性为主，目标如下：

1. 减少右侧纵向堆叠，释放表单和表格的可用高度。
2. 保留现有功能，不破坏当前协议驱动方向。
3. 让高频任务链路更顺手：选节点、改配置、看结果、看输出、看消息。
4. 让布局更接近“工作区页签 + 辅助状态”的形态，方便后续继续演进。


## 3. 总体布局方向

建议保持左侧整体结构不动，主要重构右侧。

### 3.1 左侧保持现状为主

左侧继续保留：

- `1. 输入数据源`
- `2. 工作流节点`
- `3. 计划模板`

原因：

- 左侧的职责已经比较清晰，主要是数据源、节点树、节点列表、模板入口。
- 当前紧凑感的主要问题不在左侧，而在右侧大量纵向堆叠。


### 3.2 右侧改成“三段式”

推荐将右侧重组为三段：

1. 上半区：`节点工作区 Tabs`
2. 中间区：动作按钮 + 紧凑进度
3. 下半区：`结果工作区 Tabs`

推荐结构：

- `节点工作区 Tabs`
  - `配置`
  - `说明`
- 动作按钮行
- 紧凑版 `执行进度`
- `结果工作区 Tabs`
  - `预览`
  - `输出`
  - `消息`

这样可以把“长期占高的辅助区块”改成“按需切换的页签区”。


## 4. 具体布局改造方案

## 4.1 节点配置区与节点说明区合并

### 当前问题

- `4. 节点配置` 和 `节点说明` 是上下两个独立区块。
- 节点说明长期占位，但并不是每次改配置都需要同步查看。
- 当节点表单较长时，配置区会显得比较挤。

### 建议改法

将这两个区块合并为一个 `节点工作区 Tabs`：

- `配置` 页
  - 保留当前 `config_header_label`
  - 保留 `NodeConfigForm`
  - 保留“应用节点配置”按钮

- `说明` 页
  - 放当前 `node_detail_title_label`
  - 放当前 `node_detail_meta_label`
  - 放当前 `node_detail_badges_label`
  - 放当前 `node_detail_sections`

### 预期收益

- 节点配置区能获得更多可用高度。
- 节点说明从“常驻占位”改成“按需查看”。
- 结构上更符合真实使用频率：大多数时候在“配置”，偶尔切到“说明”。

### 推荐交互规则

- 默认打开 `配置` 页。
- 切换节点后，默认仍停留在 `配置` 页。
- 如果未来要增强说明使用率，可以在用户主动切到 `说明` 页后保持页签状态，但第一版不建议做复杂记忆逻辑。


## 4.2 输出设置合并到结果区页签

### 当前问题

- `5. 输出设置` 独立占据一整段高度。
- 实际使用中，输出设置通常是在“执行前配置”或“结果后调整”，不是需要长期与表格同时展开查看的区块。
- 它与下方结果预览一起出现时，会压缩预览表格可视空间。

### 建议改法

将 `5. 输出设置` 不再作为独立 group box，而是并入 `结果工作区 Tabs` 的 `输出` 页。

`输出` 页建议放：

- 输出方式
- 输出表名
- SQLite 路径
- xlsx 输出路径
- 覆盖前备份选项

### 预期收益

- 结果区主页面能把更多空间留给表格。
- 输出配置时，用户进入专门页签，不再在狭窄区块中操作。
- 结果区结构更清晰：看表在 `预览`，配输出在 `输出`，看消息在 `消息`。


## 4.3 结果预览与消息区重组

### 当前问题

- `6. 结果预览` 下方还有独立的 `说明 / 问题 / 日志` 标签。
- 这会导致结果区其实被切成“表格 + 固定高度消息区”两层。
- 当表格列较多、行较多时，表格高度很容易不够。

### 建议改法

将右下区域改为 `结果工作区 Tabs`：

- `预览`
- `输出`
- `消息`

其中：

#### `预览` 页

放：

- `输入表` / `结果表` 按钮
- `查看表` 下拉框
- 表格标题
- `QTableView` 表格主体

说明：

- 这个页签的目标是尽量给表格本体更大的高度。
- 若保留“显示日志”按钮，建议弱化；更推荐直接让用户切到 `消息` 页。

#### `输出` 页

放当前输出设置表单。

#### `消息` 页

保留当前消息三级标签：

- `说明`
- `问题`
- `日志`

### 为什么建议保留“消息页内再分子标签”

因为现在后端已经形成了：

- `message_panel.info_body`
- `message_panel.issue_body`
- `message_panel.logs`
- `message_panel.preferred_tab`

这说明消息结构本身已经稳定，没必要为了压缩布局而把三类消息硬塞成一块文本。

因此推荐结构为：

- 大标签：`预览 / 输出 / 消息`
- 小标签：`说明 / 问题 / 日志`

这是当前最平衡的一版。


## 4.4 执行进度区压缩

### 当前问题

- 进度区目前高度偏高。
- 两条进度条本身信息有用，但视觉占位偏大。

### 建议改法

保留“双进度”结构，但压缩样式：

- 进度条高度减少约一半
- 减少 group box 内 margin / spacing
- 保留两行文本标签，但尽量紧凑

推荐目标：

- 工作流进度条：细条
- 节点进度条：细条
- 整个 `执行进度` 区块变成轻量状态区，而不是一个大块面板

### 为什么不建议第一版直接只留一条进度

因为现在已有：

- workflow 级进度
- node 级进度

这两层语义还是有区别的。第一版先压缩，不先合并，风险更低。


## 5. 建议的最终结构

推荐第一版改造后的整体结构如下：

### 左侧

- `1. 输入数据源`
- `2. 工作流节点`
- `3. 计划模板`

### 右侧

- `节点工作区 Tabs`
  - `配置`
  - `说明`
- 动作按钮行
- 紧凑版 `执行进度`
- `结果工作区 Tabs`
  - `预览`
  - `输出`
  - `消息`
    - `说明`
    - `问题`
    - `日志`


## 6. 对当前代码结构的影响范围

这次改造主要是 UI 布局层面的重组，不应改变核心逻辑。

预计主要影响：

- `ui_qt/main_window.py` 中 `_build_right_panel()`
- 相关 widget 成员变量的挂载位置
- 某些按钮的父容器与布局层级
- `show_log_text()`、消息页切换、预览页切换等少量交互入口

原则上不应影响：

- headless 引擎
- protocol / schema
- facade 的业务返回
- 节点配置应用逻辑
- 输出设置协议逻辑
- 消息面板协议逻辑

也就是说，这一版应尽量做成：

**布局重构，不做业务行为重构。**


## 7. 实施顺序建议

建议按低风险顺序拆成三步：

### 第一步：节点工作区 Tabs

先把：

- `节点配置`
- `节点说明`

合并成一个页签容器。

原因：

- 逻辑影响小
- 收益明显
- 能先释放右侧上半区空间

### 第二步：结果工作区 Tabs

再把：

- `输出设置`
- `结果预览`
- `消息区`

重组为 `预览 / 输出 / 消息`。

这是收益最大的一步，也是最能缓解 Y 轴压力的一步。

### 第三步：进度区压缩与细节收口

最后再处理：

- 进度条高度
- group box 间距
- toolbar/button 行 spacing
- 某些文字标签的高度与 margin

这样更容易在前两步结构稳定后做细调。


## 8. 需要注意的细节

### 8.1 不要让页签过度嵌套得太复杂

推荐只接受一层大标签 + 一层消息小标签。

不建议再把输出页内部继续拆太多 tab，否则只是把纵向复杂度改成横向复杂度。


### 8.2 页签切换要和现有反馈逻辑兼容

例如：

- 某次执行后如果 `message_panel.preferred_tab = issues`
- 并且当前消息在 `消息` 大页签内展示

则需要决定是否：

1. 只切换消息内部子标签
2. 还是顺带把外层大页签切到 `消息`

建议第一版规则：

- 如果收到明显错误/警告反馈，可自动切到 `消息` 大页签
- 然后再按 `preferred_tab` 切内部子标签

这样用户更容易看到关键反馈。


### 8.3 `预览` 页要避免再次被底部工具挤碎

`预览` 页的目标是给表格更多空间，因此建议：

- 工具条尽量单行
- 标题尽量轻量
- 不再在预览页底部固定挂一块日志区域


## 9. 推荐结论

当前 Qt 界面的实用性优化，最合适的方向不是继续增加控件，而是把右侧从“多个竖向 group box 堆叠”改成“两个主工作区页签 + 一个轻量状态区”。

优先落地建议为：

1. `节点配置` 与 `节点说明` 合并为 `配置 / 说明` 页签
2. `结果预览`、`输出设置`、`说明/问题/日志` 合并为 `预览 / 输出 / 消息` 页签
3. `执行进度` 保留双进度语义，但整体压缩一半高度

这版方案的优点是：

- 变化集中在布局层
- 不破坏当前协议解耦方向
- 能显著减轻右侧 Y 轴压力
- 为后续继续打磨 Qt 实用性留下清晰结构基础
