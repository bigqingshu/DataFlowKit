# DataFlowKit 界面/逻辑解耦方案（Qt6 + Headless Engine）

> 目标：让 Qt 界面尽量薄，把界面里残留的业务逻辑下沉到后端 `engine/`，使前端可随时替换（Qt / Electron / .NET / Web），并补齐当前“只显示未实现”的执行与输出能力。
>
> 本方案基于对实际代码的逐行核查，不修改任何代码。

---

## 一、现状评估：解耦的地基已经打得很好

先肯定已经做对的部分，避免重复造轮子。

| 已有能力 | 位置 | 说明 |
|---|---|---|
| **Headless 引擎 facade** | `engine/headless.py` `HeadlessWorkflowEngine` | UI-free，提供 `validate_plan / preview_plan / run_plan / list_node_catalog / get_node_ui_schema / make_default_node` |
| **节点 UI schema 系统** | `workflow/node_ui_schema.py` `build_node_ui_schema()` | 已能产出完整 `form.groups`，每个字段含 `key/label/type/choices/default/help` |
| **纯函数数据节点** | `workflow/nodes/*.py` | 无 `self.`、无 tkinter，可独立测试 |
| **跨语言 stdio 协议** | `engine/stdio_worker.py` `StdioWorker` | JSON-lines 协议，非 Python 前端可直接接 |
| **原子化配置读写** | `shared/atomic_json_utils.py` | 带备份/损坏恢复 |

**关键结论**：Qt 层（`ui_qt/`）当前只 import 了 `engine`、`shared.atomic_json_utils`、`workflow.node_ui_schema`，**没有**绕过引擎直接调用 `workflow.nodes.*` 或 `db.*`。边界基本干净。问题集中在两类：

1. **少量业务逻辑仍写在 Qt 里**（计划编辑、计划装配、字段类型推断、表格解析）。
2. **执行与输出只做了显示，没接后端实现**（用户已指出）。

---

## 二、UI 里残留、应下沉到后端的逻辑清单

> 每条都标注了文件、函数、行号，以及建议下沉到的后端接口。

### A. 计划编辑操作（节点增删改移复制）— 优先级：高

**位置**：`ui_qt/main_window.py`
- `add_node_by_type`（403-417）：插入位置规则 `insert_at = indexes[-1] + 1 if len(indexes)==1 else len(nodes)`（413）
- `delete_selected_node`（419-429）
- `move_selected_node_up / down`（431-449）
- `toggle_selected_node_enabled`（451-458）
- `copy_selected_node`（460-471）：重置 `node_id=""`、改名加 `_复制` 后缀（466-467）
- `clear_nodes`（473-476）

**问题**：这些直接 mutate `self.current_plan["nodes"]`，包含“复制时如何改名”“插入到哪”等业务规则。换前端就要重写一遍，且无法单测。

**建议**：在引擎增加无副作用的计划编辑 API（输入 plan + 操作，返回新 plan）：

```python
engine.plan_insert_node(plan, node_type_id, at=None) -> plan
engine.plan_delete_nodes(plan, indexes) -> plan
engine.plan_move_node(plan, index, direction) -> plan
engine.plan_toggle_node(plan, index) -> plan
engine.plan_duplicate_node(plan, index) -> plan   # 含改名/重置 node_id 规则
engine.plan_clear_nodes(plan) -> plan
```

UI 只负责调用 + 刷新列表。

---

### B. 计划装配 / 序列化 — 优先级：高

**位置**：`ui_qt/main_window.py` `save_plan`（632-636）

```python
self.current_plan["headers"] = list(self.current_headers)
self.current_plan["rows"] = [...]
self.current_plan["output_mode"] = str(self.output_mode_combo.currentText())
self.current_plan["output_table"] = str(self.output_table_edit.text())
self.current_plan["backup_before_overwrite"] = bool(self.backup_checkbox.isChecked())
```

**问题**：把界面控件值手工塞进 plan dict，是“plan 文档结构”的知识，泄漏到了 UI。plan schema 一变，每个前端都要改。

**建议**：引擎提供计划文档装配/读写：

```python
engine.build_plan_document(plan, headers, rows, output_settings) -> dict
engine.save_plan(path, plan_document)     # 内部用 atomic_write_json
engine.load_plan(path) -> (plan_document, warning)
```

UI 把控件值收成一个 `output_settings` dict 传进去即可，不碰 plan 的字段名。

---

### C. 节点类型解析重复 — 优先级：中

**位置**：`ui_qt/main_window.py` `_node_type_id_for_node`（731-732）

```python
return str(node.get("node_type_id") or node.get("type") or "")
```

**问题**：这复制了引擎里的 `stable_node_type_id_for_node`（`workflow/protocol_nodes.py`）。两处规则若漂移会导致 UI 显示与执行不一致。

**建议**：引擎暴露 `engine.resolve_node_type_id(node)`，UI 调用，不自己拼。

---

### D. 配置表单的字段类型推断 — 优先级：高（最影响“换界面”）

**位置**：`ui_qt/config_form.py`
- `value_kind()`（19-30）：从 Python 值类型猜 `bool/int/float/json/text`
- `_build_config_editor()`（191-207）：再用 `choices_for_field / is_long_text_field / value_kind` 拼控件类型

**问题**：引擎的 `build_node_ui_schema()` **已经**产出了 `form.groups[].fields[].type`（见 `node_ui_schema.py:653-708` `config_field_schema`），但 Qt 表单**没有消费它**，而是自己从值再推一遍类型。这是重复逻辑，且：
- 换前端时，类型推断要再写一遍；
- 引擎 schema 与 UI 推断可能不一致（例如 `int` 型默认值为 0 时两边判断一致，但 `choice`/`field_select`/`textarea` 的判定 UI 是另走 `node_ui_metadata`，容易漂移）。

**建议**：Qt 表单改成**纯 schema 驱动**——直接消费 `engine.get_node_ui_schema(node_type_id)["form"]["groups"]`，按 `field.type`（`text/number/bool/select/field_select/textarea/json`）渲染控件，不再调用 `value_kind`。这样新前端只需实现“type → 控件”的映射表，零业务判断。

---

### E. 表单值的强转与校验 — 优先级：中

**位置**：`ui_qt/config_form.py` `coerce_form_value()`（41-55）：UI 里做 `int()/float()/json.loads()` 解析和“不是有效 JSON”报错。

**问题**：控件边界上做基本强转可以接受，但“哪个字段是 int、哪个是 json、合法范围”属于 schema 知识。校验规则散在 UI，换前端要重写。

**建议**：
- 短期：保留控件层最基本的 `str→int/float`，但类型来源改为 schema 的 `field.type`（配合 D）。
- 中期：引擎提供 `engine.coerce_and_validate_config(node_type_id, raw_values) -> (config, issues)`，把校验权威化到后端，前端只展示 issues。

---

### F. 表格文件解析 — 优先级：中

**位置**：`ui_qt/table_io.py`（整文件，11-59）：JSON / CSV / TSV → `(headers, rows)`，含对象数组/二维数组的推断。

**问题**：这是数据 I/O，不是界面职责。换前端要重写解析逻辑，且解析规则（编码、表头推断）应与引擎一致。

**建议**：迁到后端，例如 `engine.load_table(path) -> TableData` 或独立 `io/table_loader.py`。UI 只负责弹文件框拿路径。

---

### G. 计划文件读写直连 shared 工具 — 优先级：中

**位置**：`ui_qt/main_window.py` `load_plan_path`（602-617）、`save_plan`（619-642）直接 import 并调用 `shared.atomic_json_utils`。

**问题**：UI 直接做文件持久化 + 计划字段装配（与 B 重叠）。前端各自接 `shared` 会造成多前端重复。

**建议**：统一走 B 里的 `engine.load_plan / engine.save_plan`，UI 不直接 import `shared.atomic_json_utils`。

---

### H. 校验结果格式化 — 优先级：低

**位置**：`ui_qt/main_window.py` `_format_validation`（734-752）：把 `validation.issues` 拼成多行文本。

**问题**：纯展示，留在 UI 可接受。但若希望多前端文案一致，可下沉。

**建议**：可选。引擎提供 `engine.format_validation(validation) -> str` 或让前端各自渲染结构化 `issues`（更推荐结构化，由前端决定展示形式）。

## 三、执行与输出：当前“只显示未实现”的部分

这是用户明确指出的短板，需要后端补实现，而不是在 UI 里补。

### 现状（已核查）

| 现象 | 证据 |
|---|---|
| “执行计划”不产生副作用 | `ui_qt/engine_client.py:83` `run_plan(..., execute_actions=False)` 写死 False |
| 输出方式是死选项 | `ui_qt/main_window.py:248` 硬编码 `["输出到主界面预览区","保存为SQLite新表","覆盖当前表","导出为xlsx"]`，仅写进 plan，无执行 |
| 引擎无输出实现 | `engine/*.py` 中无 xlsx / SQLite 写出 / `save_result` 逻辑 |
| 一批节点 headless 不支持 | `engine/headless.py` 仅支持数据节点 + 流程控制；filter / writeback / selected_columns_write / save_transit / match_value_output / plugin / file_list / batch_rename / group / loop_* 全部标记 unsupported（见 `node_ui_schema.py:547-588` 的 `仅旧执行链` badge） |

### 根因

这些不支持的节点都依赖**有副作用的服务**：SQLite 读写、文件操作、插件运行时、多表查询。引擎第一版刻意只做无副作用的纯数据/流程节点。

### 建议方向：服务注入（Service Injection），而不是让 UI 直接接

不要让 Qt 去 import `db.table_manager` 或 `plugin_runtime` 来补这些功能（那会把副作用逻辑又拉回前端）。正确做法是给引擎注入一个**服务接口**：

```python
class WorkflowServices:          # 后端定义的能力接口
    def read_table(self, name): ...
    def write_table(self, name, headers, rows, mode, backup): ...
    def run_plugin(self, plugin_id, input_table, params, context): ...
    def list_sqlite_tables(self): ...
    def export_xlsx(self, path, headers, rows): ...

engine = HeadlessWorkflowEngine(services=WorkflowServices(...))
engine.run_plan(plan, execute_actions=True)   # 副作用节点经 services 落地
```

- 副作用全部收敛在 `services` 实现里（复用现有 `db/table_manager.py`、`plugin_runtime/`）。
- UI 只调 `engine.run_plan(execute_actions=True)`，传一个 `output_settings`。
- 输出（SQLite/覆盖/xlsx）做成引擎的“输出阶段”或一个特殊输出节点，由 services 执行。
- 换任何前端，执行与输出能力都不用重写。

### 进度与取消

引擎 `run_plan` 已支持 `progress_callback` 和 `cancel_event`（`engine/headless.py:290-291`），但 Qt 现在没用——只在结束后把进度条设成 100（`main_window.py:679-682`）。建议：
- 后台线程跑 `run_plan`，`progress_callback` 通过 Qt 信号回主线程更新进度条；
- “取消”按钮 set `cancel_event`。

这属于前端集成工作，逻辑本身已在后端。

## 四、目标边界（一句话原则）

> **前端只做三件事**：把 schema 渲染成控件、把用户操作转成对 `engine` 的调用、把 `engine` 返回的结构渲染出来。**任何“计算、校验、装配、解析、落地”都在 engine 后面。**

对照检查表（换前端时，前端不应包含以下任何一项）：

- [ ] 不推断字段类型（消费 schema 的 `field.type`）
- [ ] 不拼 plan 的字段结构（调 `engine.build_plan_document`）
- [ ] 不实现节点增删改移复制规则（调 `engine.plan_*`）
- [ ] 不解析表格/计划文件（调 `engine.load_table / load_plan`）
- [ ] 不判断节点类型 id（调 `engine.resolve_node_type_id`）
- [ ] 不直接 import `db / plugin_runtime / workflow.nodes / shared`

## 五、实施优先级路线图

### 阶段 1：消除 UI 里的纯逻辑（低风险，高回报）

1. **D + E**：Qt 表单改为消费 `schema["form"]["groups"]`，删掉 `value_kind` 推断。（最影响“换界面”，先做）
2. **F + G**：`table_io.py` 解析、计划文件读写下沉到 `engine.load_table / load_plan / save_plan`。
3. **C**：节点类型解析改调 `engine.resolve_node_type_id`。

> 验证：跑现有 `tests/test_qt6_ui_shell.py`、`tests/test_qt_compat.py`，并为新引擎接口补单测。

### 阶段 2：计划编辑 API 化（中风险）

4. **A + B**：实现 `engine.plan_*` 系列与 `build_plan_document`，Qt 改为调用。
5. 给这些纯函数 API 写单测（不依赖 Qt）。

### 阶段 3：补执行与输出能力（核心功能）

6. 定义 `WorkflowServices` 接口，用现有 `db/table_manager.py`、`plugin_runtime/` 实现。
7. 引擎接入 services，打通 `execute_actions=True` 路径。
8. 逐个把 unsupported 节点接上（建议顺序：`save_transit` → `selected_columns_write` → `writeback` → `filter` → `plugin` → 文件/循环/组）。
9. 输出阶段（SQLite/覆盖/xlsx）经 services 实现，UI 的 `output_mode` 选项改为由引擎 `list_output_modes()` 提供，不再硬编码。

### 阶段 4：进度/取消/异步（体验）

10. Qt 后台线程跑 `run_plan` + `progress_callback`（Qt 信号）+ `cancel_event`。

## 六、是否继续单纯 Python？

就“界面解耦”这个目标而言，**当前不需要引入第二语言**。理由：

- 解耦的本质是“逻辑下沉到 engine”，与语言无关；
- 已有 `StdioWorker` JSON 协议，未来若上 Electron/.NET 前端，可直接走 stdio/HTTP 调同一个 Python engine，不必重写逻辑；
- C/C++ 只在“单节点千万行级运算”才有价值，目前 `MAX_EXPANDED_ROWS=200000` 规模用不上。

**结论**：保持 Python 后端，先把上面阶段 1-3 做完，前端是 Qt 还是别的都能平滑替换。
