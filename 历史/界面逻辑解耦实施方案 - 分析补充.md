这份 GPT 写的实施方案整体质量很高,服务分层、JobService/OutputService、stdio action 映射这些都补得不错,方向我认同。我来对照实际代码做批判性审查,找出**它漏掉或可能不准的地方**。先核实几个关键事实再给补充。核实完毕,几个关键事实印证了这份方案的盲点。整体方向我认同(服务分层、JobService/OutputService、stdio action 都补得好),但有几处**基于实际代码必须补的硬缺口**,它没提到。

## 必须补充的(高优先级)

**1. Schema 覆盖现状是个大坑,方案严重低估了工作量。**
实测 `NODE_CONFIG_LAYOUTS` 现在**只定义了 2 个节点**(`core.new_columns`、`core.replace`),其余约 30 个节点全部 fallback 到通用"其他参数"扁平键值表单。也就是说"schema-first 表单"第一刀做完后,**复杂节点仍然是裸 JSON/键值编辑器**。方案把 schema-first 列为"低风险高收益第一刀",但没说清:真正的工作量是**为那 30 个节点逐个补写 schema layout**。而且规格来源现成——`ui/workflow/config/` 里还留着 19 个旧 tkinter 配置 UI(filter 646 行、writeback 528 行、selected_columns 590 行),这些就是复杂节点表单的事实规格。方案应明确"以旧 config UI 为规格,逐节点补 schema",否则会低估到失控。

**2. Schema 模型本身表达力不够,这是 schema-driven 最难的部分,方案完全没提。**
当前 `config_field_schema` 是**扁平结构**(只有 key/label/type/choices/default/help),没有字段间依赖。但复杂节点的核心交互恰恰是动态联动:筛选节点选了副表才出现该表字段、值来源选"字段值"才切成字段选择器、选了"正则"模式才显示正则参数。这些靠扁平 schema 根本表达不了。方案列了一堆字段类型(`multi_field_select`/`rule_list`/`mapping_table`),但漏了更关键的一层——**`visible_when` / `depends_on` / 动态 options 刷新**的 schema 模型扩展。不补这个,schema-first 对复杂节点就是空话。

**3. Plan 迁移与版本化是硬需求,不是可选项。**
实测真实 `plan/*.json` 用的还是旧中文 `type`(如 `'获取文件列表'`)、**没有 `node_type_id`**。方案提了 `resolve_node_type_id`,但缺三样:`migrate_plan()`(旧计划升级到新结构)、plan 的 `schema_version` 字段、**用真实 plan 文件做回归测试**。换界面前不解决迁移,用户现有的工作流在新引擎/新 schema 下会静默错乱。

**4. 预览零副作用要作为安全不变量单独声明。**
方案说"执行和预览语义分离",但没把它提到红线高度。`WorkflowServices` 的 dry_run/execute 标志必须贯穿所有副作用节点——预览即使遇到 writeback/file/plugin **也绝不能落地**。旧代码里 `execute_actions` 检查是散落各处的(我之前审查发现过),新服务边界必须**集中强制**这个不变量,否则一次预览就可能改库/改文件。这是数据安全问题,值得单列。

## 建议补充的(中优先级)

**5. 双 UI 栈并存策略缺失。** `ui_qt/` 和 `ui/workflow/config/`(17 个 tkinter 文件)现在同时存在。方案默认 Qt 是唯一前端,没交代旧 tkinter 配置窗口怎么处置:是留作复杂节点 schema 的规格参考?阶段 4 完成前继续用旧 UI 执行不支持的节点?何时下线?需要一个明确的过渡/切换计划。

**6. 统一错误/issue schema。** 现在三种形态并存:`validate_plan` 返回结构化 issues、`run_plan` 抛 RuntimeError 字符串、未来 `coerce_config` 又是一种。建议跨 validate/coerce/run/output **统一一种 issue 结构**(severity/code/node_index/message),前端一个渲染器搞定。方案说了"结构化 errors"但没强调统一。

**7. 大表跨边界传输。** 一旦走 stdio/HTTP,20 万行 `headers/rows` 全量 JSON 序列化很重。方案的 `TableDataService` 应加**表句柄 + 分页/预览行上限**概念(`load_table` 返回 handle,再窗口化取行),否则"未来 Electron/HTTP"目标扩展不了。进程内 Qt 传对象没事,但方案目标是多前端,这点要前置设计。

**8. 撤销/重做是 plan-command 的免费红利。** `apply_plan_command` 返回新 plan,天然适合命令历史做 undo/redo——这是用户迟早会要的能力,顺手在 command 模型里设计进去,比事后补便宜得多。方案没提。

## 小修正

**9. 单一数据源避免漂移。** `FIELD_CHOICES` 里的算子列表(如 `op`)、`compare_values` 实际支持的算子、`default_configs` 的默认值是**三处分开维护**的,容易漂移。建议 choices 从校验/执行同一处派生。

**10. 测试保持 headless 可导入。** 我之前发现部分节点测试经 `DataFlowKit`→tkinter 导入,在无 GUI 环境 collection 直接失败。engine/schema/command 的测试**必须不经 tkinter/Qt**,否则 CI 和无头环境跑不了——这是支撑整个解耦的基础,值得写进验收标准。
