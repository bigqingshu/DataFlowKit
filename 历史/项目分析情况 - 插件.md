我建议把插件处理分三档。

第一档，建议现在就做
- 让 Qt 节点目录真正加载插件注册表，把插件显示在节点树里，能新增到计划中。
- 选中插件节点时，右侧直接用 `get_plugin_schema()` 返回的 form 渲染配置，不再把插件当特殊弹窗对象。
- 支持插件节点的基础校验、预览、执行状态显示，优先只走已有 headless/plugin service。
- 把插件的 `load_status`、`available_run_modes`、`import_ok`、warning/badges 明确展示在节点说明里。

这一档做完，插件就已经纳入“统一节点协议 + 统一右侧配置面板 + 统一执行链”了。这个对 Qt、后续 .NET、甚至 stdio worker 都最值钱。

第二档，Qt 阶段建议做
- 插件节点专门的“配置摘要区”或“运行模式区”，把这些插件特有字段整理得更像一个插件卡片：
  - `plugin_id`
  - `run_mode`
  - `external_entry`
  - `external_env_dir`
  - 输出模式
  - 日志保存策略
  - 失败策略
- 插件节点详情页里增加“插件能力/执行环境/风险说明/输出行为”的结构化说明。
- 为“仅独立环境运行”的插件做更清晰的交互提示，而不是等执行时报错。
- 插件列表刷新、插件扫描错误展示、插件不可用原因展示。

这一档主要是实用性和可解释性，不会破坏主架构。

第三档，可以更后做
- 兼容老插件的“自定义窗口”或 `open_config_window`
- 为插件提供更丰富的前端扩展点，比如自定义子页签、自定义复杂编辑器、嵌入式面板

这个要后做，因为一旦太早做，很容易把你现在辛苦建立的“UI 与逻辑分离”又拉回去。

关于“插件窗口怎么处理最好”，我比较明确地推荐这套策略：

- 默认路线：不做“插件自己弹 Tk/Qt 窗口”，而是让插件只提供 schema、默认配置、校验、执行、输出声明，界面统一由主程序渲染。
- 兼容路线：保留老插件窗口能力，但把它降级为“兼容模式/高级模式/旧版设置窗口”，不要让它成为主路径。
- 长期路线：如果以后确实需要复杂插件 UI，也不要让插件直接操控主界面控件，而是给它定义一层“插件 UI schema 扩展协议”或“插件页面描述协议”。

也就是说，插件窗口最好分成两类：

- `标准插件窗口`：其实不是独立窗口，而是主界面右侧表单/页签里的统一渲染结果。
- `兼容插件窗口`：只给那些历史上已经写死了复杂交互、短期改不掉的插件临时保留，通过一个“打开旧版插件设置”按钮进入。

这样做的好处很直接：
- Qt / .NET / Electron / Flutter 都能复用同一套插件节点协议。
- 插件逻辑不会再绑死 Tk 组件树。
- 后续如果上 worker/远程执行，插件配置仍然是可序列化的，不依赖某个 GUI 进程。
- 旧插件还能活，不会一次性全炸。

什么时候处理插件节点最合适，我给你的时序建议是：

1. 现在 Qt 主干继续收口时，就把“插件节点纳入统一节点体系”做掉。
2. 等插件节点能在 Qt 里新增、配置、校验、跑通基础执行后，再看哪些具体插件仍然需要旧窗口兼容。
3. 真正的“插件窗口重构/插件 UI 扩展协议”放到 Qt 第一档可用之后，再单独开分支做。

如果按你现在这条解耦路线，我会把下一步插件工作切成两个安全块：
