# 项目阶段性总结 - Qt优先与多UI兼容

## 1. 当前项目所处阶段

从当前代码和最近一段提交来看，项目已经不再是“是否要解耦、是否要做多界面”的早期讨论阶段，而是进入了一个比较明确的中段：

1. **协议层方向已经基本确定**
   - 节点协议以 JSON 为核心载体。
   - 配置、能力、动作、表单字段、节点说明，正在逐步从“界面里临时拼出来”转向“由协议和结构化描述统一提供”。
   - 这一步很关键，因为后续 Qt、Tk、.NET、Electron、Flutter 是否能共用后端，核心就看协议能否稳定。

2. **执行层已经开始脱离原始 UI**
   - 已有 `headless-engine-api`、`stdio-worker-api`、`tkinter-adapter` 这条路线的基础成果。
   - 说明当前已经不再把“主窗口 UI 直接驱动全部逻辑”当成唯一模式，而是开始形成“执行引擎 + 协议 + 适配层”的结构。

3. **Qt 已经从原型期进入可用壳阶段**
   - Qt 不是只有一个空壳了，而是已经具备节点详情、配置面板、结构化字段、消息面板、表格动作、上下文联动等基础能力。
   - 但它目前仍然主要处于“可展示、可配置一部分、可验证一部分”的阶段，还没有完全达到“可稳定替代原 Tk 工作流主界面”的程度。

4. **多 UI 兼容已经从理念变成约束**
   - 现在不仅有 Tk 和 Qt，还出现了 `.NET` UI 壳。
   - 这意味着后续设计不能再默认“Qt 可以随便加一点本地逻辑”，而必须开始考虑：哪些内容必须沉到共享后端，哪些只是某个前端自己的表现层增强。

所以，当前更准确的判断是：

**项目正处于“协议与执行层基本成型，Qt 主界面进入核心收口阶段，多 UI 兼容开始成为正式设计约束”的阶段。**


## 2. 当前已经形成的成果

### 2.1 已经比较明确的主路线

当前主路线已经比较清晰：

- **Python 继续作为主编排核心**，负责工作流执行、插件承载、数据流调度。
- **节点协议 JSON 固定化**，后续通过 schema / ui schema / action schema 扩展，而不是继续堆散落的 UI 特判。
- **Qt 作为下一代主桌面 UI 方向**，优先使用 Qt6，同时保留 PyQt5 兼容思路以照顾 Win7 或历史环境。
- **Tk 保留为兼容入口与过渡实现**，逐步通过 adapter/facade 接入新的协议执行层。
- **其他 UI（.NET / Electron / Flutter）暂时不抢主线**，但其兼容性要求会反过来约束当前接口设计。

### 2.2 当前架构上的正向变化

已经能看到几个非常重要的变化：

- 节点信息不再只依赖中文 `type` 和旧 UI service 的隐式判断。
- 配置界面开始从“原始 JSON 编辑器”往“结构化表单”迁移。
- 表字段、单字段、多选字段、节点能力说明、消息反馈等都在向协议化描述靠拢。
- 引擎层和 worker 层已经有了明确边界，后续更适合继续抽离成本地服务 / 子进程服务 / 远程服务。

这些变化说明，当前不是在单纯“换个 UI”，而是在把项目逐步改造成“一个有稳定后端接口、可以挂多个前端”的系统。


## 3. 当前还不算完成的地方

虽然方向已经比较稳，但目前还不能说“方案已经完全落地”，原因主要有四类：

### 3.1 协议覆盖还不够彻底

现在很多节点已经能映射到结构化 UI，但仍然会存在这些情况：

- 某些复杂节点只能部分结构化，剩余部分仍回退到 JSON 文本编辑。
- 某些字段的候选项、联动规则、默认值、校验逻辑，还存在前端临时推断。
- 某些节点说明、子菜单说明、警告说明，还没有形成统一的结构化来源。

### 3.2 Qt 还没有完全形成“主工作流界面闭环”

目前 Qt 已经能展示和编辑很多内容，但距离“原版工作流主界面的真正可替代版本”仍有差距，尤其在：

- 关键工作流链路是否都能完整配置
- 配置后是否都能稳定校验
- 运行前后的结果反馈是否足够完整
- 某些高阶操作是否仍依赖旧 Tk 逻辑

### 3.3 后端接口还没有完全收干净

现在虽然已经开始解耦，但仍可能残留：

- 某些动作入口还带着 UI 侧假设
- 某些服务返回值还偏向旧界面使用习惯
- 某些执行前后的状态整理、错误解释、可读说明，仍由前端补位

### 3.4 多 UI 兼容还处于“设计上兼容”，不是“工程上成熟兼容”

换句话说：

- 当前架构已经开始为多 UI 做准备，方向是对的；
- 但距离“.NET UI 直接接协议就能顺利做完整界面”还有工程差距；
- 这个差距主要不在通信技术，而在**协议覆盖率、能力描述完整度、动作语义稳定度**。


## 4. 当前阶段的判断

如果把整体拆成几个阶段，当前大致处于：

### 第一阶段：方向确定
- 保留 Python 核心
- 节点协议 JSON 化
- 多 UI 路线成立

这一阶段已经完成。

### 第二阶段：核心解耦
- 执行层脱离 Tk 主窗口
- headless / worker / adapter 成形
- 节点描述开始结构化

这一阶段已经完成大半，基础骨架已经搭起来。

### 第三阶段：Qt 主界面收口
- Qt 不再只是演示壳，而是逐步承担真实配置与运行职责
- 关键节点和关键工作流在 Qt 里形成闭环
- 剩余 UI 推断逻辑回收到协议/后端

**当前主要就在这一阶段。**

### 第四阶段：多 UI 真正复用
- Qt、Tk、.NET 共用后端协议
- 新前端只做展示与交互，不再重写业务逻辑
- 后续网页端或远程 worker 接入成本明显下降

这一阶段还没有完成，目前只是提前为它铺路。


## 5. 为什么当前仍然要优先完成 Qt

虽然已经有 `.NET` UI 壳，但当前仍建议把 Qt 放在第一优先级，原因很现实：

1. **Qt 是当前最接近“下一代正式桌面主界面”的实现。**
2. **Qt 已经有连续提交和真实功能积累，继续推进产出更快。**
3. **Qt 能更充分暴露协议设计问题。**
   - 如果协议不足，Qt 这种更强交互的界面会很快暴露缺口。
4. **先把 Qt 做通，会直接提高 .NET / Electron / Flutter 的成功率。**
   - 因为后者真正复用的，不是 Qt 的代码，而是 Qt 倒逼出来的共享协议、共享动作和共享服务。

所以当前最合理的策略不是“Qt 和 .NET 同时深做”，而是：

**先把 Qt 做成第一个真正跑通共享协议的强前端，再让 .NET 等前端复用已经沉淀好的接口。**


## 6. 第一档 Qt 需完成的核心任务

这里说的“第一档”，指的是那些**必须先完成，否则 Qt 仍然只是一个半成品壳**的任务。

### 6.1 核心任务一：把关键节点配置真正协议化、表单化

当前最关键的事情，不是再加几个视觉控件，而是继续把真实节点的配置能力沉到协议里。

重点包括：

- 复杂节点的字段 schema 补全
- 复杂列表类配置的结构化表达
- 子菜单说明、字段说明、警告说明的结构化输出
- 默认值、可选项、字段联动、依赖条件的统一描述
- 尽量减少“前端看到一个字段名后自己猜怎么渲染”

这一步完成后，Qt 才能真正做到：

- 不依赖大段 JSON 文本编辑
- 能清晰显示节点内容与设置
- 能把中文说明、警告、子菜单信息直接挂到界面里
- 后续也更方便给 `.NET`、Electron、Flutter 复用

这是**第一优先级中的第一项**。

### 6.2 核心任务二：把 Qt 仍在本地承担的业务判断继续回收到后端

如果目标是“界面更容易替换”，那就必须继续减少 Qt 侧的业务判断。

优先回收的内容通常包括：

- 字段类型判断
- 配置值规范化 / 类型转换
- 复杂候选项刷新规则
- 节点能力说明拼装
- 某些错误提示、警告提示的来源生成
- 某些执行前预检查的归类与结果组织

目标不是让 Qt 变笨，而是让 Qt 更纯粹：

- Qt 负责展示表单、触发动作、显示结果；
- 后端负责告诉它：能做什么、该怎么做、为什么不行。

这一步对后续 `.NET` UI 很重要，因为如果这些逻辑继续写死在 Qt 里，后面 `.NET` 还得重新做一遍。

### 6.3 核心任务三：把关键工作流路径在 Qt 中跑成闭环

Qt 现在已经不缺“能显示一些内容”的能力，接下来更重要的是验证关键工作流链路是否真正闭环。

建议优先围绕高频路径建立闭环：

- 选择节点
- 查看节点说明
- 编辑结构化配置
- 获取依赖字段/表上下文
- 运行校验
- 预览结果/消息
- 触发执行或交给 worker 执行

也就是说，要从“单点控件可用”升级到“典型任务链路可用”。

如果这一点不先做，Qt 很容易出现一种假成熟状态：界面看起来已经很多，但真实流程仍需要回到 Tk 或手改 JSON。

### 6.4 核心任务四：建立 Qt 功能覆盖矩阵

当前项目已经积累了不少节点和协议能力，此时很需要一张明确的覆盖矩阵，而不是靠印象判断进度。

建议至少按下面几个维度标记：

- 是否已有稳定节点协议
- 是否已有 UI schema
- Qt 是否可结构化配置
- 是否可获取完整上下文
- 是否可在 headless 中校验
- 是否可在 worker 中运行
- 是否仍依赖 Tk 专属 service

这张矩阵的价值很大：

- 能快速看出 Qt 离主用还差哪些块
- 能帮助安排分支拆分和并行开发
- 能避免后续重复补洞
- 也能直接指导 `.NET` UI 先接哪一批能力

### 6.5 核心任务五：补齐“说明、警告、帮助信息”的统一输出模型

前面已经讨论过，用户希望中文说明能够直接显示在子菜单里，也希望警告提示更清晰。

如果只是 Qt 临时把说明文本塞进菜单，短期能用，但长期不利于多 UI 复用。

更合理的做法是把这些内容也做成协议输出，例如：

- 节点简介
- 字段说明
- 子菜单项说明
- 风险警告
- 前置条件提示
- 兼容性提示

这样做有几个好处：

- Qt 可以直接展示
- Tk 也能逐步复用
- `.NET` / Electron / Web 前端都能直接消费
- 文案、警告、帮助不会再散落在不同 UI 实现里

这一项虽然看起来像“体验优化”，但实际上它对解耦非常关键。

### 6.6 核心任务六：建立第一档回归验证基线

Qt 越往前做，越不能只靠手点验证。

第一档至少应建立几类稳定验证：

- 节点协议 schema 校验
- UI schema / 字段动作 schema 校验
- headless 请求与返回结构校验
- stdio worker 协议收发校验
- 若干典型节点配置的最小回归用例

这里不一定一上来就做很重的 GUI 自动化，但至少要保证：

- 协议没被改坏
- 表单描述没被改坏
- worker 通信没被改坏
- 关键节点的结构化配置路径没被改坏

否则 Qt 做得越多，返工成本越高。


## 7. 第一档 Qt 核心任务的建议拆分顺序

为了便于后续分叉并行，建议第一档按下面顺序拆：

### A 组：协议补全

目标：把关键复杂节点从“半结构化/半 JSON”推进到“结构化优先”。

建议内容：

- 补字段 schema
- 补结构化列表 schema
- 补说明/警告/子菜单元数据
- 补字段联动与默认值规则

这是最值得优先分叉的一组，因为它对所有 UI 都有价值。

当前进度（2026-06-22）：

- 已完成字段说明、校验要求、动态显示/启用语义、字段动作提示的共享协议输出。
- 已完成字段 `help_sections` 下沉，Qt tooltip 已直接消费共享 payload，而不是本地重新拼接。
- 已补强 `高级筛选`、`字段映射写入表` 等复杂节点的协议表单分组、结构化列表列定义、表驱动字段联动与动态规则。
- 当前 A 组已进入“继续扩复杂节点覆盖面”阶段，基础模式已经跑通。

### B 组：后端接口回收

目标：把 Qt 本地承担的判断继续沉到共享层。

建议内容：

- 候选项刷新规则下沉
- 配置规范化下沉
- 能力说明拼装下沉
- 预检查/错误归类下沉

这组做完后，Qt、Tk、.NET 的重复劳动会明显减少。

当前进度（2026-06-22）：

- 已开始把节点详情里的表单能力摘要、兼容性摘要继续下沉到共享 `detail payload`。
- 已补充共享 `detail payload` 的 `meta_items/meta_text`，把节点详情里的分类、类型、风险、执行层、兼容性摘要整理为可直接渲染的数据。
- Qt 节点详情面板已优先消费共享详情元信息，而不是继续在前端本地拼接 `meta_parts`。
- 已补充 `describe_validation_feedback()` 的结构化 `sections/summary_lines` 输出，把基础校验、跳转校验、权限预检的说明分段下沉到 facade。
- Qt 反馈面板已开始优先消费结构化校验摘要，说明区与问题区的内容组织进一步减少前端本地解释。
- 已把导入输入表格、打开计划、保存计划三类状态补充为共享 `message_panel` payload，Qt 不再需要为这几类操作单独本地拼装提示面板。
- 下一步重点是继续把能力说明、兼容性说明、预检查/提示信息组织逻辑往 facade/schema 输出层收口，并逐步统一更多状态流到通用消息面板协议。

### C 组：Qt 闭环打通

目标：让高频工作流在 Qt 里真正可用。

建议内容：

- 节点详情到表单编辑闭环
- 表上下文和字段联动闭环
- 校验、消息、结果反馈闭环
- 执行动作与 worker 交互闭环

这组是 Qt 是否能进入“主用候选”的关键。

### D 组：覆盖矩阵与回归基线

目标：把项目进度和风险可视化。

建议内容：

- 节点/能力覆盖矩阵
- 关键节点样例集
- schema 回归测试
- worker 协议回归测试

这组看起来不如前面显眼，但它会直接决定后续推进效率。


## 8. 对 .NET UI 壳的建议定位

当前 `ui_net/DataFlowKit_net_ui` 可以保留，但建议定位清楚：

- **现在先作为兼容性观察样板，不作为当前主战场。**
- 它的价值主要是帮助判断：
  - 哪些协议字段定义还不够清晰
  - 哪些动作语义过度依赖 Qt/Tk 假设
  - 哪些说明与警告还没有标准化输出

也就是说，当前不是要马上把 `.NET` UI 做完整，而是让它成为一个“反向检查器”：

**凡是 .NET 难接的地方，大概率就是当前共享协议还不够完整的地方。**


## 9. 现阶段最推荐的结论

如果只给一个简化结论，那么当前最推荐的路线是：

1. **继续以 Qt 为当前第一前端目标。**
2. **把第一档重点放在协议补全、后端回收、关键链路闭环。**
3. **不要为了 Qt 进度而重新把逻辑写回 Qt。**
4. **把 .NET UI 先作为兼容性约束，而不是抢主线实现。**
5. **等 Qt 真正跑通后，再让 .NET / Electron / Flutter 去复用同一套协议和服务。**

这样推进，后续无论是继续做 PyQt5/PyQt6 双轨，还是往更高级 UI、网页端、远程 worker 扩展，整体都会更稳。


## 10. 一句话阶段判断

当前项目不是“还在讨论怎么解耦”，也不是“已经完成多 UI 架构”，而是：

**已经完成方向定型和基础解耦，正在进入 Qt 主界面收口的关键阶段；这一阶段做得是否干净，决定后续 .NET、Electron、Flutter 这些界面能否真正低成本复用后端。**
